- **Single-flight verification coordination (`scripts/verification-flight.py`).** Implement and inline Review-and-Fix now coordinate each same-checkout full-verification flight through a new non-executing coordination ledger (`claim`/`mark-running`/`finish`, attach/`wait`/consume) instead of relaunching the same unchanged suite within one lifecycle. The helper launches no subprocess and accepts no executable argv — existing callers keep command ownership and allowlist enforcement — and coordinates on a SHA-256 command descriptor plus a full checkout fingerprint (HEAD, index, tracked, untracked). States are `claimed`/`running`/`passed`/`failed`/`timed_out`/`cancelled`/`stale`/`incomplete`; only a `passed` handle with complete matching bindings satisfies verification, and a missing/partial/timed-out/unreadable/stale handle never becomes a pass and never authorizes an automatic relaunch. Adds a versioned `verification_flight` config namespace and grants the vendored helper in `devflow.yml` and `devflow-implement.yml` only (never the read-only reviewer). CI-grounded standalone `/devflow:review` creates no flight. (#528)
- **Review hardening (#579).** `_emit_telemetry` now also swallows `TypeError`/`ValueError` (a non-serializable payload) so best-effort telemetry can never harden into a coordination failure; the `phase-3-review.md` re-anchor prose now states explicitly that re-anchoring recomputes the flight key from the *current* tree (a bare re-read of the stored key without a fresh current-checkout is not a valid consume); the `verification_flight.lease_seconds` / `wait_timeout_seconds` config keys are documented as declarative in Wave 2; and a negative test pins that `output_roots`/`external_services` are excluded from the command descriptor so a regression that added them (silently churning the flight key and defeating all reuse) fails closed.
- **Auto-review CI-completion re-trigger now covers every PR-gating workflow (#579).** `devflow-review.yml`'s `workflow_run` re-trigger list was `[CI]` only, but `require_ci_green` defers a review until **every** other Actions run on the head completes — so a second gating workflow that finished after `CI` (here `Matcher probe`, which runs when the allowlist/workflow files change) left the review stranded at the neutral "waiting: other CI not green" check with no event left to clear it. The list is now `[CI, Matcher probe]`, and a new self-sync guard (`lib/test/check-review-retrigger-coverage.py`, driven by `lib/test/run.sh`) fails closed if any first-party PR-triggering workflow is missing from the list, so it cannot silently rot again. `require_ci_green`'s semantics are unchanged (no risk of reviewing red code); install-time guidance and the workflow-trigger docs now say to list every PR-gating workflow, not just the primary CI one.
