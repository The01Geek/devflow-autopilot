# Reference: Pre-fix Gates (Step 2.5 + Parked-class sweep)

### Step 2.5: Pre-fix verification gate

Before applying any fixes, classify **every finding this iteration routed to the fixer** — i.e. the effective fix set from Step 2: each finding at or above `$FIX_THRESHOLD` plus each REJECT-driver from the widening rule (so when `$FIX_THRESHOLD` is `suggestion`, Suggestion/Minor findings admitted to the fixer get this same protection, not just Critical/Important ones). The goal is to keep the loop from auto-applying confidently-stated-but-wrong fixes; LLM verbalized confidence is poorly calibrated, especially on claims about external tool/framework behavior. Phase 2 checklist FAILs and findings raised by ≥2 Phase 3 agents are *corroborated* — pass them straight to Step 3. The gate targets the remaining single-source findings.

For each such fixer-bound finding raised by exactly one Phase 3 agent:

1. **Classify the claim:**
   - **External-tool claim** — the finding rests on a specific external framework, CLI flag, GitHub Actions semantic, library API, or platform behavior the orchestrator could look up in docs (e.g. *"id-token: write only takes effect at workflow level"*, *"--permission-mode acceptEdits denies Bash"*, *"`@/`syntax must be quoted in claude-code-action"*). Run web verification.
   - **Codebase claim** — the finding is about this repository only (e.g. *"this method bypasses the project's EntityService pattern"*, *"caller doesn't handle the empty-array return"*). External docs cannot adjudicate these; pass through to Step 3 unchanged.

2. **Web verification** (up to a per-iteration cap of **5** WebFetches; remaining external-tool claims that don't fit the budget become *advisory*):
   - Compose ONE focused query naming the tool and the claimed behavior. Prefer queries that target the canonical documentation source (e.g. `site:docs.github.com id-token permission job level`) over generic web search.
   - WebFetch the most-authoritative-looking source. Preference order: official documentation → tool's GitHub repo / release notes → blog or third-party tracker. Use WebSearch to find the URL first only when no canonical doc URL is obvious.
   - Classify the result:
     - **Confirmed** (the docs explicitly support the agent's claim) → keep finding; auto-fix in Step 3.
     - **Refuted** (the docs explicitly contradict the agent's claim) → **demote to *advisory* with a `refuted by {url}` tag**. Do NOT auto-fix. Also append a line to the workpad's `Devflow Reflection` section: `verified false positive — {claim text} — refuted by {url}` to build the evidence trail. The finding still surfaces in this iteration's `## Advisory Findings` section so the human reviewer can override if the docs were wrong about the codebase. (Earlier versions of this skill *dropped* refuted findings entirely; that hid user-visible evidence and was a primary drift mechanism vs. /devflow:review.)
     - **Inconclusive** (the docs don't directly address the claim, or the fetched page is ambiguous) → demote to *advisory* — do NOT auto-fix.

3. **Add an `## Advisory Findings` section to the iteration's report** listing every advisory finding verbatim (the original agent's claim plus a one-line reason: `refuted by {url}`, `inconclusive after web verification`, or `over verification budget`). Advisory findings:
   - Are surfaced for human attention but are **not** auto-fixed.
   - Do **not** contribute to the per-iteration REJECT/APPROVE verdict — they're parked, not failing, so the loop can converge. They are therefore NEVER a REJECT trigger that the Loop Exit `### Pre-mapping: Step-3-evaluated REJECT downgrade` section has to evaluate; the gate's qualifying `skip_category` set deliberately does not include `advisory-parked` because advisory findings can't reach the gate as triggers in the first place.
   - **Do** contribute to the final reported verdict at Loop Exit: if any advisory findings survive when the engine would otherwise return a clean APPROVE, the final verdict becomes **APPROVE WITH ADVISORY NOTES** and the full advisory list lands in the chat-only final report (see "Verdict → chat output"). For this Loop Exit trigger, advisory findings exclude `decision: "below-threshold"` rows. This prevents the loop from silently dismissing concerns it couldn't fix without letting the sweep-only producer alter verdict semantics.
   - Carry forward across iterations unchanged; do not re-verify the same advisory finding on a later iteration in the same run.
   - **Are recorded in the workpad at demotion time, not at Step 3 / item 7.** When demoting a finding to advisory in this step, append a row to `fix_decisions` of the form `{finding_id, decision: "advisory", source_file, claim_text, skip_category: "advisory-parked", evidence: <one-line demotion reason>, parking_evidence: {basis, failing_input, source, finding_ref}}`. This keeps the workpad's `fix_decisions` array the single source of truth for every per-finding outcome (applied / pushed_back / deferred / advisory / severity-calibrated) — the item-6-anchored Write (record shape in Step 3 item 7) will then only need to carry rows for the applied / pushed_back / deferred / severity-calibrated decisions it sees, since advisory rows are already present.

**Record `parking_evidence` on the advisory row at demotion time (the parking-time record).** Beside the retained one-line `evidence` string, write the structured `parking_evidence` object `{basis, failing_input, source, finding_ref}` per its authoritative shape (Persistent workpad → Schema): `basis` is the one-line causal rationale for parking — on an **uncitable** demotion (`web_inconclusive`, `over_budget`, or `tools_unavailable`), `basis` **names the recorded `step25_classification` outcome** so condition (c)'s uncitable branch has a real operand; `source` is the refutation URL on a `web_refuted` demotion (else JSON `null`); `finding_ref` is the `{iter, index}` join to the finding's parking-time `phase3_findings` record and is **always** written (the Schema block above owns the null-is-a-writer-omission → fail-closed semantics). The existing `evidence` string is retained unchanged (the `deferrals.json` `explanation` input).

**Agreement heuristic.** Two findings agree when they describe the same defect (same root cause + same affected file/line span); identical wording is not required. Use your own judgment; do not invoke a subagent for this.

**When WebFetch/WebSearch are unavailable** (older workflow, local invocation with restricted tools), skip the web step: external-tool claims that cannot be verified are demoted to advisory directly, and this arm records `step25_classification: "tools_unavailable"` on the demoted finding (the additive enum value — a distinct, uncitable demotion outcome from `web_inconclusive`/`over_budget`, so condition (c)'s uncitable branch has a real operand on tools-restricted tiers). The gate still provides value via the cross-agent corroboration filter.

### Parked-class sweep (pre-shadow convergence gate)

Run this gate at both convergence entries: after Step 2 forms any tentative non-REJECT final verdict and on Step 4.5's non-REJECT early-exit path. Run it **before** the convergence-time shadow trigger whenever final recorded state carries at least one parked finding. This placement deliberately diverges from the Park-calibration gate's post-shadow Decide-outcome-1 site: enumerating and registering siblings first lets the blinded shadow corroborate the post-sweep state and lets an above-threshold sibling enter normal promotion machinery. The Step-3-evaluated REJECT-downgrade conclusion is an explicit non-goal because it passes through neither convergence entry and already routes its deferrals to follow-up work; Loop Exit records `parked-class sweep not applicable: downgrade-path conclusion` there.

**Inputs.** Build the same three-source union the Park-calibration gate reads: `fix_decisions` rows with `skip_category: "advisory-parked"`; every unactioned Suggestion/Minor finding derived from recorded `phase3_findings` minus `applied` dispositions (including mixed-severity iterations); and Yes-downgrade deferrals. Include the new `decision: "below-threshold"` rows only through this derivation. Exclude a finding when its recorded evidence establishes the claim false: an advisory row carrying `refuted by {url}`, or a `claim-quality` pushback whose evidence disproves the claim. A false class is not a useful enumeration seed; priming on it manufactures lookalikes instead of siblings.

**Enumeration and bounds.** Generalize each remaining input to its recorded `defect_signature.kind` and scan exactly Step 3 item 3's changed surface: the PR diff plus fix-touched files, never pre-existing untouched code. The shared engine tolerates a finding that omitted its signature, so a missing or malformed `defect_signature.kind` enters the bounded semantic batch under the literal `unknown-kind`; it is never dropped from the union. Every union member must receive a class disposition before a clean sentinel is legal. The orchestrator handles mechanically matchable kinds directly. Dispatch the semantic remainder to bounded agents under the engine's full `defect_signature` plus severity output contract. Require each semantic dispatch to return a result envelope containing `status`, `findings`, and `seed_dispositions`. The `seed_dispositions` array joins 1:1 to the assigned union members by `finding_id`; every row has `{finding_id, status: "classified" | "not_verified", kind_literal: <non-empty string or null>}`. An `unknown-kind` seed counts as classified only when the semantic batch supplies its non-empty `kind_literal`; an empty `findings` array alone never proves that the seed was examined. Envelope `status` is `"complete"` only after the assigned batch was fully examined and every assigned seed has exactly one `classified` disposition with a non-empty kind; `findings` may then be empty. A missing, duplicate, `not_verified`, null-kind, or extra disposition makes the envelope non-complete and enters the bounded retry below. Only a well-formed result envelope with `status: "complete"` and the exact disposition join may contribute an empty sibling set or a clean sentinel. If the semantic batch cannot establish a usable class for an `unknown-kind` seed after the single bounded retry, record `parked-class sweep not verified: missing defect_signature.kind for {finding_id}`, set `dispatch: "not_verified"`, and take the not-verified fallthrough. An absent, malformed, evidence-empty, or non-complete envelope is a dispatch failure, never a successful empty result, and enters the bounded retry below. This is deliberately class-primed enumeration, not an independence audit, and its dispatch is not part of the Step 2.6 shadow prompt set. For more than 10 changed files or a parked-class set too large for one prompt, batch per class or by file count as Phase 1 does; record every batch and name any capacity-driven truncation in `parked_class_sweep`—there are no silent caps. A non-null `truncation` is incomplete coverage: record `parked-class sweep not verified: capacity truncation ({details})` and take the not-verified fallthrough. On a transient dispatch failure, re-dispatch exactly once as Step 3.5 does. If it still fails, record the distinct Reflection bullet `parked-class sweep not verified: {cause}`, set `dispatch: "not_verified"`, and take the completeness gate's not-verified fallthrough; never convert it into a clean run.

**Registration, identity, and severity.** Append every discovered sibling to the triggering iteration’s `phase3_findings` with its assigned severity and full `defect_signature` before the shadow runs. This registration makes a shadow re-derivation overlap under Phase 3.2, preserves Decide outcome 1's subset test, supplies the corroboration carve-out's severity comparand, and keeps an at-or-below re-raise from spuriously entering Decide outcome 2. A sibling inherits its source finding's original engine severity by default; the semantic dispatch may grade it differently, after which Step 2.5 and Step 3 item 2 calibrate normally. For sibling dedup and the not-re-swept ledger, identify sites by Phase 3.2's file plus overlapping `line_range` against registered findings and earlier sweep entries. Treat `kind` as a matching input, never as cross-producer identity, because free-text labels drift; record the exact `kind_literal` swept for each class.

**Routing and termination.** A discovered sibling at or above `$FIX_THRESHOLD` enters Step 2.5 → Step 3 as a promoted iteration using the same machinery as Decide outcome 2. This route is identical at both firing sites: Phase 1+2 stay skipped, Step 0.9 short-circuits, the promoted iteration counts toward `$MAX_ITERS`, and it is re-shadowed. A below-threshold sibling is parked in `## Advisory Findings` and the final report with an advisory row whose distinct pinned evidence marker is `parked-sibling: class-sweep`; that parker also writes the structured `parking_evidence` object `{basis, failing_input, source, finding_ref}` on the sibling's row at parking time (per its authoritative shape in Persistent workpad → Schema), whose `basis` **names the sweep registration** (it is a rationale-bearing producer for the Park-calibration gate's evidence classification) and whose `finding_ref` is the `{iter, index}` join to the sibling's registered `phase3_findings` record. The not-re-swept ledger keys cross-producer identity on site overlap alone; `kind_literal` remains enumeration input and recorded metadata, never ledger identity. Sites whose overlap already appears in a prior `parked_class_sweep` block **whose `dispatch` was `verified` and whose `truncation` was `null`** this run are not re-swept (`truncation` is an independent axis from `dispatch` — a capacity-truncated batch legitimately carries `dispatch: "verified"`, so keying re-sweep on `dispatch` alone would strand it); sites recorded under a `not_verified` (or capacity-truncated) prior sweep are NOT treated as already-swept and are re-examined at re-convergence, so a transient/terminal dispatch failure is never laundered into permanent silence. At re-convergence sweep only newly parked sites plus any such non-verified carry-forward sites. The sweep itself is uncounted, like the shadow; only a spawned promoted iteration counts, while `$MAX_ITERS` and skip-twice escalation bound the loop.

At the iteration cap the sweep still runs and registers every sibling, and the tentative verdict remains inside the convergence-time trigger's enumerated list so the shadow runs normally. Decide outcome 1 is barred while any unfixed at-or-above-threshold sweep-registered sibling exists. Decide then surfaces `APPROVE WITH UNRESOLVED SHADOW FINDINGS` (or REJECT for Critical) post-shadow, never an advisory note. For this sweep-at-cap arm, the chat headline assertion, Coverage selection, and post-shadow edit gate read the **current** triggering iteration's shadow block rather than the ordinary one-iter-back promotion block.

**Record.** Write the schema's `parked_class_sweep` block with the Write tool as part of the convergence-point emit: source finding ids, per-class kind literals, sites examined, new sibling ids/sites/severities/dispositions, dispatch status, and any deliberate truncation. Record one `## Devflow Reflection` bullet per run: per-class results when siblings exist, `parked-class sweep clean: no new siblings` when none exist, the not-verified bullet on terminal dispatch failure, or the sanctioned downgrade-path not-applicable sentinel. **A non-null `truncation` overrides the per-class-results branch:** when the sweep was capacity-truncated it records `parked-class sweep not verified: capacity truncation ({details})` even if its completed batches found siblings — the not-verified bullet takes precedence, because a truncated sweep never examined the whole changed surface. These are the completeness gate's evidence operands.
