# Reference: Fixing (Step 3)

### Step 3: Fix Findings

Apply the `devflow:receiving-code-review` principles — including the **share-the-contract / parse-don't-validate** principle (when a fix adds a guard protecting a downstream consumer, prefer the consumer's own operation as the guard rather than a separate validator that re-derives the contract), which Step 3.5's #62/#98 operand-contract check then verifies on the fix delta. After Step 2.5, the findings reaching Step 3 are: Phase 2 checklist FAILs, corroborated Phase 3 findings, confirmed-by-web findings, and codebase-claim findings. Refuted and inconclusive findings have been demoted to advisory and are not in this list; they stay parked.

**Test-first gate at fix-authoring time (before any fix that adds automated-testable behavior is written).** A fix authored code-first ships its own delta untested and encodes whatever the code happens to do, not what the finding requires — the same §2.3 test-first discipline the original diff received applies to the fix delta. **Before you write a fix that adds or alters behavior an automated test can exercise** (a return value, a CLI/API contract, an exit code, a parser's handling of an input shape, a state transition, a raised error, or an end-to-end path an integration test drives), write the test **first**, run it, and confirm it **fails for the right reason** (the behavior does not exist yet), then implement until it passes. **Record the gate's call durably at gate time** — the workpad `--note` when implement-driven (Phase 3.3, `$ISSUE_NUMBER` present); when **standalone**, use the Write tool to create or re-write `.devflow/tmp/review/<slug>/<run-id>/iter-<N>-gate-evidence.json` as a JSON array of `{decision, command, result, evidence}` entries, preserving any entries already recorded there before adding the new logical entry. Item 6's fused `iter-<N>.json` Write consumes that exact scratch file into `test_first_gate` (at gate time, pre-commit, the iteration's durable sink file does not yet exist standalone) — so item 3b verifies an addressable durable record, not conversation memory. This gate carries the same stub-blindness and mutation-check discipline the §2.3 test-first gate and item 4 already state; those are unchanged and not restated here.

- **Ungranted-command degraded arm.** When the test invocation is **permission-denied**, the durable record states `gate: unrunnable (command ungranted on this tier)`, and the remedy is, **keyed on tier first and driver second**: cloud implement tier → grant the command in `devflow_implement.allowed_tools`; cloud command path (`devflow.yml`) → grant it in `devflow.allowed_tools`; local tier, regardless of driver → the operator adds the command to their local agent allowlist settings (in this repository, the `CLAUDE.md` classifier-fallback convention — cited as this repo's instance, not as a document a consumer necessarily has). The test is **still written un-run where feasible**, Step 3.5 plus the shadow are named as the backstop, and **a denied command is never routed to the no-automated-test arm** (a permission denial is not "no automated test applies").
- **No-automated-test arm.** When no conventional automated test can exercise the fix, preserve Phase 2.4's verification taxonomy instead of collapsing every prose-shaped deliverable into one static check. For a logic-bearing artifact or human-facing prose, the fixer **executes the Phase-2.4-defined adversarial dry-trace / shape-matrix inline at gate time**, scoped to the fix delta, and records the trace in the evidence sink. **When the fix changes prose that enters a model's context as instruction** — including a `SKILL.md` procedure, an injected prompt block, or an agent brief — run the Phase-2.4 subagent RED/GREEN micro-test, including its no-guidance control, and record those behavioral results in the evidence sink; model-context instructions are never discharged by a dry-trace. If that micro-test is unavailable, unusable, or fails, record the attempted command/dispatch, `result: "unrunnable"` or `result: "fail"`, and the concrete reason in the same evidence sink; the finding remains unresolved and blocks convergence rather than silently falling through to Step 3.5. The **blinded Step 3.5 gate is cited only as the independent backstop, never as the discharge** — the fixer runs the applicable Phase 2.4 verification itself and does not hand the obligation to Step 3.5.

1. **Read all findings** without reacting. Understand the full picture before fixing anything.

2. **Evaluate each finding technically:**
   - For verification checklist FAILs: Read the evidence. Verify it yourself by reading the source file cited. If the evidence is correct, fix the code. If the evidence is wrong (the verifier misread the source), skip the fix and document why.
   - For Critical/Important findings from review agents: Read the finding. Check if it's valid for this codebase. If valid, fix it. If not, skip and document why. (Note: external-tool claims that survived Step 2.5 are already either web-confirmed or corroborated by ≥2 agents — be slow to dismiss them as invalid.) **Also calibrate its severity, not just its validity** — applying the receiving-code-review symmetric-severity-calibration principle, evaluate the finding's emitted `Critical`/`Important` grade against its observable fail-direction and impact (does it fail closed / does the suite catch it RED / is its only impact diagnostic?). Record that technical evaluation as a structured `fix_decisions` entry (`decision: "severity-calibrated"`, with `original_severity` / `calibrated_severity` and the observable fail-direction/impact as `evidence`). This recorded evaluation **is** the evidence the **Over-grade calibration gate** (Step 2.6) requires before a promote-path `Critical`/`Important` finding may drive a Decide-outcome-2 promotion — the gate *enforces* what the principle *states*; a promote-path finding with no such recorded evaluation blocks at that gate as non-convergence.
   - For Suggestion/Minor findings: Fix only if trivial and clearly correct. Do not spend time on cosmetic issues.
   - **For a finding that prescribes changing existing _documented_ behavior** (a reclassification, a "should be X not Y", or a contradiction of a documented invariant or taxonomy): treat it as a **recorded verify-step before you apply it** — read the doc, comment, or spec the prescription cites or would change, and **verify the prescription against its cited source of truth** before applying it. This applies the receiving-code-review verify-before-implementing principle to a reviewer's reclassification: a confidently-stated grade or relabeling is a *claim*, not a fact, until checked against the codebase's own documented design. Record the verification as a structured decision the same way item 5 records a pushback — a prescription that **contradicts** the source of truth is recorded as a pushback (`skip_category: claim-quality`, `evidence` citing the documented invariant it contradicts) and is **not** applied; a prescription **consistent** with the source of truth is applied, with the verifying source cited in the `fix_decisions` entry. Do not apply a reclassification that contradicts the documented design just because the reviewer stated it with confidence. (The #160 example: a reviewer prescribed relabeling the `severity-calibrated` outcome as `noise`, which contradicted `lib/efficiency-trace.jq`'s documented taxonomy reserving `noise` for false positives and `null` for a real-but-unapplied finding — verifying against that source of truth turns the prescription into a pushback, not an applied change.)

3. **Fix one issue at a time — then fix its whole class, not just the reported instance.** After each fix, verify the surrounding code still makes sense. Then generalize the finding to its `defect_signature.kind` and scan the **changed surface** (the files/hunks in this PR's diff, plus any file your fix just touched) for other instances of the same class; fix every sibling in this **same** iteration, applying item 2's severity triage to each. A reviewer reports the one instance it happened to land on, but a bug class usually recurs across the diff (e.g. one unchecked-type jq op flagged → sweep the program for every unguarded `startswith`/index/`// ""`-on-maybe-non-string op). Catching siblings here, deterministically, is far cheaper than letting the next Step 2.6 shadow rediscover them one at a time — each shadow that promotes a missed sibling costs a full extra engine pass plus a promoted fix iteration (see the Cost note under Step 2.6). **Bound the sweep to the PR's own changed surface — never to pre-existing untouched code** (that is what the `out-of-scope` skip and the Loop Exit widens-surface guard are for); a sibling you deliberately choose not to fix is recorded through the same pushback flow as any skip (item 5). This fix-triggered sweep remains unchanged; the pre-shadow parked-class sweep handles the complementary no-fix convergence shape.

   **3a. Mechanism-scoped self-authored-claim re-sweep — when a fix changes a *mechanism* (a guard, predicate, exclusion pattern, or helper that existing comments describe — *and equally a job-gating or control-flow change: a changed `if:` condition, a moved conclusion arm, a rerouted step, a relocated where-a-decision-concludes*), re-verify every comment describing that mechanism, located by the mechanism's identifiers across the touched files — not limited to the fix's own diff hunks.** A change that re-gates *where* work happens — moving which job/branch reaches a conclusion, flipping an `if:`, or rerouting a step to a different arm — is a mechanism change even when no single identifier was renamed, and comments describing the old routing go stale exactly as a renamed predicate's do (the #304 iteration-1 fix redesigned job gating so a deferral concluded in one job and another was skipped, yet two comments describing the old routing survived to iteration 5 because the change did not read as a "mechanism" change). A fix that tightens a guard, narrows an exclusion, renames a predicate, moves a helper's contract, or reroutes control flow / job gating leaves any comment describing the *old* mechanism stale, and a stale comment usually sits *away from* the changed line — at the function header, in a block comment, in a sibling helper — so spot-checking only the fix's own hunks misses it. This is the same self-authored-claim drift the implement skill's Phase 2.3.4a sweep targets, re-scoped to a mechanism a fix just changed; run it after item 3's class-sweep, once per fix that changed a mechanism:
   - **Identify the mechanism's identifiers** — the function / variable / flag / pattern name(s) the fix changed (the renamed predicate, the tightened exclusion's variable, the helper whose contract moved).
   - **Locate every comment describing the mechanism by those identifiers across the touched files.** `git grep -n` the identifier(s) in the files this PR's diff touched and read the comment at each hit, *plus* any block / header comment in those files that names the mechanism. This search is **identifier-located, not hunk-located**: a comment that names the mechanism is in scope even when the fix did not edit its line.
   - **Re-dispatch `devflow:comment-analyzer`** scoped to those comments — pass it the touched file paths, the mechanism's identifiers, and the instruction to verify each comment describing that mechanism against the *post-fix* behavior. This reuses the existing review agent: **no new agent is introduced.** The agent analyzes and reports only — it never edits — so this advisory dispatch does not violate the rule that fixes are applied directly and never delegated to a subagent; the orchestrator applies any resulting correction itself.
   - **A comment that still describes the pre-change mechanism is a finding** — fold it into this same iteration (fix the comment alongside the mechanism change, as a swept sibling under item 3) or, if deliberately left, record it through the item 5 pushback flow.

   **3b. Fix-delta authoring-side sweeps — on every iteration in which Step 3 applied fixes, run the implement Phase 2.3 authoring-side sweeps against the fix delta.** A fix is new code and new prose that never received the authoring-side sweeps the original diff did, so nearly half a run's defect load can be authored inside the fix loop on a surface no sweep ever examined and left for the blinded machinery to rediscover one at a time at a full engine pass each. This item is the author-side counterpart of the blinded Step 3.5 fix-delta verification gate: it front-loads what that gate would otherwise rediscover, and the Step 3.5 gate itself is unchanged (this item never alters it). Scope every sweep to the fix delta — the code and prose *this iteration's* fixes add and change, the same changed surface item 3's class-sweep bounds itself to, never pre-existing untouched code.

   **Re-anchor onto the full §2.3 index.** Classify the fix delta exactly the way Phase 2.3's **Sweep selection (run first)** preamble classifies a diff, and **run every Phase 2.3 sweep the classification warrants** — the trigger-gated sweeps whose triggers the delta matches **plus every sweep the §2.3 preamble names as always-on** (its **Always** entries, cited by reference, never re-counted here) — **each gated by its own Phase 2.3 trigger condition** exactly as that sweep's definition states it, so **a sweep whose trigger the delta does not match runs not at all**. This item carries **no hand-enumerated sweep subset**: the §2.3 index in the executing bundle's `../implement/phases/phase-2-implement.md` sibling is the single source, so a sweep #474 or any future change adds to that index reaches this loop with no edit here — sweep **selection** inherits through this re-anchor. Record the diff shape you classified and the sweeps you are running in the iteration's evidence sink — an auditable commitment a reviewer or the weekly retrospective can check, exactly as the §2.3 preamble's own selection-note rule requires; a "ran just the always-on five" record on a delta that in fact tightened a guard is a visible error, where an unrecorded mental skip is not.

   **Read protocol (durable operand — bounds the context bill, preserves the heading-wins fail-safe).** Resolve the source from this executing skill's portable skill-directory anchor as `../implement/phases/phase-2-implement.md` (the repo-own checkout and installed/vendored bundle share that sibling layout; never resolve it from the consumer repository's current working directory). On the run's **first fix-applying iteration**, read that §2.3 preamble and the full sweep definitions from the resolved source, and **record a `sweep_defs_read` list** — the source path plus the sweep identifiers read — in the iteration's evidence record. A read counts as complete only when the Read operation succeeds through EOF and the returned content contains both the §2.3 sweep-selection preamble and the later `### 2.4 Test` boundary; an empty, truncated, denied, or marker-incomplete result takes the unreadable-source arm below and is never recorded as a completed definition read. On each subsequent fix-applying iteration, re-read the preamble and **every sweep heading** (so a sweep the index forgot still announces itself through its own authoritative heading — the §2.3 fail-safe-by-headings property operating inside this loop), then read the full definition of any warranted sweep absent from the recorded `sweep_defs_read`. The decision is **adjudicated against that durable record, never against recall**: a warranted sweep **absent from that record** — the post-compaction shape, where the read *content* is gone from context but the durable record is not — forces the full re-read rather than the agent self-diagnosing its own compaction, and **a sweep is never executed from recall alone**. When the §2.3 source **cannot be read completely** — a partial-copy deployment, an unresolvable skill anchor, a denied read, or a truncated/marker-incomplete result under the completeness check above — the loop does **not** stall and does **not** silently skip: record `sweeps: unrunnable (phase-2 source unreadable at <path>)` in the iteration's evidence sink, name **Step 3.5 and the shadow** as the covering backstop for this iteration's delta, and continue the loop. A degraded environment yields a visible degradation record, never an undistinguishable clean pass.

   **Preserved invariants.** Every sweep is scoped to the fix delta (never pre-existing untouched code); each sweep is gated by its Phase 2.3 trigger (an unmatched trigger means that sweep does not run); each triggered sweep honors its own Phase 2.3 evidence obligation exactly as its definition states it, with that evidence directed to the loop's own `iter-<N>.json` iteration records when the loop runs standalone with no issue workpad to write to. **A finding any of these sweeps surfaces folds into this same iteration** exactly as item 3's class-sweep siblings do — fixed in-iteration, or, if deliberately left, recorded through the item 5 pushback flow.

   **Fix-loop context mapping table (imported obligations resolve to fix-loop destinations, drift-guarded).** The imported §2.3 sweep bodies were written for the implement context, so their evidence obligations and routing arms name implement-only destinations.    **Command forms (ignore-aware, in order).** The imported sweeps' grep-evidence obligations were written against `git grep` (tracked files only), and `git grep -n` is ungranted in both cloud allowlists. Use the **Grep tool first** (ignore-aware, always present in the runner), `rg` when it resolves on the host, and **`grep -rnE` last, and only with an explicit obligation to exclude VCS, dependency, and build directories**, to locate qualitative matches — **never `git grep -n`**. For quantitative sweep match-count evidence, preserve the original tracked-file operand explicitly: enumerate it with `git ls-files -z` and pipe that NUL-delimited list through granted `xargs -0` into `rg` or, when `rg` is unavailable, `grep -nE`; a recursive-search count that also admits untracked or ignored files does not discharge the obligation. Match-count evidence is therefore over **tracked/non-ignored files**, and an ignore-blind count over `node_modules/`-class trees is noise rather than an every-site-reconciled termination condition. These command forms are already granted, so this change makes **no `TOOLS='…'` line change and no `devflow_implement.allowed_tools` change**.

   **Termination.** A sweep-surfaced fold-in edit is **not re-swept within the same 3b pass**; any residue rides the same iteration's **Step 3.5 cumulative-delta review** — fold-ins included, since that gate spans this iteration's first-fix parent through HEAD — and the convergence shadow's whole-diff audit. Step 3.5-routed inner re-fixes **run only the trigger-gated sweeps their re-fix delta matches** — a stated, decided narrowing of §2.3's Always rule for the inner path alone: the trigger-gated set covers the evidenced fix-defect classes (coupled-mirror prose via §2.3.0/§2.3.0a's substrate-agnostic triggers; hasty guard rewrites via §2.3.0c), the inner delta (at most two commits) is the only surface immediately re-reviewed by the blinded Step 3.5 re-gate and also rides the shadow, and running the always-on battery per inner attempt would multiply per-iteration sweep cost up to threefold in the loop's one deliberately surgical path for minimal marginal coverage. The inner path skips the per-iteration re-read only; the `sweep_defs_read` record governs its reads exactly as the main path's (an absent record still forces the re-read before sweeping).

   **Test-first-gate verify half.** Step 3's fix application runs the §2.3 test-first gate at fix-authoring time (see Step 3's **Test-first gate** block above); item 3b **verifies that the gate's durable gate-time record is present** for any fix in this iteration that added automated-testable behavior — the workpad `--note` when implement-driven, the scratch iteration-evidence append that item 6's fused `iter-<N>.json` Write consumes when standalone — treating an absent record as a finding folded into this iteration or recorded through item 5, never trusting conversation memory.

4. **Run tests** after all fixes. Check CLAUDE.md, README, or project configuration for the project's test and lint commands. If tests fail, fix the test failures before continuing.

   **Record the suite run as `verification_evidence` in this iteration's workpad.** Capture the suite command you ran, its `result`, and how long it took, and carry them into the `iter-<N>.json` record as a `verification_evidence` object — `command` (the exact command string), `result` (`pass` when the suite passed, `fail` when it failed, or `skipped` **with a `reason`** — the reason arm covers a no-fix iteration with nothing to re-run and a cloud run whose suite command is ungranted per the tooling-gap constraint), and `duration_s` (elapsed wall-clock seconds, best-effort). This object rides the **existing item-6-anchored emit** (it is written as part of the same `iter-<N>.json` Write in item 6/7, not a separate write). One caveat, recorded in the field's spec: Step 3.5's inner re-fix attempts commit **after** this recorded suite run, so the evidence reflects the pre-3.5 state — the field records what was actually run this iteration, never a guarantee about later commits. **Consumption is deferred:** this field is captured in the workpad only — `lib/efficiency-trace.jq` does not propagate it into the per-run efficiency record, so it does **not** reach `experiment-records.jsonl` today. It is forward-looking capture for the cross-run analyzer; do not read its presence here as evidence that a downstream consumer already reads it.

   **A test suite can *pass while a check inside it self-skipped*, and that is not a clean pass — record it, do not launder it.** A test runner may report a check as **skipped**: it neither passed nor failed because the environment could not run it or express its condition, so a summary line like "N passed, 0 failed" says nothing about whether that check actually ran. When the suite's own output reports one or more skipped checks, the `verification_evidence` object also carries a **`skipped_checks`** list — one entry per skipped check, each naming the **check**, its **kind** (whether it was a real gate that should have run here but could not, versus a check the host cannot express), and the **reason** the runner gave. This is a **different fact** from `result: skipped`, which means *the whole suite run was skipped* (a no-fix iteration, or an ungranted suite command); leave the `result` enum unchanged and put the per-check information in `skipped_checks`. Detect skips from **the runner's own reported output** — the summary tally and the per-check skip lines (or machine-readable report) the runner itself prints — never by inference from the exit code, which a skipped check deliberately does not change. **A run whose `skipped_checks` is non-empty is not reported by the loop as a clean pass:** name the skipped checks in the loop's summary instead of recording an unqualified pass, because a skipped check may be a gate that never actually ran on this run. (This wording is deliberately runner-agnostic — a consumer's own test runner reports skips too; it names no specific command or flag.)

   **Single-flight verification coordination (issue #528).** Resolve the gate the same way every other config gate in this engine is resolved — `"${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../scripts/config-get.sh .verification_flight.enabled true` — which **defaults `true`**; only an explicit `false` disables it, and a resolver failure (rc≠0) warns and takes the default. When it resolves enabled, coordinate this same-checkout verification launch through the non-executing `verification-flight.py` ledger (it never launches the command — invoked by the same `"${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../scripts/` anchor the other helpers use) rather than relaunching unconditionally. `claim` first (data-only, launches nothing). **The owner token is returned once and never persisted in cleartext, so it must survive between the three calls:** `claim`, `mark-running`, the verification command, and `finish` are separate invocations, and a shell variable does not survive from one tool call into a later one — capture the token from `claim`'s printed JSON and carry it in the prompt context (or write it to a run-scoped scratch file) before launching the command. A lost token makes `mark-running`/`finish` fail CAS with `token_mismatch`, which silently degrades this run to an uncoordinated direct launch: treat that rejection as a coordination failure to record, not a no-op to ignore. If it returns role `owner`: `mark-running`, run the existing already-authorized verification command **unchanged** (same command, same allowlist), then `finish --result passed|failed|timed_out|cancelled` with a summary file carrying `command`/`exit_status`/totals/`failure_text`/`skipped_checks`. If it returns role `attacher` and `status` (or a bounded `wait`) shows a `passed` flight for the **identical** descriptor+checkout, consume that terminal evidence as this iteration's `verification_evidence` instead of relaunching. **Unknown is not a pass:** a missing, partial, timed-out, unreadable, stale, or incomplete handle NEVER counts as a pass — fall back to a direct launch, never a relaunch the ledger authorized on its own. On `wait_expired`: a standalone Review-and-Fix run records `verification_evidence.result: skipped` with reason `flight_wait_expired`, follows its locally-unverified terminal arm, and cannot report a clean pass; an implement-driven run takes the existing Blocked path without relaunch. Persist the flight handle + terminal evidence in this iteration's `iter-<N>.json` (it rides the existing item-6-anchored emit). The existing suite failure text, exit status, pass/fail/skip totals, per-skip details, and loop-exit verification rules retain their meaning.

   **For any added or edited test guard in the diff** (a new regression test, a coverage check, a drift/sync assertion — whether it is the fix-loop's own deliverable or primary implementation work the diff introduces), a green suite is necessary but **not sufficient** — a *vacuous* guard passes too. **Mutation-check it:** break the thing it is supposed to catch (delete the line/block it pins, flip the condition) **on a copy of the file — never edit the working-tree file in place** — and confirm the guard goes **RED** against that copy, then discard the copy. A copy-based mutation-check is never destructive: an interrupted or crashing check can never leave the real working-tree file broken, whereas an in-place "break then restore" can. In particular, never restore with `git checkout -- <file>`: it cannot restore an untracked file and silently appears to succeed, leaving every mutation in place. A guard you never saw fail may assert nothing — e.g. a substring `grep` that also matches unrelated text elsewhere in the file, so it stays green even when the guarded content is gone (pin a *target-unique* phrase instead). This is TDD's red-before-green applied to the guards you add while fixing; skipping it is how a "fix" ships a test that protects nothing. Two further requirements on any guard you add: **bake the half-revert into the suite** — express the pin through **`assert_pin_red_under`** (the mutation-taking removal-proof assertion in `lib/test/run.sh`, which proves *PASS with the pinned text → FAIL once a `sed -E` mutation removes it*) rather than running the half-revert once by hand, so it re-runs on every suite execution; and **confirm the guard registered** — that its named assertion appears in the run as a PASS and the suite's assertion count rose by what you added. A green suite alone does not prove a guard ran: one that silently no-ops (an assertion helper invoked before it is defined → command-not-found, a test file the runner never sources, a setup probe that returns success on failure) asserts nothing while the suite stays green.

   **When the guard you add is a *behavioral-fix pin*** — a coverage pin you add *specifically because* removing the pinned text would re-introduce a named bug or regression — the same operative-vs-framing discipline applies: pin the **operative sentence** (the minimal text whose removal alone re-introduces the bug), not an adjacent framing clause, and **express the pin through `assert_pin_red_under`** (the mutation-taking removal-proof assertion in `lib/test/run.sh`), passing a `sed -E` **mutation that re-introduces the named bug** by removing only that operative sentence — so a framing-only pin that survives the mutation is reported RED. **Record the evidence, not an attestation:** the workpad `--note` records **the mutation you ran and the pin you observed go RED** under it, replacing the old operative-sentence-substring self-testimony. This is the same auditable per-pin commitment the implement skill's Phase 2.3 behavioral-fix-pin rule requires, re-scoped to any behavioral-fix guard this fix loop adds; it does **not** apply to pins on literal constants, token names, count-based guards, or absence pins, where no operative-vs-framing distinction exists.

5. **Track pushbacks.** For each finding you skipped (whether checklist FAIL or Phase 3 finding), record a structured entry: `{source_file, claim_text, skip_category, evidence}`. **For a Yes-downgrade deferral** (`skip_category` one of `claim-quality` / `out-of-scope` / `already-tracked`) also write the structured `parking_evidence` object `{basis, failing_input, source, finding_ref}` beside the retained one-line `evidence` string, per its authoritative shape (root `### Schema` example; field semantics in `references/loop-control.md` *Schema field semantics*): `basis` is the one-line parking rationale, `source` is the citation the category requires (source span / `git blame` proof / issue reference — a null `source` on one of these categories fails condition (c) at comparison time, since these categories require citations by contract), and `finding_ref` is the `{iter, index}` join to the finding's parking-time `phase3_findings` record and is **always** written (`references/loop-control.md` *Schema field semantics* owns the null-is-a-writer-omission → fail-closed semantics). The retained `evidence` string is unchanged (the `deferrals.json` `explanation` input); the `uncategorized` and sweep-only `below-threshold` rows are not Yes-downgrade deferrals and gain no `parking_evidence`. `skip_category` MUST be one of the values defined in the **`skip_category` enum (authoritative)** block below — this is the single source of truth for the enum, referenced by both this step and the Loop Exit Pre-mapping gate. Adding a new category requires editing only this block; both consumers read from it.

   #### `skip_category` enum (authoritative)

   If the same `(source_file, claim_text)` pair was also skipped in the previous iteration, escalate to the user: "Finding persists after pushback: {claim}. Manual review needed." and stop the loop.

6. **Commit fixes** before re-running the review. **Stage only the specific files this iteration's fix touched, by explicit path** — `git add <file1> <file2> …`, listing every file the fix changed (one or many; run `git status --porcelain` first to enumerate them, including any new untracked files a plain `git diff` would miss). The fix-commit-fused Write below then records exactly this committed set as `fix_files` via `git diff --name-only HEAD~1 HEAD` (its full record shape is specified in item 7) — that derivation is a post-commit *record* of what you staged, not a pre-commit source to stage from, so enumerate from the working tree, not from `fix_files`. **Never use `git add -A` or `git add .` at the fix-commit step:** an unscoped stage sweeps unrelated working-tree state — a local `.devflow/config.json` edit, stray debug-capture files, leftover temp artifacts — onto the feature branch, where the blinded shadow then flags it as a Critical that costs an extra fix iteration to revert.
   ```bash
   git add <file1> <file2> …   # the specific files this fix touched — never `git add -A` / `git add .`
   git commit -m "fix: address review findings (iteration {N})"
   ```
   **Capture the fix commit's SHA, then Write `iter-<N>.json`, as one step — the workpad emit is fused to this fix-commit moment.** Run `git rev-parse HEAD` to capture `fix_commit_sha`, and **immediately Write `iter-<N>.json`** (with the Write tool, never a shell `>`/heredoc redirect) carrying the full record shape specified in item 7 below — do not defer this Write to a later "persist" step. Fusing the Write into this commit moment is what makes the emit unloseable on an inline-driven loop: committing the fix and recording the iteration are one atomic step here, with no seam between "fix landed" and "record exists." (`fix_commit_sha` is read by Step 0.9 of iter-(N+1); item 7 remains the authoritative specification of every field the record carries.)

   **If `--push-each-iteration` is set**, bring the branch up to date with the base and push in one step via the shared checkpoint helper (**base-branch update Checkpoint 3**, issue #448) — run it **immediately before** the push, so the helper's own push carries this iteration's fix and the base merge together (one `synchronize` event, and the pushed state the auto-review evaluates is current). Invoke the helper **alone** in its fence, and **gate the push on the token it prints** (below) — never chain a bare `git push` after it in the same fence, which would push on *every* token, including the two that prescribe a stop:
   ```bash
   "${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../scripts/update-branch-checkpoint.sh
   ```
   Handle the helper's printed token by the **context split**: inside a `/devflow:implement`-driven loop (Phase 3.3, where the orchestrator holds `$ISSUE_NUMBER`) record outcomes on the **issue workpad** and take the **Blocked** hard stop on `MERGE_IN_PROGRESS` / a failed conflict resolution, exactly per phase-1-setup.md §1.4.1; in a **standalone** run there is no issue workpad, so record the outcome in this iteration's `iter-<N>.json` and, on `MERGE_IN_PROGRESS` or a failed conflict resolution, take the loop's native **"stop and report"** (report in chat and end the loop) instead of a workpad flip. **In *both* contexts also write the `base_checkpoint` field to this iteration's `iter-<N>.json`** — it is the *named, durable operand* the Loop-exit suite rule in `references/loop-exit.md` reads, so it is never context-conditional: an implement-driven iteration that recorded its outcome only as workpad prose would leave that rule with no comparand and it would silently fail open (see item 7, which enumerates the field).

   **Split the token when you write it — do not store the helper's printed line verbatim.** The helper prints `UPDATED <behind>` (e.g. `UPDATED 3`) as a *single* stdout line, so a record that stores that line whole gives `base_checkpoint.token == "UPDATED 3"`, and the Loop-exit rule's `token == "UPDATED"` test is then **false for every real merge** — silently re-opening the exact fail-open this field exists to close. Write `token` as the **bare token word** (`UPDATED`, never `UPDATED 3`) and carry the count in `behind_by` (an integer for `UPDATED`, `null` for every other token): `{"token": "UPDATED", "behind_by": 3}`.

   **Write the record AFTER this step, not before it — and after item 6a.** The `iter-<N>.json` Write mandated at the fix-commit moment above happens *before* the checkpoint runs, so `base_checkpoint` does not exist yet at that point; item 6a's `stale_prose_check` does not exist yet either. **Execution order within this item is: fix commit → item 6a's stale-prose pre-check → this checkpoint + its push-gating → re-Write `iter-<N>.json` with BOTH `base_checkpoint` and `stale_prose_check` included.** (Item 6a is *printed* below this block but is *anchored* to the same fix-commit moment, so it runs before this re-Write; the re-Write is the single point at which every post-commit field lands, so no third Write is ever needed.) The fix-commit-anchored Write is what makes the record unloseable; this re-Write is what makes it *complete*. An orchestrator that writes the record once, before the checkpoint, and never re-writes it emits a record with no `base_checkpoint` at all — which is exactly the absent-comparand shape the Loop-exit suite rule in `references/loop-exit.md` must fail closed on.

   **Push gating — only these tokens push, and the fence above never pushes on its own:**
   - `UPDATED` — the helper **already pushed** (fix + base merge together). Do **not** push again.
   - `UP_TO_DATE` / `DISABLED` — the helper did not push, and the tree is in a good state: push now to propagate the fix commit.
   - `UNVERIFIED` — nothing was merged and the tree is untouched, so the fix commit still needs propagating: record the breadcrumb, push, and continue.

   **Push the same way the helper does — never a bare `git push` on an unverified checkout.** Under `push.default=simple` a bare `git push` **refuses** both when the branch has no upstream *and* when its upstream's remote ref is named differently from the local branch (`fatal: The upstream branch of your current branch does not match the name of your current branch`) — the shepherd-worktree shape, where a checkout named `worktree-pr-N` tracks a PR head `issue-N-…`. Push to an **explicit destination ref** instead (`git push origin HEAD:refs/heads/<the PR's head ref>`), exactly as `update-branch-checkpoint.sh`'s own `_do_push` resolves it. A bare `git push` here fails the propagation *loudly*, so it is not a silent-corruption risk — but it strands the fix commit unpushed on precisely the checkouts this loop most often runs in.
   - `PUSH_REJECTED` — the helper already attempted the push twice and restored the branch; do **not** push again (a third attempt fails identically). Record and continue — **unless** the helper's stderr carries the failed-restore `WARNING` (see the `PUSH_REJECTED` caveat in phase-1-setup.md §1.4.1), in which case the tree may still carry an unpushed base-merge commit and the run **hard-stops** per its context rather than continuing.
   - `CONFLICT` — do **not** push the unresolved tree. Resolve the conflicts, run the project test suite, `git add`+`git commit`+`git push`, record the conflicted files, and re-run the changed-contract sweep before continuing the iteration; a resolution the suite **fails** is aborted (`git merge --abort`) and the run hard-stops per its context. This suite launch is coordinated by the same single-flight helper (see the Single-flight verification coordination note in Step 3's test-run step).
   - `MERGE_IN_PROGRESS` — nothing was touched and the run **hard-stops** per its context. Do **not** push.

   Track that an `UPDATED` merge landed this iteration by that `base_checkpoint` field (for the Loop-exit suite rule in `references/loop-exit.md`). When the flag is absent (the default), **skip both the checkpoint and the push entirely** — a direct invocation without the flag never touches the base.
   If the push fails (no upstream, rejected non-fast-forward, blocked by policy), do **not** abort the loop — report the failure in chat and continue. Pushing propagates fixes to the remote (CI, crash-durability); it is not what makes the next iteration see them (Step 1's head-override handles that), so a failed push never breaks convergence.

   **6a. Stale counted-prose pre-check (the same helper the engine runs in Phase 0.6).** Immediately after capturing `fix_commit_sha`, run the *same* `stale-prose-lint.py` the shared review engine runs in its **Phase 0.6** — here as a fast local pre-check that **recomputes the full branch diff at the just-committed (post-fix) `HEAD`** against the exact base operand the engine's Phase 0.2 resolved this iteration. In PR mode carry the held `$HEAD_OVERRIDE_BASE`: normally the fetched tip of `origin/$PR_BASE_BRANCH`, or the retained `$PR_BASE_SHA` only after Phase 0.2 authoritatively confirmed the named base was deleted. In current-branch mode `$PR_BASE_BRANCH` is unset, so resolve `$BASE` via the guarded `config-get.sh .base_branch main` capture (defaulting to `main` on an empty or failed read — the same guarded-capture shape Phase 0.2's current-branch fence uses; `scripts/update-branch-checkpoint.sh` shares the capture idiom but, because it merges and pushes, stops with `UNVERIFIED` on a hard read failure rather than defaulting — this read-only pre-check defaults instead, and its downstream `git diff` still fails closed if the resolved base is unreachable). The fence keys on `$HEAD_OVERRIDE_BASE`'s presence — carrying that operand is sufficient, and it is honored even if `$PR_BASE_BRANCH` was not threaded into this separate shell; `$PR_BASE_BRANCH` only distinguishes a PR-mode iteration whose operand was **not** carried (fail closed, `exit 65`) from a genuine current-branch run, so the fence never silently falls to the current-branch base when a PR operand was carried. The diff remains three-dot, so it is the branch's own changes since it diverged from the base. Hold that selected operand as `$REVIEW_DIFF_BASE`; **never independently reconstruct a PR ref after Phase 0.2 has selected its deleted-base fallback, and never substitute the stale run-start SHA on a live base**. Recomputing at the post-fix `HEAD` — **not** reading the pre-fix `diff.patch` Phase 0.2 cached before this iteration's fix commit — is what puts a stale claim *this iteration's fix* just introduced into scope, including the PR #328 shape (a claim falsified under a header added by an **earlier** branch commit: that earlier-added header is still present as an added line in the full-branch `$REVIEW_DIFF_BASE...HEAD` diff, so it is examined even though this iteration's fix touched only the block line under it). A fix-introduced stale counted claim is thus reconciled **within the same iteration**, before it costs a full engine pass to detect. The helper reads the diff on stdin and resolves referents at `--rev HEAD`; it never derives the diff range itself — the fence supplies the range:
   ```bash
   # BEGIN CURRENT_BRANCH_BASE_CAPTURE
   if test -n "${HEAD_OVERRIDE_BASE:-}"; then
     # PR mode with Phase 0.2's operand carried (the prose-instructed path): use it
     # directly. Keying on the OPERAND's presence — not the PR_BASE_BRANCH mode-signal —
     # means a carried operand is honored even if PR_BASE_BRANCH was not threaded into
     # this separate shell, closing the fail-open where the fence would otherwise fall to
     # the current-branch base and silently re-scope the diff (issue #503 via a side door).
     REVIEW_DIFF_BASE=$HEAD_OVERRIDE_BASE
   elif test -n "${PR_BASE_BRANCH:-}"; then
     # PR-mode intent is signaled (PR_BASE_BRANCH present) but Phase 0.2's operand was NOT
     # carried: fail closed rather than silently fall back to the current-branch base.
     echo "::error::devflow review-and-fix: Phase 0.2 did not carry its resolved PR base operand; stale-prose pre-check cannot run" >&2
     exit 65
   else
     if ! BASE=$("${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../scripts/config-get.sh .base_branch main); then
       echo "::warning::devflow review-and-fix: could not read .base_branch (config-get.sh rc≠0); falling back to 'main'" >&2
       BASE=main
     fi
     if test -z "$BASE"; then
       echo "::warning::devflow review-and-fix: .base_branch resolved empty; falling back to 'main'" >&2
       BASE=main
     fi
     REVIEW_DIFF_BASE=origin/$BASE
   fi
   # END CURRENT_BRANCH_BASE_CAPTURE
   set -o pipefail
   git diff "$REVIEW_DIFF_BASE...HEAD" | "${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../scripts/stale-prose-lint.py --rev HEAD
   ```
   With `pipefail` set the pipeline's exit code routes exactly as Phase 0.6 routes on the helper's own: `0` (no STALE), `1` (a STALE row present), and `2` (helper internal error) are the helper's arms. **Any other non-zero code is a producer-or-helper failure** — a `git diff` that could not resolve the base (`git diff` exits `128` on an unresolved base ref / detached-HEAD edge, piping empty stdout into the helper), or the helper itself failing to launch (`127`, or a signal death — the harness-refused / helper-absent arms Phase 0.6 defines). Record it as a degradation note carrying **the observed exit code** (`stale-prose pre-check: producer/helper failed (rc=<code>)`) and route per Phase 0.6's arms; do **not** read the empty output as a clean pass or hard-assert `git diff` as the culprit when the code is not `128`. Record the invocation outcome (the TSV rows, the producer/helper-failure note, or the helper's degradation arm) in this iteration's `iter-<N>.json` under a `stale_prose_check` entry. Reconcile any `STALE` row in this same iteration (fix the counted claim). A STALE row you judge a **false positive** is recorded through the existing `fix_decisions` pushback shape (`skip_category` plus evidence), exactly like any other skipped finding. A STALE row that survives unresolved to the `$MAX_ITERS` cap follows the existing `APPROVE WITH UNRESOLVED SHADOW FINDINGS` path — this pre-check adds **no new verdict rule and no new iteration-accounting rule**. `UNRESOLVABLE` rows are informational and never gate. The rule classes, the config gate, and the degradation arms are defined once in the engine's Phase 0.6 and the helper's own header; this step invokes that helper and does not restate them.
