## Prerequisites

If `$ARGUMENTS` is empty, ask the user to describe their user story, bug report, or feature idea before proceeding.

## Core principle

**An issue is the output of resolved decisions — not a place to park unresolved ones.**

Every decision a developer would otherwise have to guess at MUST be resolved by asking the user *before* the issue is written. Whatever the user genuinely will not or cannot resolve goes into one explicit **Blocked** section — never disguised as an "option", a "recommended approach", an "Open Question", a default, or conditional wording scattered through the body.

## Completion checklist (do this first)

This skill is a **pipeline that ends with a created GitHub issue and a *gated* offer to start implementation** — the offer is presented, or a one-line reason is printed for why it was withheld — not with a documentation report. Before doing anything else, create a TodoWrite todo list with exactly these items:

1. Run `/devflow:docs-verify --report-only` and capture its findings report
2. Clarify the user story until the **Definition of Ready** is met (Step 2)
3. Draft the issue and pass the **no-options gate** (Step 3)
4. Steelman the draft against the code, revise, and re-pass the no-options gate (Step 3.5)
5. Audit the draft in a fresh context, act on the verdict, and re-gate any revision (Step 3.6)
6. Present the rendered issue, get the user's explicit confirmation, then create it (Step 4, sub-steps 1–5)
7. After creation succeeds, run the gated implement-offer step — present the offer, or print the withheld-offer reason (Step 4, sub-step 6)

Mark each todo `in_progress` when you start and `completed` only when done. **The issue is created only after the user explicitly confirms the rendered draft (todo 6) — never before.** A finished `/devflow:docs-verify` report is only todo 1. If the user has not yet confirmed, the pipeline is paused at todo 6, not complete; that is a valid waiting state, not a reason to create the issue anyway. Todo 7 runs only after a successful creation — it is the post-creation hand-off, not a gate on creating the issue.

## Steps

### Step 1: Assess current state (read-only)
Invoke the `/devflow:docs-verify` skill in report-only mode with the topic extracted from the user story (e.g., `/devflow:docs-verify --report-only survey module`).

This verifies internal docs against the code and **returns a findings report** — current behavior, relevant files, and any doc/code drift — **without editing, committing, or pushing anything.**

`/devflow:docs-verify` is a standalone workflow, so it announces its own completion when it finishes. That signal ends *its* report (todo 1), not this skill — keep going to Step 2. Carry the findings, including any drift noted, forward into Step 3.

**Completion-wait discipline (mandatory, mirroring Step 3.6's synchronous dispatch).** The docs-verify findings report must be **complete and captured before the first Step 2 clarification question** — and, on a run so complete it asks **zero** clarifying questions, **before Step 3 drafting begins**. When a runner executes `/devflow:docs-verify` as a subagent, **that dispatch blocks on the completed result**; a **launch acknowledgment is never treated as the findings report**. Do not open Step 2 clarification (or Step 3 drafting) on the strength of "docs-verify is running" — wait for its findings to land, exactly as Step 3.6 waits for the audit subagent's *completed* return before proceeding. This wait exists because clarification questions that arrive before the code findings that ground them interrogate the user prematurely (observed live: questions arrived before the docs-verify subagent had finished).

### Step 2: Clarify until the Definition of Ready is met

**Clarification is the default, not the exception.** Do not assess "is this clear enough to skip" — assess "which Definition-of-Ready facts are still unknown" and ask about those. The only way to skip a question is to already know its answer from the user story or the Step 1 findings.

**Definition of Ready — every item below must have a single, decided answer before you draft:**

- [ ] **Independent derivation before anchoring (run the pass below FIRST, on every run).** You have re-derived the full Definition of Ready from the problem and the Step 1 findings *before* weighing the user's supplied criteria — challenging supplied acceptance criteria on both **completeness** and **correctness** — **written that derivation to its observable artifact (`.devflow/tmp/issue-derivation-<slug>.md`) before any clarification round**, and you drive clarification from the **delta plus conflicts**. The gate before the first clarification round (and the per-round re-check) confirm the file — or, in a read-only sandbox, its visible inline-in-chat stand-in — exists, so a skipped pass is surfaced rather than assumed. See the independent-derivation pass below; it mirrors "Solution-space expansion before convergence," generalized from the implementation-approach fork to the problem, behavior, edge-case, and acceptance-criteria forks.
- [ ] **Problem & beneficiary** — who hits this pain and why it matters. (Not "users want export" — *which* users, doing *what*, blocked *how*.)
- [ ] **Single coherent scope** — the issue is exactly one feature/fix. If the story bundles two or more ("export results *and* notify when ready"), ask the user whether to split; default to one issue per feature.
- [ ] **One decided behavior per fork** — every place the story could mean different things (format, channel, trigger condition, access model, edge-case handling) has *one* chosen answer. No "or", no "either", no "default for now".
- [ ] **Solution-space expansion before convergence (do this BEFORE the implementation-approach fork below).** The user's proposed mechanism is an *input*, not the menu. Before asking the user to pick an approach, independently generate the full space of mechanisms that could solve the stated problem — and in particular ask: *is there a categorically STRONGER class of mechanism than what the user proposed?* Mechanisms differ in **strength of guarantee**, not just shape. A useful ladder, weakest→strongest: (a) documentation/wording that asks a human or agent to remember; (b) an in-process self-check that *warns* when the thing was skipped; (c) a *deterministic* mechanism that runs regardless of the actor's choices (a harness/hook/wrapper that fires after-the-fact, an enforced gate, an idempotent backstop that re-does the work). If the user proposed only (a) and/or (b), you MUST generate at least one (c)-class candidate and put it on the menu. Name the strongest viable mechanism even if the user did not think of it — surfacing it is your job, not theirs. If after genuine effort no stronger class exists (or the stronger classes don't apply to this problem — e.g. a harness fires at the wrong point), say so explicitly in the question rationale rather than silently omitting it.
- [ ] **One implementation approach** — present the expanded mechanism menu (including the strongest class found above), and the user has picked one. You surface each option's **guarantee-strength** and trade-offs *in the question*, not in the issue.
- [ ] **Concrete acceptance criteria** — you can state each as a single unconditional, testable assertion. If an AC would need a conditional ("if links are public…"), the underlying fork isn't resolved yet — go ask.
- [ ] **Known prerequisites (cross-issue ordering)** — candidate prerequisite issues (another issue/PR that must land before this work can *start*) are inferred from the story and the Step 1 findings, and **when at least one candidate exists** they are confirmed with the user via the runner's user-question tool. A story with **no** inferable prerequisite gains **no new question** — like the visual-specification path, this item adds a question only when it applies. Confirmed **still-open** prerequisites populate the draft's `## Dependencies` section as `Blocked by #N — <reason>` lines (see `references/issue-template.md`); already-closed ones are recorded as Technical Context provenance instead. (This is the deliberate authoring producer for the `/devflow:implement` Phase 1 Pass 4 sequencing gate.)
- [ ] **Visual specification (user-visible UI changes only)** — for an issue you infer involves user-visible UI, either a screenshot/mockup is recorded or the visual details are verbally verified with the user. **Skipped entirely for non-UI issues**, which gain no new questions. See the *Visual-specification guidance* below for the dimensions to verify and the screenshot-preferred-not-mandatory rule.

**Independent-derivation pass (mandatory, before any clarification round — runs on every run regardless of how complete the story looks).**

A story that *looks* fully baked — structured sections, its own acceptance-criteria list — is the trap this pass defuses: when criteria arrive pre-written, the temptation is to treat them as answers "already known from the user story" and skip derivation, so the supplied list silently anchors the whole issue. A terse one-line story produces a *better* issue precisely because there is nothing to anchor on. This pass removes the anchor by **ordering**, not by isolation:

1. **Derive first, read the supplied answers second — and write the derivation to an observable artifact.** Before reading the user's acceptance criteria (and any other pre-supplied decisions) *as answers*, independently re-derive the full Definition of Ready above — problem/beneficiary framing, every behavioral fork, edge/error cases, and the acceptance criteria — from the stated problem and the Step 1 codebase findings. **Write that derivation to `.devflow/tmp/issue-derivation-<slug>.md` before any clarification round begins** — run `mkdir -p .devflow/tmp` first, then **pick the kebab-case `<slug>` once (from the user story's topic) and reuse that exact path for the rest of this run**, so the gate and per-round re-checks below look for the same file and a later re-derivation can't make them false-fire. **As the very first action of this pass — before you derive anything — delete any pre-existing `issue-derivation-<slug>.md`, then write it fresh** (a prior run on the same topic yields the same slug, so a leftover would otherwise sit at this path; deleting it up front means the only way the file exists at gate time is that *this* run's pass produced it — a skipped pass leaves no file for the gate to pass on). This reuses Step 4's gitignored-`.devflow/tmp/`, pick-the-slug-once convenience-copy convention (Step 4 writes `issue-draft-<slug>.md` under the same rule); reuse the same `<slug>` for both so one issue keeps a single stem. **Note the deliberate root divergence:** this derivation artifact stays **cwd/worktree-anchored** (it is an internal gate re-checked across turns within one session, where cwd is stable), whereas Step 4's user-facing *draft* is written under the **main** repo root (`resolve-main-root.sh`) so its displayed path opens correctly from a worktree — same `.devflow/tmp/` subpath and slug, intentionally different root; do not unify them. If the write genuinely fails (e.g. a read-only sandbox) — the delete above fails the same way, which is itself the signal you are in the read-only case — say so in chat and record the derivation inline in your message **as a visible block — the actual derived list, not a bare claim that you derived it** — so it is still observable; never silently skip writing it down. **When the filesystem is read-only, do not trust any on-disk `issue-derivation-<slug>.md` (the failed delete may have left a stale leftover from a prior run); rely solely on the visible inline block as the gate's stand-in.** The artifact is the observable proof of the ordering: you commit your own list to a file before the user's list can frame your thinking. This is a forced self-check **with an observable artifact**, not an isolation guarantee — you still see the supplied criteria; you simply derive, and record the derivation, *before* weighing them. (A blind-subagent variant that never sees the supplied list was rejected here, to keep this **derivation pass ordering-based rather than isolation-based** and avoid the added latency; the derivation artifact adds no new script, dependency, or subagent — only the `mkdir -p` + file write Step 4 already performs. The pipeline stays inline overall; its sole subagent is the **Step 3.6 fresh-context auditor**, whose entire value *is* context isolation — a distinct mechanism applied at a distinct seam, after the draft is assembled, not during derivation.)
2. **Treat supplied acceptance criteria as suspect on two axes.** *Completeness* — which behavioral forks, edge cases, and factors does your independent list have that theirs omits? *Correctness* — is each supplied criterion atomic, testable, and a genuinely resolved decision, or a buried unresolved fork wearing the costume of a decided one? A polished, comprehensive-looking list earns the **same** scrutiny a terse story gets, since it has more to challenge.
3. **Drive the clarification rounds from the delta.** Diff your independently-derived list against the story and clarify from the **delta plus any conflicts** — the forks, edge cases, and factors you derived that the story left unresolved; the supplied criteria that fail the correctness test; and anywhere the two lists disagree. The supplied criteria are one of the two inputs to that diff, never a shortcut past it.

This pass **feeds** the clarification rounds; the no-options gate (Step 3) still governs the final body. Its output is a working derivation to clarify against — never a place to park options in the issue itself.

**Visual-specification guidance (user-visible UI changes only — skipped entirely otherwise).**

Some issues change what the user sees, and for those a text-only description leaves the visual
intent ambiguous. Handle the visual specification as part of this Step 2 clarification — as
**prose guidance you follow, not a new hard gate** (it adds no enforcement mechanism; an
unresolved item lands in the existing Blocked section like any other):

1. **Infer whether the issue involves user-visible UI changes** as part of your normal scope assessment above — an inference, not a dedicated "is this UI?" question. When the issue is obviously non-UI (a script, a config key, an internal doc, a CLI-only change), **the whole path below is skipped and adds no new questions**; a false positive here is the failure to avoid, so do not ask visual questions on a non-UI issue.
2. **On a UI change, check the user-provided resources/context** — pasted images, attached files, URLs, and design-tool links such as Figma — for an existing screenshot or mockup before asking for anything.
3. **If a screenshot/mockup is present, record it** in the issue's Visual Specification section (see `references/issue-template.md`): embed it when a hosted URL is available, otherwise reference it with a one-line note on how the implementer can obtain it. Do not then ask the user for one they already supplied.
4. **If none is present, ask the user to provide a screenshot or mockup** via the runner's user-question tool (the same tool Step 2 uses elsewhere).
5. **If the user has none, verify the visual details with the user before finalizing the draft** — pinning down, as applicable to the specific task: placement & layout, visual states (hover/focus/error/empty/loading/disabled), responsive behavior across breakpoints, and design-system/style match, plus any further visual dimension the task makes relevant. This checklist is a **non-exhaustive prompt**, not a fixed form: add task-specific dimensions and skip inapplicable ones.
6. **Write the screenshot reference and/or the verified placement description into the Visual Specification section** of the drafted issue.

A screenshot/mockup is **preferred, not mandatory** — verbal verification is an accepted substitute, so a UI issue is never blocked solely for lacking an image. If a UI-placement detail is still unresolved when the user disengages, it **flows to the existing `## 🚫 Blocked` section like any other unresolved decision** (the existing mechanism from the disengagement rule below — not a new gate).

**How to ask:**

- **Gate — confirm *this run's* derivation artifact exists before the first clarification round.** Before the first clarification question — **or, if the story is complete enough that you will ask none, before drafting in Step 3** (the gate must fire on every run, not only ones that clarify; see the Step 3 drafting precondition, which is the unconditional backstop for the zero-question path) — verify `.devflow/tmp/issue-derivation-<slug>.md` is present **and holds the derivation you produced for the current story** — presence alone is not enough, because the slug is deterministic from the topic, so a prior run on the same topic can leave a stale file at this exact path. (The pass above deletes any leftover before writing, so on a correct run the file is unambiguously this run's; the freshness check is the backstop for the case the pass was skipped, where deletion never happened either.) If the file is missing, or you cannot confirm its content is this run's derivation, the independent-derivation pass was skipped — **stop, run the pass and write the artifact now, and only then ask anything.** Do not begin clarification without it; surface its absence rather than silently continuing. (If the write itself failed in a read-only sandbox, a **visible block you posted in chat this run containing the full derived Definition of Ready** — the actual list, not a bare claim of having derived it nor a pointer to earlier prose — stands in for the file. "Present" means it is in *this run's* transcript; because later checks run in subsequent turns, on a read-only filesystem re-post that full block in the current turn whenever you reach a check that fires there. A derivation in neither this run's file nor such a visible block means the pass did not run.)
- **Ask via the runner's user-question tool** — `AskUserQuestion` (Claude Code, the canonical example), or the equivalent your runner exposes (e.g. `ask_user` on GitHub Copilot CLI). Source the questions from the independent-derivation **delta plus conflicts** (the pass above). Batching is **conditional on the tool**: where it supports multi-question batches (as `AskUserQuestion` does), batch 2–4 related questions per call rather than one long interrogation; where it is one-question-per-call, ask them sequentially (as on `ask_user`) — the total budget below is the same either way.
- For each question, offer concrete multiple-choice options. When the codebase or findings make one choice clearly best, list it **first** and mark it `(Recommended)` with a one-line why. **Where the tool cannot present structured choices** (a free-text-only question tool), enumerate the concrete options in the question text itself — best first, that one marked `(Recommended)` — so the multiple-choice discipline holds regardless of the tool's affordances.
- After each round of answers, **re-check the Definition of Ready** — including that the derivation artifact still exists and still holds this run's derivation (or, in a read-only sandbox where the write failed, re-post the visible inline-in-chat stand-in in this turn). If the file has vanished — or, on a read-only filesystem, you have not re-posted the inline derivation this turn — treat it exactly as the gate above: stop and re-write the file (or re-post the inline derivation) before asking the next batch. If gaps remain, ask another batch. Keep going until the list is fully satisfied — do not draft with items still open.
- Cap clarification at a **runner-neutral total-clarifying-question budget** of ~18 questions total (equivalent to the ~6 batches of up to 3 this previously allowed) — a *count of **answered** questions*, not a count of rounds or of questions merely asked, so a one-question-per-call runner gets the same budget as a batching one instead of being starved to ~6 single questions. A silent non-response is **not** an answer: it never counts toward the budget and never trips "treat the remainder as disengagement" — re-ask it instead (see the silent-non-response rule below, which defines what counts as silence). If the budget is exhausted by **explicit** answers and facts are still missing, treat the remainder as disengagement (below).

**Push back once before accepting total disengagement.** If the user disengages while the issue is still *unbuildable* — no decided scope, or the single core behavior fork is still open (the ticket would be almost entirely Blocked) — do not silently produce a hollow issue. Say so plainly, once: e.g. *"As-is this issue won't be buildable — every decision is still open. Can you answer just two things: (1) is this one feature or several, and (2) <the single most defining behavior fork>? Otherwise I'll file it with everything flagged as blocked."* Ask those via the runner's user-question tool (one batch where it batches, sequentially where it is one-question-per-call). If the user answers, continue clarifying; if they give an explicit disengagement signal again (silence is not one — see the silent-non-response rule below), proceed below. This push-back happens **at most once** — do not nag. It does not apply when only peripheral forks remain open (e.g. link expiry) — Block those without comment.

**A silent non-response is not disengagement — never proceed on silence.** A question-tool timeout, a `No response after 60s` return, or the user simply stepping away is **not** a disengagement signal — the user's answer often arrives seconds later. When a question has been asked and no answer has come back, **pause and re-ask that question in your final chat message** (where the user can answer at their own pace); do **not** proceed, draft the issue, guess a default, or route the item to the Blocked section on that basis. Only an **explicit** reply from the user — "just create it", "you decide", or "I don't know" — or an exhausted clarification budget of **explicit** answers (the budget rule above), engages the disengagement / draft-from-decided / Blocked path below; a silent non-response never does. This mirrors the Step 4 confirmation gate: when the user has not responded, the pipeline stays paused rather than moving on without them.

**When the user disengages** — gives one of the explicit signals "just create it", "I don't know", or "you decide" to a Definition-of-Ready question (these three explicit replies are the *only* user-reply disengagement triggers — the sole non-reply entry is an exhausted budget of explicit answers, above; silence is never a trigger — see the rule directly above):

- Stop asking. Draft the issue from what *is* decided.
- Every still-unresolved Definition-of-Ready item goes into the issue's **`## 🚫 Blocked — resolve before implementation`** section, phrased as a direct question with one line on why it blocks work (see template).
- Do **not** invent a default and bury it in the body. Do **not** rephrase the open decision as an "option" or "recommended approach" elsewhere. The Blocked section is the *only* place an unresolved decision may appear.
- "You decide" is not permission to guess silently — it is an unresolved item that belongs in Blocked, unless the choice is genuinely inconsequential to scope (e.g. a variable name).

### Step 3: Draft the issue and pass the no-options gate

**Precondition — the Step 2 derivation-artifact gate applies here too, unconditionally.** Before drafting, confirm `.devflow/tmp/issue-derivation-<slug>.md` exists and holds *this run's* derivation — **or, in a read-only sandbox, rely solely on the visible inline-in-chat stand-in re-posted in this turn and do not trust any on-disk file (it can only be a stale leftover; the read-only distrust rule from the Step 2 gate applies here too)**. A fully specified story can reach this point having asked **zero** clarifying questions — the very "looks fully baked" case the derivation pass exists to defuse — so the Step 2 gate's first-clarification-question trigger may never have fired. This drafting precondition is the unconditional backstop: drafting happens on every run, clarification does not. If the artifact is missing or you cannot confirm it is this run's, the independent-derivation pass was skipped — **stop and run it now (Step 2) before drafting.**

Draft the issue **from the context you already hold** — the documentation findings from Step 1 (relevant files, current behavior, any drift) and the decisions from Step 2 — doing only targeted verification reads where a specific claim needs confirming. Do not re-explore the whole codebase; the findings are your map.

Follow `references/issue-template.md` for the required section structure, the **no-options rule**, the quality checklist, and autolink hygiene. Key rules:

- **No-options gate (run before showing the draft):** re-read the rendered body. Outside the `## 🚫 Blocked` section it must contain **no** unresolved-decision language — no "or", "either", "alternatively", "could", "we might", "TBD", "option", "approach A vs B", "(optional)"-for-undecided, "e.g. X or Y" where X and Y are competing choices. Each acceptance criterion is one concrete unconditional assertion. If you find any such language, you skipped a decision: either ask the user now, or move it to the Blocked section. Do not proceed to Step 4 until the body is clean.

Drafting produces a candidate issue **in your message only** — nothing is posted to GitHub in this step. Posting happens in Step 4, and only after the user confirms — but first the draft must survive Step 3.5.

### Step 3.5: Steelman the draft against the code (mandatory, before the user sees it)

*"Steelman this. Did we miss anything?"* — run that challenge against your own draft **every run**, immediately after the no-options gate passes and before Step 4 presents anything. The moment a draft passes the gate is when it looks most finished and is examined least; this step exists because a plausible-reading draft routinely carries claims the code contradicts, and the user should never be the one to catch them.

This is a **code-grounded verification loop, not a re-read**. Ambient context — project memory, the Step 1 findings, your recollection of the codebase — is where the wrong claims *came from*; checking the draft against the same context it was drafted from verifies nothing. Every check below runs against the **actual repository content, via tool calls made now** (targeted `Read`s and `Grep`s — never a full re-exploration; the same premise-verification discipline `references/issue-template.md` defines under *"Verify every load-bearing premise before drafting"*, re-applied to the assembled draft):

1. **Re-read the rendered draft end-to-end as an adversarial reviewer** whose job is to find what is wrong or missing, not to admire what is right.
2. **Verify each load-bearing claim, file reference, and acceptance criterion against the code.** Every named file/path exists as named (a `Glob`/`ls` per reference); every "the code currently does X" claim is confirmed by reading the code that does X; every AC is checked to be assertable against the code surface it names. Match the verification method to the claim class per the template's discipline — including the **relied-on-third-party-behavior** class: every behavior of an external platform, API, or service the draft **relies on** — load-bearing for the Desired Behavior, an acceptance criterion, **or** the Implementation Notes Approach, not only an AC's mechanism (webhook / event delivery, trigger syntax, token scopes, endpoint behavior, response shapes, rate limits) — is verified here too against the vendor's **official documentation via `WebFetch`**; when official docs are unreachable, via **`WebSearch`**; when search is unavailable or fails, by **asking the user to provide the documentation** (not memory), and the verified fact + source URL is recorded in the draft's `Technical Context`. An empty or inconclusive result is **unverified, never confirmed** — rewrite such a claim as an explicitly flagged assumption (the template's `— assumption, confirm before implementing` form). Terminal arm, two ways (mirroring the template): a relied-on behavior with **no** documentation and **no** in-repo example proving it becomes a `## 🚫 Blocked` vendor-behavior question; one a working in-repo example proves is written as a flagged `— assumption, confirm before implementing` line citing that example. An **incidental** third-party mention (not load-bearing) stays light and triggers no verification. **A mechanical claim is verified-or-obligation, never a bare prediction** (mirroring the template's Testing Strategy Move 3 rule): a Testing Strategy assertion of the form "running X reports Y" / "this must fail RED reporting Y" is written **either** as verified output you actually ran while drafting and are citing, **or** as an obligation on the implementer ("the pin must cover X") — never a prediction of the specific result you did not execute, which reads as a decided requirement and sends the implementer to encode a falsehood. `Relevant Classes/Files` references cite the symbol or section, not a `file:line` anchor, which rots. **Value-comparison ACs are checked for observed-output grounding (mirroring the template's Acceptance Criteria rule):** an AC (or Testing-Strategy assertion) comparing a produced value against a literal must state the comparison in the producing surface's observed-output terms, grounded by a probe that **exercises the distinguishing boundary fixture** (for a type-sensitive comparison, the type-boundary fixture — a JSON string `"true"` vs. a boolean `true`) or by a named implementer obligation stating the decided semantics or exact fixture-and-observed-output command. **Adjective-only ("explicit/exactly X") or probe-silent-on-the-axis comparison language is flagged non-conforming** and rewritten to observed-output terms — or, when the axis is a specification *choice* rather than a probeable fact, routed to Step 2's user-question machinery, not an obligation. **Universal-quantifier sweep (mandatory, same carve-out as the template's Acceptance Criteria universal-claim rule):** scan every universal quantifier the assembled draft asserts about the system under change — "never", "always", "each", "every", "all", "cannot" — outside `## 🚫 Blocked`, and confirm each is grounded one of the three decided ways (pinned per-arm/per-element, with an accepted-loss/suppression claim pinned by a fixture in which the suppressed input is present; scoped to the mechanism's supported form; or removed); the only exemptions are mandated-verbatim template boilerplate and rule text the change ships as artifact content, and an AC or Desired-Behavior sentence is never exempt however imperative its phrasing. A **detector/guard coverage claim** additionally requires a planted-defect positive-control obligation on the implementer. Rewrite or ground any ungrounded quantifier now, before the user sees the draft. **Occurrence counts and coupled-site lists are checked the same way (mirroring the template's occurrence-count premise class):** an in-repo occurrence count or mirror-site list is grounded by an executed whitespace-normalized search cited as a "Verified:" bullet with its command and hit list where the tier grants one, else by the specific evidence records consulted record-by-record — never a recall count. **A "code does X" premise is verified with its enclosing gates on the path to X (mirroring the template's conditional-path clause):** read the gates, conditionals, and defaults reaching X, and a claim that holds only under a non-default configuration states that precondition inside the claim.
3. **Hunt for what the draft omits**, specifically: **missed acceptance criteria** (behavior the Desired Behavior implies but no AC pins), **missed edge cases** (error paths, empty/absent inputs, concurrent or repeated events), **wrong assumptions** (claims step 2 could not confirm), **unstated scope** (surfaces the change touches that the draft never mentions — grep the identifiers the draft names for consumers it missed), **a missing input-is-data guard** (a new LLM/semantic-judgment surface over third-party text without the guard-AC-plus-hostile-case pair), **an undeclared enumerated AC list** (an enumerated test/case/example list inside an AC that declares neither a floor marker nor a closed-set exhaustiveness statement), and **an unreconciled multi-state contract** (when the draft states a multi-state contract — a set of outcome tokens, error/exit codes, a status enum, or a state machine, where each state has one or more trigger causes — in **more than one form**, e.g. a summary or table form *plus* the detailed per-state acceptance criteria, verify the forms agree: **no summary or table form lists fewer causes for a state than the detailed per-state ACs** collectively specify. This is a **within-text** comparison of two forms the draft already contains, so it is scoped only to drafts that actually state such a contract — a draft that states none draws no new hunt, question, or revision here, mirroring the "skipped entirely for non-UI issues" discipline. It makes **no** claim to catch a state that only a not-yet-written implementation will emit — that stays a review-time catch, out of reach at drafting time. A disagreement found here is fixed by revising the draft (step 5's revise-and-re-gate loop), not surfaced to the user). Also run the **AC mutual-consistency check (mandatory):** no acceptance criterion forbids a surface — a path, a file class, a tier — that another AC's discharge must touch; a pair that conflicts is an unresolved scope fork, not two independent criteria. And the **trust-boundary closure check (mirroring the template's Acceptance Criteria rule):** when an AC establishes a trust or integrity boundary over executable artifacts (scripts, hooks, anything sourced, exec'd, or imported), confirm the protected set is defined over the transitive source/exec/import closure of its entry points — a protected entry point that sources an unprotected sibling leaves the boundary open one hop deeper — or that the residual unprotected surface is stated explicitly.
4. **Sweep the draft's own unstated mechanism dependencies (mandatory).** Beyond the *stated* premises checked in step 2, enumerate the in-repo behaviors the draft's **designed mechanism relies on but never asserts as a claim** — a helper's exit code, a resolver's output shape, a gate's semantics — the exact class the claim-driven loop above cannot see, because nothing states it (issue #446's config-unreadable arm rested on `config-get.sh`'s malformed-input non-zero exit, unasserted, so nothing verified it). Resolve each with a **cited probe** (a "Verified:" bullet citing observed output) or an **implementer-obligation AC** — the obligation subject to the execution-tier constraint (a command already granted in `devflow_implement.allowed_tools`, or a code-reading obligation citing the producer code, never an ungranted-helper run). This extends the existing claim-driven discipline to unstated in-repo reliances and adds **no duplicate premise class**; it is the in-repo sibling of the relied-on-third-party-behavior class in step 2. **Extend the sweep to stated-but-unbound inputs (mandatory):** a mechanism input the draft describes by *role* — "the value the caller already tracks", "the branch diff it already has" — with no named referent is bound to a named variable, artifact, or file with a **cited code reference proving it exists at the consuming site**; when drafting cannot establish that binding, the draft carries an **implementer-obligation AC naming the binding as work**, never an unbound role description that expands to nothing at implement time. This is the stated-but-unbound sibling of the unstated-dependency sweep — the dependency sweep hunts reliances the draft never states, this clause binds inputs the draft names only by role.
5. **Revise the draft yourself and re-run the Step 3 no-options gate on the revision.** Then run **Revision-delta verification** (the shared procedure below). Fixing a false claim or adding a missed AC is your work, done now — do not present the flaws to the user as questions attached to an unrevised draft. If — and only if — the steelman surfaces a genuinely **new unresolved decision fork** (competing behaviors only the user can choose between), route it through the existing Step 2 machinery: ask via the runner's user-question tool, and on disengagement it lands in `## 🚫 Blocked` — no new decision-handling path.
6. **Self-check the draft against the Step 3.6 audit dimensions.** Before summarizing, run the same **dimension checklist the auditor will receive** — the Step 3.6 **generic dimension checklist** plus any consumer `## Audit dimensions` section, the latter **re-loaded fresh at check time** via the same `load-prompt-extension.sh create-issue` recipe (and for the same compaction rationale Step 3.6 states: turn-one extension text can be evicted, and a heading-less extension makes the loss invisible). Judge the assembled draft against each dimension, **fix what the check finds, and re-run the Step 3 no-options gate on any revision.** Then run **Revision-delta verification**. This is a drafter *self*-catch of checklist-class defects (a silent-output leak, a fails-open selection, a coupled doc site, an environment-variance flaw) before the subagent is dispatched — it reduces dependence on the auditor rather than adding audit rounds.
7. **Log the outcome either way.** Report a one-line summary in your next message — claims/file-refs/ACs checked (count), what was revised, or "steelman: N claims verified against the code, no revisions needed". **The summary additionally reports both new sweeps — the value-comparison type-semantics check (step 2) and the unstated-mechanism-dependency hunt (step 4).** When the hunt finds **no** dependencies, the summary states the falsifiable claim itself — "the mechanism invokes no in-repo helpers, resolvers, or gates" (checkable against the draft's Implementation Notes) — never a bare zero count. The summary **also reports the universal-quantifier sweep** — on a zero-findings run stating the falsifiable claim itself, "the draft carries no ungrounded universal quantifier", mirroring the unstated-dependency zero arm rather than a bare zero count. **The summary also reports the Step 3.6 dimension self-check (step 6) — its finding and revision counts, and on a zero-findings run the falsifiable claim "no dimension-checklist finding" rather than omitting the mention.** A silent pass is indistinguishable from a skipped step; the summary line is the evidence it ran.

Only a draft that has survived this step — revised and re-gated if anything was found — proceeds to Step 3.6.

### Revision-delta verification (shared procedure — referenced by every revise-and-re-gate site)

Every revise-and-re-gate sentence in this skill (Step 3.5's items 5 and 6, Step 3.6's `VERDICT: REVISE` handling and its user-chosen rounds, and Step 4 sub-step 4's two revision sentences) references **this one procedure by name**; it is stated **here, once**, and no site paraphrases it. It runs **at every revision event** — before any re-audit dispatch at that site and before any presentation of the revised draft — so audit rounds are spent on genuinely fresh defects, and a revision that reaches filing on a declined re-audit has had its walked delta verified rather than only language-gated. (A "revision event" is one revise-and-re-gate occurrence; a multi-finding `VERDICT: REVISE` is one event spanning several verify-and-edit batches.)

1. **Bind and walk the delta per edit-batch.** An **edit-batch** is the set of draft edits you apply in one assistant turn; you authored them in that turn, so each batch's walk (point 2) runs **in that same turn** and its binding depends on no recall across compaction. A revision event that spans several turns accumulates one walk per batch, and the event's closing evidence line (point 7) sums the batch walks, reading them from their own transcript records. The walk runs at edit time, never retroactively.
2. **Each batch walk covers all six delta classes, per class**, recording what the batch introduces, changes, or **removes** in each — a removal counts, because a deleted mechanism can orphan a producer/consumer pair elsewhere in the draft: (a) mechanisms, (b) lifecycle rules (artifact and state lifecycles, ordering), (c) execution-tier assumptions, (d) dependencies (in-repo helper/resolver/gate/artifact reliances, and relied-on third-party behavior), (e) universal guarantees and quantifiers, and (f) **the residual class** — any other new load-bearing claim, file reference, value comparison, or delta content fitting none of (a)–(e). Class (f)'s residual definition is what makes the six-class partition **total**, so no delta content is outside the walk by construction. The walk record is transcript text: **one compact entry per class — the class's enumerated items, else a stated falsifiable zero** naming why the batch has none ("no new mechanism: the batch touches only the Testing Strategy's fixture names"), never a bare zero and never a single aggregate covering all six classes. Tool evidence inside a walk entry is cited as a **one-line summary, never pasted output** — the compactness rule that keeps the procedure's own context cost from becoming the compaction it defends against.
3. **An all-zeros walk ends the batch's work at the walk record.** Wording fixes, citation additions, and scope narrowing enumerate to nothing, so they impose no re-steelman — the cost of such a batch is the six one-clause zero entries plus the summary line. There is no separate classify-then-skip gate: the per-class walk itself decides, and its **stated zeros are the auditable record of that skip decision**. The enumeration remains a **drafter judgment that can under-enumerate** — a zero forced per class is auditable where a single aggregate "nothing here" is not, but this ships **no cannot-fail-open claim**.
4. **Verify each enumerated (non-empty-class) item** with the delta-scoped reuse of the two existing Step 3.5 disciplines, **referenced by discipline name, never copied as a list that would drift from its source**: the **load-bearing-claim premise verification** of Step 3.5's item 2 (with its value-comparison type-axis grounding, universal-quantifier grounding, occurrence-count grounding, conditional-path reading, and the relied-on-third-party documentation ladder when the delta introduces such a reliance) and the **unstated-mechanism-dependency sweep** of Step 3.5's item 4 (with its stated-but-unbound-input binding clause) — via targeted tool reads against the repository scoped to the enumerated items, never ambient context. The whole-draft omissions hunt (item 3) and the Step 3.6 dimension self-check (item 6) stay one-shot whole-draft passes; this procedure reuses checks, it does not re-run Step 3.5.
5. **Fix what it finds inline, re-run the Step 3 no-options gate, and walk the fix as its own edit-batch; a batch whose walk is all-zeros ends the loop.** This is the same inline-revise shape as Step 3.5's item 5, and each successive batch covers only its own edits, so the loop's work shrinks with the delta.
6. **Route a genuinely new unresolved decision fork through the existing Step 2 machinery** (the runner's user-question tool; `## 🚫 Blocked` on disengagement) — add no new decision-handling path.
7. **Close every revision event with one standalone evidence line, at the per-site anchor below**: `revision-delta check: <N> enumerated, <V> verified, <F> fixed` (summing the event's batch walks) when any class in any batch was non-empty, or `revision-delta check: no verifiable delta` when every batch walked all-zeros. The line is chat text — it does not extend the Step 4 audit summary line and writes to no artifact. The five per-site anchors cover all six wired sites because the two Step 4 sub-step 4 revision sentences share one anchor: at Step 3.5's item 5, immediately after that item's gate re-run; at Step 3.5's item 6, immediately before the initial Step 3.6 dispatch's pre-dispatch draft write; at the Step 3.6 `VERDICT: REVISE` site, immediately before the re-audit dispatch; at a user-chosen round, immediately before the T1/T2 trigger re-evaluation and its offer message; at the Step 4 sub-step 4 sites, immediately before the re-presented draft. Because the sub-step 4 ordering places the re-presentation **before** the re-audit offer, every flow that later reaches a declined offer passed one of these anchors first, so the declined-offer filing path carries the most recent evidence line adjacent to its presentation with no separate anchor.

### Step 3.6: Fresh-context audit (mandatory, before the user sees it)

Step 3.5 is strong at verifying what the draft *says* and structurally weak at seeing what the draft *misses*, because it runs **inline, in the same context that drafted the issue** — it inherits the drafter's anchoring. This step removes that anchoring the only way that works: **information removal**. Separated-context critics outperform same-session self-review, with the largest gains on critical errors, precisely because the auditor cannot see the reasoning that produced the draft. So after Step 3.5 passes and before Step 4 presents anything, dispatch **one fresh-context audit subagent** whose entire value is that it did not draft the issue.

**Obey the state owner (the contract governing this whole step).** The deterministic audit lifecycle — transition legality, round numbering, the automatic budget, retry bounds and their precedence, dispatch-arm routing, digest and sentinel generation and comparison, the T1/T2 triggers, override records, presentation eligibility, and the audit-summary field set — is owned by the bundled `issue-audit-state.py`, **not by this prose**. This step **records each lifecycle event through that tool and obeys the answer it returns.** Never re-derive a transition, a budget, a retry bound, a dispatch arm, or eligibility from this prose, from the round history you remember, or from your own reading of the run so far — the tool's answer *is* the decision, and this prose describes only which call to make and when. **A draft you are certain is clean is presented for approval only after `query-eligibility --mode approve` answers `eligible=yes`.** Your confidence that a revision addressed every finding is not an eligibility answer, and neither is a clean no-options gate.

The tool's CLI has **two classes**, and the branch you take depends on which one you called:

- **Queries** (`query-arm`, `query-next-action`, `query-triggers`, `query-eligibility`, `query-summary`, `query-nonce`) **always exit 0 once their arguments parse** (an argparse usage error — a missing required flag or an unknown one — exits 2 before the query logic runs) and print one decided single-line token — fail-closed answers included. They are strictly read-only.
- **Mutations** (`init`, `record-*`) exit **non-zero with a named stderr breadcrumb** on an illegal transition and on an unpersistable state alike.
- `emit-body` is neither: it is a **gated emitter** — exit 0 plus the audited body bytes when eligible, exit non-zero with **empty stdout** when not.

**An illegal-transition rejection is NOT an unavailability signal.** When a mutation exits non-zero and its breadcrumb names an **illegal transition** — a nonce mismatch included — call `query-next-action` and obey that answer. Never route an illegal transition to the `state-owner unavailable` fallback below, and never improvise around it.

Invoke the tool with `python3` plus the portable anchor, resolved **inline in the statement that uses it** (never captured into a variable a later statement reads), substituting the `<slug>` and the nonce you hold:

**File-arm carriage / identity check (closes the write-to-read race).** The canonical draft path is a shared deterministic main-root path, so a concurrent same-topic session could overwrite it between this run's write and the auditor's read, letting the auditor audit foreign bytes undetectably. **Why the tool never re-hashes at compare time.** It compares the auditor's quoted ID against the digest recorded **at dispatch**, never a fresh compare-time re-hash of the on-disk file — a re-hash would see the same foreign bytes the auditor did and pass a concurrent overwrite vacuously. The two state paths are named for the reason the file arm names them: either one re-anchors the auditor on this run's prior verdicts and revision records.

This arm carries its own **carriage check** an LLM can discharge exactly: **the tool generates the two sentinel tokens** and prints them from `record-dispatch` (`sentinel_open=` / `sentinel_close=`). Omit the flags when the return quoted no sentinels. > **Per-finding bar** — every finding must: quote the exact draft line it attacks; name the concrete failure *mechanism*, not a category; verify each claim against the repository and **report an unverifiable claim as unverifiable rather than asserting it**; carry a **severity graded by observable blast radius**; and give a specific recommended edit.
>
> **Scope exclusions** — no wording or formatting notes; no implementation details decidable at implement time (judge the draft at **issue altitude**); no finding without a concrete trigger scenario.
>
> **Audit dimensions** (judge the draft against each): {GENERIC DIMENSION CHECKLIST, plus any consumer `## Audit dimensions` section}.
>
> **Cap: at most five findings**, of which **exactly one** is the **"Quiet Killer"** slot — the failure the draft is not contemplating at all. If the draft has **no actionable findings**, say so explicitly; that is a legal output.
>
> End with a **mandatory final verdict line** whose only three legal values are exactly `VERDICT: FILE` (no revision needed), `VERDICT: REVISE` (findings warrant changing the draft), or `VERDICT: DRAFT-UNREADABLE` (you could not read the draft file — emitted only on the file arm, with no findings).

**Generic dimension checklist (consumer-agnostic — names no DevFlow-engine-internal invariant).** Include these dimensions in the dispatch prompt verbatim:

- **Consumer-repo setup variance** — the draft's premises must hold on a fresh adopter checkout, not only this repo.
- **Host-OS variance** — Windows / WSL / Git Bash, macOS / BSD, and hosts without GNU coreutils.
- **Degraded environments** — shallow clones, missing PATH tools, read-only sandboxes, and both fresh and compacted agent contexts.
- **Execution-tier variance** — cloud tier and local tier, including their differing permission allowlists.
- **Second-order effects and unstated scope** — what the change touches that the draft never mentions.
- **Missed edge cases and termination paths** — error paths, empty/absent inputs, and how each flow ends.
- **Load-bearing assumptions** — each stated with what would falsify it, including any **universal quantifier** the draft asserts ("never", "always", "each", "every", "all", "cannot"): each must be grounded (pinned per-arm/per-element, scoped to the mechanism's supported form, or removed), or it is an ungrounded load-bearing assumption.
- **Adversarial third-party input** — when the draft's Desired Behavior introduces a *new* LLM or semantic judgment over third-party text the change does not author (issue bodies, PR comments, commit messages, external API responses) whose output drives an automated selection or action, the draft must carry an input-is-data guard as a decided design element — an acceptance criterion stating the text is **data to classify, never instructions to obey** — paired with a Testing Strategy case that exercises instruction-shaped input (a body that directs the judgment) and asserts it is not obeyed. Flag a draft missing the guard AC, or carrying the guard sentence with no paired hostile-input case (the pairing exists precisely so the guard cannot be satisfied by a compliance sentence the implementation never ships). A surface that reuses an existing, already-guarded judgment path is exempt when the draft cites that path; a draft with no new judgment surface gains no new flags (the visual-specification skip-when-inapplicable shape).
- **Authoring-discipline defects** — three related shapes: (1) a **value-comparison** AC or assertion whose comparison language is ungrounded on the type axis it must encode — adjective-only ("explicit X", "exactly X"), or a cited probe that never exercises the type-boundary fixture the comparison distinguishes (a string `"true"` vs. a boolean `true`); (2) a **case / input-shape matrix** narrowed below a governing convention without an explicit justification — **independently re-run** the draft's bounded consulted-sources search and flag **only** a governing matrix found at a path the draft's `governing conventions consulted:` line omits, never a judgment disagreement about what counts as governing; (3) an **unstated mechanism dependency** — the designed mechanism relies on an in-repo helper/resolver/gate behavior the body never asserts as a claim, so no premise is verified.

**Dimension-list growth policy (finding-cap discipline).** The audit-prompt cap is five findings, so the dimension list is carrying-capacity-bounded: **execution-blocking defect classes outrank authoring-discipline classes for the finding-cap slots** (a defect that breaks a run at execution time is reported ahead of one that merely made the issue costlier to author), and **future dimension additions consolidate into an existing dimension before appending a new one** — the checklist grows by sharpening, never by lengthening past the cap's carrying capacity. (The adversarial-input dimension above is a sanctioned standalone addition, not a breach of consolidate-before-appending: it is a distinct **security** class — an unguarded injection-shaped design ships an exploitable defect, a higher-stakes class than the authoring-discipline dimensions the ranking above already deprioritizes for the cap slots — and it is orthogonal to every environment and discipline dimension, so it cannot be expressed by sharpening one.)

**Extension forwarding (`## Audit dimensions`).** At **dispatch time** — not from your memory of the load performed at the top of this skill — **re-run `load-prompt-extension.sh create-issue`** (via the portable anchor recipe above) and extract from that **fresh** output any section headed exactly `## Audit dimensions`; append it verbatim to the generic checklist in the subagent prompt. On long runs, context compaction and nested-skill body eviction can drop the turn-one extension text, and the designed no-op of a heading-less extension makes that loss otherwise invisible — so the re-load is mandatory. An extension without that heading changes nothing. Record in the audit summary line whether a consumer audit-dimensions section was appended.

**Write the audit report to an observable artifact.** Reuse this run's `<slug>` (the Step 2 / Step 4 stem) and write the auditor's findings and verdict to `.devflow/tmp/issue-audit-<slug>.md` — **deleting any same-slug leftover first**, exactly like the Step 2 derivation artifact, so the file at gate time can only be this run's. The state owner's record `.devflow/tmp/issue-audit-state-<slug>.json` is a *separate, sibling* file the tool owns exclusively — it is **not** part of this report artifact's delete/overwrite cycle, precisely because this report is deleted-leftover-first at each round's write and overwritten by a Step 4 sub-step 4 re-audit, so a round record kept *inside* it would not survive the round that follows it. **Never hand-write, hand-edit, or delete the state `.json`**; the tool's `init` owns its lifecycle, including the cold-start wipe of a leftover. In a **read-only sandbox** the write fails (as the delete does); fall back to a **visible inline-in-chat block** carrying the audit findings and verdict, and — per the Step 2 distrust rule — **do not trust any on-disk `issue-audit-<slug>.md`** (it can only be a stale leftover). The Step 4 presentation gate confirms this artifact (or its inline stand-in) exists before the draft is shown.

**Where the state lives (the root divergence is deliberate — do not unify it).** The tool's record `.devflow/tmp/issue-audit-state-<slug>.json` is **cwd/worktree-anchored**, while the draft file is **main-root-anchored** via `resolve-main-root.sh`. That divergence is load-bearing: main-root anchoring would share one record across concurrent worktree runs, letting a foreign cold-start wipe this run's state. The draft is main-root-anchored for the opposite reason — its displayed path must open from the user's editor. Different roots, different jobs.

**Parse the verdict, then let the tool classify it.** Extracting the `VERDICT:` token from the auditor's prose is your work — semantic extraction is LLM work. Deciding what that token *means for the run* is not. Record the return and obey the classification:

```bash
python3 "${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../scripts/issue-audit-state.py record-return "<slug>" --nonce "<nonce>" --round "<round>" --verdict FILE --findings-count 2 --carriage-object-id "<the ID the auditor quoted>"
```

**Omit `--verdict` entirely when the return carried no parseable `VERDICT:` line** — the tool classifies the absence. Never map an unparseable return onto a verdict token yourself, and never pass a token the auditor did not emit; the tool validates the token fail-closed against its closed set. Add `--consumer-dimensions-appended` when the dispatch carried a consumer `## Audit dimensions` section. Then:

```bash
python3 "${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../scripts/issue-audit-state.py query-next-action "<slug>" --nonce "<nonce>" --round "<round>"
```

**Obey the answer verbatim** — it is one of `dispatch-embed-retry`, `dispatch-retry-same-arm`, `dispatch-inline-degraded`, `proceed`, `revise-and-reaudit`, `revise-then-evaluate-offer`, `round-open-awaiting-return`, or `round-closed-no-verdict`. On `round-open-awaiting-return`, the round's dispatch is recorded but its return is not — collect and record the pending auditor return for that round (`record-return`) before anything else; never treat it as a proceed. On any `revise-*` answer, **verify each finding against the code before acting** (a finding can be wrong), revise the draft, **re-run the Step 3 no-options gate**, then run **Revision-delta verification**. On `revise-then-evaluate-offer`, the surviving findings are **quoted verbatim** in the presentation message rather than revised for — **the audit informs, it never deadlocks filing.** If a finding surfaces a **genuinely new unresolved decision fork** (competing behaviors only the user can choose between), route it through the **existing Step 2 machinery** (the runner's user-question tool; the Blocked section on disengagement) — add no new decision-handling path. `round-closed-no-verdict` is also the fail-closed answer when the state is unestablishable or the round is unknown — check the stderr breadcrumb before treating it as a genuinely closed verdict-less round.

**The offer at the Step 3.6 → Step 4 boundary.** Call `query-triggers`; it answers `t1=hold|not-hold t2=hold|not-hold reason=…`. While **either** holds, **offer one more audit round via the runner's user-question tool, naming which trigger fired** (and naming the unestablished state when `reason=state-unestablished` — unknown is not zero). The offer is **outside the Step 2 clarification budget and draws nothing from it**, and pausing for the answer is the same sanctioned waiting state as the Step 4 confirmation gate. Record the outcome with `record-offer` (`--accepted` on "yes") and obey it: the tool owns the per-run ceiling and refuses an accepted offer past it, so **never count rounds yourself**. On **"yes"**, run the full round: verify findings against the code, revise, **re-run the Step 3 no-options gate**, then run **Revision-delta verification** before re-evaluating the triggers. On **"no"** — an **explicit decline** through the question tool, or one of **Step 2's three explicit disengagement replies** — record it with `record-override --kind user-decline --surface t1t2-boundary` and proceed to Step 4 with surviving findings quoted verbatim. When the tool refuses the offer because the ceiling is reached, record `record-override --kind cap-reached` and proceed; the summary line names the ceiling. On a **file-arm epoch**, every `record-override` call additionally passes `--draft-file "<absolute issue-draft-<slug>.md path>"` so the override is **digest-bound** to the bytes it was recorded over — a digest-unbound override survives byte changes until the next revision record; on **embed/inline epochs** omit the flag (there is no trustworthy file to bind to). A **silent non-response** follows the existing Step 2 silent-non-response rule: **pause and re-ask in the final chat message; never dispatch and never proceed on silence.** When neither trigger holds, proceed to Step 4 with no offer. This boundary offer fires only *before* presentation; once inside Step 4's iterate-on-feedback loop, sub-step 4's re-audit offer is the only offer surface — one offer per pause.

**The call sequence, in order.** The normal clean run:

`init` → write the draft → `query-arm` → `record-dispatch` → dispatch the auditor → `record-return` → `query-next-action` → `query-triggers` → `query-eligibility --mode approve` → present → `record-creation-epoch` → `emit-body` (file arm) → `gh issue create` → `record-creation-attestation` → `query-summary` → render the summary line.

The variants, each obeying the same contract:

- **Revise-and-recover:** … `record-return` (REVISE) → `query-next-action` (`revise-and-reaudit`) → revise → `record-revision` → `query-eligibility --mode iterate` for the in-loop re-show → resolve the re-audit offer → re-dispatch → … → `approve`.
- **DRAFT-UNREADABLE retry:** `record-return --verdict DRAFT-UNREADABLE` → `query-next-action` → `dispatch-embed-retry` → `record-dispatch --arm embed --marker file-unreadable` (bytes on stdin, **same round** — no second round record); the embed arm hard-requires `--marker`, and `file-unreadable` is the marker `query-arm` answers for a prior `DRAFT-UNREADABLE`.
- **File-arm same-arm retry escalating to embed:** on a `dispatch-retry-same-arm` answer for a **file-arm** round, the canonical file can become unhashable between the return and the retry (a concurrent overwrite/delete). The tool permits escalating that retry to the **embed** arm rather than deadlocking — and the embed arm hard-requires `--marker`, so re-run `query-arm` and pass **the marker it answers**: `digest-unrecorded` when the file is now unhashable, or `file-unreadable` when a prior `DRAFT-UNREADABLE` is recorded (bytes on stdin, **same round**). The escalation is never silent — the marker is recorded in `embed_markers` and rendered in the summary, exactly like the `dispatch-embed-retry` escalation above.
- **Degraded/inline:** `record-dispatch --arm inline --round <round>` (bytes on stdin) + `record-degraded --round <round> --reason <no-subagent-tool|dispatch-error|no-parseable-verdict-exhausted>` (both, like every mutation, also take the `<slug>` positional and `--nonce <nonce>`; `--round` is **required** on both — omitting it is an argparse usage error, not an illegal transition).

**Fallback — `state-owner unavailable`.** Exactly **2 classes** route here, and nothing else:

1. The tool invocation produces **no contract output** at all — the interpreter is absent, the invocation is denied, or it crashes.
2. A **mutation exits non-zero because the tool cannot establish or persist this run's state.** This is the environmental class, defined by a **principle, not an enumerated list** (so it cannot be under-specified): it is **any breadcrumb from persisting or loading the state that is not a nonce mismatch** — a persistence failure (it could not write its state); a draft-file read failure, a git-execution failure, or a leftover-delete failure; **and every load-time state error the tool raises when reading the existing state file**, whether physical (the file is missing — `no state file … run init first` — empty, unparseable JSON, or otherwise unreadable) **or semantic** (the file reads and parses but fails the tool's own schema validation — a breadcrumb naming a *recorded* field that is corrupt or outside its canonical set: e.g. `the creation record names an attestation status outside … 'maybe'`, `round N findings_count -1 is not a non-negative integer`, an out-of-set arm/marker, a broken ordinal chain, an empty digest). The tool collapses any such state to *unestablished*, so the run's lifecycle state cannot be trusted → this fallback. **Discriminator against Route B below:** a `_validate` breadcrumb is about the *on-disk state* the tool loaded (Route C); an argument-validation breadcrumb is about a value *this skill just supplied on the command line* (Route B) — e.g. a state file whose recorded `findings_count` is `-1` is corrupt state (Route C), whereas `record-return --findings-count -1` is a bad argument you passed (Route B). Route by whose input is bad — the loaded state, or the call you just made.

Two other non-zero mutation exits route **elsewhere, never here** — so every breadcrumb has exactly one route (complete by construction across all three destinations):
- An **illegal-transition or transition-legality rejection** (a breadcrumb the tool raises from its OWN transition logic — not from loading the state — refusing an illegal state move: an illegal transition, a nonce mismatch included, or another legality refusal such as a re-init over recorded rounds or already-frozen attestation, or a record against a missing/closed round or a missing creation epoch) → obey `query-next-action` instead, per the contract above (never re-issue the illegal move, and never route it to the fallback — that would turn the tool's fail-closed refusal into a licence to improvise around it).
- A **caller-contract / input-validation rejection** — a breadcrumb naming a bad *argument this skill supplied* (a negative `--findings-count`, an unknown `--marker`, an out-of-order or unfunded `--round`, a `user-decline` override missing its `--surface`, a `cap-reached` recorded before the ceiling) → the state owner is fully **available** and correctly rejecting a malformed call, so **correct the offending argument and re-issue the identical call**. Never route it here (that would degrade the lifecycle for a fixable argument and misattribute a skill bug to host unavailability) and never treat it as an illegal transition.

On the `state-owner unavailable` fallback (the two classes above), run **exactly one** audit round (a fresh-context dispatch where a subagent tool exists; the inline template where none does), keep the findings and verdict **in-chat**, offer **exactly one** continue/decline choice, and proceed to presentation **only on the user's explicit election** — which on this path is inherently the recorded-in-chat override. The audit summary line carries the distinct marker **`state-owner unavailable`**. This is deliberately **not** a retained copy of the lifecycle the tool owns: it is an acknowledged behavior change on such hosts, bounded to one round and one question.

Entered **after ≥1 completed round**, the fallback sources that round's findings and verdict **from the query surface whenever any query still answers** — queries are read-only, so a mutation-persistence failure leaves them answering — and **never from memory**. When no query answers and the findings are no longer in context (compaction), **dispatch the single fresh round instead**; when that dispatch is also impossible, the single continue/decline offer **names the unrecoverable state explicitly**. **A findings summary reconstructed from memory and presented as a round's real findings is never a legal discharge.** The `state-owner unavailable` marker is **distinct from `degraded`**, which keeps its existing meaning — the inline audit arm — and the two never substitute for one another. A fallback lifecycle is **never silent**: the mandatory-summary-line contract holds here exactly as everywhere else.

**Degraded arm — attempt-first, never pre-detected.** Dispatch **whenever a subagent tool is exposed** (nested contexts included; nested dispatch was live-probe-verified working at drafting time — never pre-detect a nested context and skip). You reach this arm when the runner exposes **no** subagent-dispatch tool, when the dispatch **call itself errors**, or when `query-next-action` answers `dispatch-inline-degraded` — a *still-running* dispatch is awaited, never retried as a bad return. On the degraded arm, run the **same audit prompt inline** against the rendered draft, then record it (`record-dispatch --arm inline --round <round>` with the bytes on stdin, plus `record-degraded --round <round> --reason …` — `--round` is required on both) so the summary line is marked degraded — **a skipped audit is never silent.** The degraded inline arm audits the **rendered draft already in context** and is **exempt from the read-the-file instruction** (there is no subagent to read it) and emits **no `VERDICT: DRAFT-UNREADABLE`** — that third value belongs to the file arm only, and the "degraded" summary marker already signals this arm's weaker properties.

Only a draft that has passed this step — audited, and revised-and-re-gated if the verdict required it — proceeds to Step 4.

### Step 4: Review with the user, then create

**The issue is never created until the user has seen the full rendered draft and explicitly approved it.** The user story they gave you was a rough input; this is their one chance to read the assembled ticket as a whole and correct it before it becomes a real, notification-sending GitHub issue. This gate is **unconditional** — it applies no matter how thoroughly Step 2 resolved every decision, and no matter what the user said earlier.

**Presentation gate — confirm *this run's* audit artifact exists before the draft is shown.** Mirroring the Step 3 derivation-artifact gate: before showing anything, confirm `.devflow/tmp/issue-audit-<slug>.md` is present **and holds this run's audit** (a Step 3.6 report, not a same-slug leftover from a prior run — the Step 3.6 write deletes any leftover first, so on a correct run the file is unambiguously this run's). **In a read-only sandbox, rely solely on the visible inline-in-chat audit block re-posted this turn and do not trust any on-disk `issue-audit-<slug>.md`** (it can only be a stale leftover — the same read-only distrust rule the Step 2 gate applies). If neither this run's artifact nor its inline stand-in is present, Step 3.6 was skipped — **stop and run it now before presenting the draft.**

1. **Show the complete rendered issue in chat.** Post the exact title and the full body — every section, verbatim, as Markdown — directly in your message. Do not summarize it, abridge it, or describe it; the user reviews the literal text that would be filed. Do not use the runner's user-question tool (`AskUserQuestion` or its runner equivalent) to stand in for showing the body (such tools truncate); render the body first, then you may use it for the confirm/edit prompt if you wish.
2. **Also write the draft to a file for easy review — at the MAIN repo root, so its displayed path is correct inside a worktree.** Reuse the kebab-case `<slug>` you picked for the Step 2 derivation artifact (so one issue keeps a single stem; Step 2's pass runs on every run, so there normally is one). Only if Step 2 never bound a slug should you derive a short kebab-case slug from the issue title here (e.g. "Add CSV export for survey results" → `add-csv-export-for-survey-results`) — its different source (title vs. the topic Step 2 uses) doesn't matter, since the reuse path is the norm. **Resolve the main working-tree root** with the bundled `resolve-main-root.sh` (via the same portable anchor recipe used above), then create its `.devflow/tmp` and write there:
   ```bash
   MAIN_ROOT="$("${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../scripts/resolve-main-root.sh)"
   mkdir -p "$MAIN_ROOT/.devflow/tmp"
   ```
   Capturing the resolver's **output** (the root path) into `MAIN_ROOT` is fine — the *anchor* itself is still expanded inline in that single statement, per the never-capture-the-anchor rule above; only the resolved root is reused. Write the rendered title + body to the **absolute path** `<main-root>/.devflow/tmp/issue-draft-<slug>.md` (i.e. `$MAIN_ROOT/.devflow/tmp/issue-draft-<slug>.md`), with the title as a top `# ` heading above the body — that path is gitignored, so it never lands in a commit. Using the **main** root (not the cwd) is deliberate: inside a linked Claude worktree (`.claude/worktrees/<name>/`) the cwd-relative path would resolve to a file the user — whose editor is rooted at the main repo — cannot open; the resolver returns the main repo root in a worktree, the repo root in a normal checkout, and falls back to cwd (with a stderr breadcrumb) when git can't answer. This is a distinct resolution from the `.devflow/config.json` read, which stays cwd/worktree-anchored (per #295) — do **not** unify them. **Keep this one `<slug>` for the issue throughout, reusing that exact absolute path from then on** — so revisions overwrite the one file (next sub-step) and a second or third issue you draft later gets its *own* distinct file instead of clobbering this one. This is a **convenience copy** so the user can open the draft in their editor; it never replaces showing the body in chat (todo above). On an **embed/inline-arm epoch** it is **not** the source the issue is created from; on a **file-arm epoch** this canonical file *is* the posting source — but only **through the gated `emit-body` emitter** (sub-step 5), never by passing the path to `--body-file <path>` directly. If the write fails (e.g. read-only main root / sandbox), note it briefly and continue **without showing a path** — the chat render is what matters, and a path to a file that was not written would only mislead.
3. **Ask for explicit confirmation or feedback.** Plainly invite the user to either approve creation or request changes. **The draft-save-location note (the absolute `<main-root>/.devflow/tmp/issue-draft-<slug>.md` path from sub-step 2) always renders *below* the rendered issue preview — never above it**, so every confirmation message has the same layout: the full rendered body (sub-step 1) first, then the **one-line Step 3.6 audit summary**, then the draft-path note, then the confirm/edit question. **The audit summary line is mandatory and always renders** — even on a clean `VERDICT: FILE` with zero findings. A skipped or degraded audit is **never silent**; the summary line is the evidence the audit ran and which arm it took. **Do not assemble its fields from your own recollection of the run — read them from `query-summary`** and render what it reports:

```bash
python3 "${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../scripts/issue-audit-state.py query-summary "<slug>" --nonce "<nonce>" --draft-file "<absolute issue-draft-<slug>.md path>"
```

It reports the findings count, revisions applied, verdict, rounds run, whether a consumer `## Audit dimensions` section was appended, the degraded flag, the user-declined and cap-reached flags, any embed-arm markers, the eligibility token, and — as its trailing final field — `attestation=<match|mismatch|attestation-unavailable|none>` (what the post-creation attestation established; `none` before any creation attempt) — render them as one line (e.g. *"Fresh-context audit: 2 findings, 1 revision applied, VERDICT: FILE, 2 rounds (no consumer audit dimensions)."*), carrying each flag the query reports: the word "degraded" when it reports degraded, the phrase `user declined further audit` when it reports a decline, the ceiling when it reports the cap, and the embed marker when it names one. On the `state-owner unavailable` fallback — where the query itself is what failed — the line carries that marker instead and is composed in-chat; it still renders. Display the **same absolute path you actually wrote** (`$MAIN_ROOT/.devflow/tmp/issue-draft-<slug>.md`), never a bare relative `.devflow/tmp/…` string — since the write and the display use the identical resolved root, the shown path always resolves to the exact file written. Separate the note from the preview rather than embedding it inline ahead of the body — e.g. after the rendered body, add a line such as *"Draft also saved to `<main-root>/.devflow/tmp/issue-draft-<slug>.md` for review."* (with `<main-root>` replaced by the resolved absolute path) and then ask *"Want me to create it as-is, or change anything first?"* If the sub-step 2 write failed, omit the draft-path note entirely rather than showing a path to a nonexistent file.
4. **Iterate on feedback.** If the user requests any change, revise the draft, re-run the no-options gate (Step 3), then run **Revision-delta verification**, and record the revision with `record-revision --after-round <round>`, and **overwrite the same `<main-root>/.devflow/tmp/issue-draft-<slug>.md` file (the identical absolute path) you chose in sub-step 2** (keep the original filename even if the title changes, so you don't leave orphaned drafts behind). Each revision is re-presented in full — never apply edits and create in the same turn without showing the updated draft.

   **Order matters here — resolve the re-audit offer BEFORE asking for approval.** A just-revised draft is shown in-loop under `query-eligibility --mode iterate` (`iterate=ok ordinal=N`), which authorizes the **re-show only** — it is never a ground for acting on approval and never a ground for creation. So: show the revised draft, then **offer a fresh re-audit** via the runner's user-question tool. On **"yes"**, dispatch one. On **"no"** — the user declining further auditing on these just-revised bytes — record that decline with `record-override --kind user-decline --surface step4-offer` (on a **file-arm epoch** additionally pass `--draft-file "<absolute issue-draft-<slug>.md path>"` so the override is digest-bound to the bytes; omit it on embed/inline epochs). Recording this override is what lets `query-eligibility --mode approve` answer `eligible=yes` for the still-unaudited bytes — **without it, a draft the auditor keeps returning `VERDICT: REVISE` on can never clear the sub-step 5 eligibility gate and filing deadlocks, which the audit must never do (it informs, it never blocks filing).** Only once that offer is resolved do you ask the confirm/edit approval question, on bytes that `query-eligibility --mode approve` answers `eligible=yes` for. **Approval is solicited only on bytes eligible in `approve` mode**; asking the user to approve a draft whose eligibility has not been established invites a yes you cannot act on. Repeat until the user approves.

   This sub-step-4 offer is the **only offer surface inside Step 4's iterate-on-feedback loop** — the Step 3.6 → Step 4 boundary's T1/T2 offer fires only *before* presentation, so it never double-asks about a revision this loop handles (one offer per pause). A dispatched re-audit runs the pre-dispatch canonical-draft write first, then `query-arm` → `record-dispatch` → dispatch, exactly as Step 3.6 specifies. Its return is handled by the **same loop as Step 3.6** — `record-return`, obey `query-next-action`, verify findings against the code, revise, re-run the no-options gate, then run **Revision-delta verification**, and re-present the full draft — and its report **overwrites the same `.devflow/tmp/issue-audit-<slug>.md` artifact** (updating the audit summary line accordingly).
5. **Create only on explicit approval.** Once the user clearly says to create it (e.g. "yes", "create it", "looks good, file it"), **first confirm `query-eligibility --mode approve` answers `eligible=yes`** — the user's approval and the tool's eligibility are two different gates, and creation needs both. **If it instead answers `not-eligible` because the bytes still carry an unresolved `VERDICT: REVISE`/`no-verdict` and the user — after the re-audit offer surface has been exhausted (an accepted re-audit returned `VERDICT: REVISE` and they have chosen to file the revised bytes without another round) — is explicitly electing to file anyway, record that election with `record-override --kind user-decline --surface step4-approval-after-exhausted-offer` (`--draft-file`-bound on a file-arm epoch) and re-query.
