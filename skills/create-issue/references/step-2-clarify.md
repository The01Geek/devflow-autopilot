### Step 2: Clarify until the Definition of Ready is met

**Clarification is the default, not the exception.** Do not assess "is this clear enough to skip" — assess "which Definition-of-Ready facts are still unknown" and ask about those. The only way to skip a question is to already know its answer from the user story or the Step 1 findings.

**Definition of Ready — every item below must have a single, decided answer before you draft:**

- [ ] **Independent derivation before anchoring (run the pass below FIRST, on every run).** You have re-derived the full Definition of Ready from the problem and the Step 1 findings *before* weighing the user's supplied criteria — challenging supplied acceptance criteria on both **completeness** and **correctness** — **written that derivation to its observable artifact (`.devflow/tmp/issue-derivation-<slug>.md`) before any clarification round**, and you drive clarification from the **delta plus conflicts**. The gate before the first clarification round (and the per-round re-check) confirm the file — or, in a read-only sandbox, its visible inline-in-chat stand-in — exists, so a skipped pass is surfaced rather than assumed. See the independent-derivation pass below; it mirrors "Solution-space expansion before convergence," generalized from the implementation-approach fork to the problem, behavior, edge-case, and acceptance-criteria forks.
- [ ] **Problem & beneficiary** — who hits this pain and why it matters. (Not "users want export" — *which* users, doing *what*, blocked *how*.)
- [ ] **Single coherent scope** — the issue is exactly one feature/fix. If the story bundles two or more ("export results *and* notify when ready"), ask the user whether to split; default to one issue per feature.
- [ ] **One decided behavior per fork** — every place the story could mean different things (format, channel, trigger condition, access model, edge-case handling) has *one* chosen answer. No "or", no "either", no "default for now".
- [ ] **Solution-space expansion before convergence (do this BEFORE the implementation-approach fork below).** The user's proposed mechanism is an *input*, not the menu. Before asking the user to pick an approach, independently generate the full space of mechanisms that could solve the stated problem — and in particular ask: *is there a categorically STRONGER class of mechanism than what the user proposed?* Mechanisms differ in **strength of guarantee**, not just shape. A useful ladder, weakest→strongest: (a) documentation/wording that asks a human or agent to remember; (b) an in-process self-check that *warns* when the thing was skipped; (c) a *deterministic* mechanism that runs regardless of the actor's choices (a harness/hook/wrapper that fires after-the-fact, an enforced gate, an idempotent backstop that re-does the work). If the user proposed only (a) and/or (b), you MUST generate at least one (c)-class candidate and put it on the menu. Name the strongest viable mechanism even if the user did not think of it — surfacing it is your job, not theirs. If after genuine effort no stronger class exists (or the stronger classes don't apply to this problem — e.g. a harness fires at the wrong point), say so explicitly in the question rationale rather than silently omitting it.
- [ ] **One implementation approach** — present the expanded mechanism menu (including the strongest class found above), and the user has picked one. You surface each option's **guarantee-strength** and trade-offs *in the question*, not in the issue.
- [ ] **Concrete acceptance criteria** — you can state each as a single unconditional, testable assertion. If an AC would need a conditional ("if links are public…"), the underlying fork isn't resolved yet — go ask.
- [ ] **Known prerequisites (cross-issue ordering)** — candidate prerequisite issues (another issue/PR that must land before this work can *start*) are inferred from the story and the Step 1 findings, and **when at least one candidate exists** they are confirmed with the user via the runner's user-question tool. A story with **no** inferable prerequisite gains **no new question** — like the visual-specification path, this item adds a question only when it applies. Confirmed **still-open** prerequisites populate the draft's `## Dependencies` section as `Blocked by #N — <reason>` lines (see `references/issue-template.md`); already-closed ones are recorded as Technical Context provenance instead. (This is the deliberate authoring producer for the `/devflow:implement` early Phase 1 dependency preflight.)
- [ ] **Visual specification (user-visible UI changes only)** — for an issue you infer involves user-visible UI, either a screenshot/mockup is recorded or the visual details are verbally verified with the user. **Skipped entirely for non-UI issues**, which gain no new questions. See the *Visual-specification guidance* below for the dimensions to verify and the screenshot-preferred-not-mandatory rule.

**Independent-derivation pass (mandatory, before any clarification round — runs on every run regardless of how complete the story looks).**

A story that *looks* fully baked — structured sections, its own acceptance-criteria list — is the trap this pass defuses: when criteria arrive pre-written, the temptation is to treat them as answers "already known from the user story" and skip derivation, so the supplied list silently anchors the whole issue. A terse one-line story produces a *better* issue precisely because there is nothing to anchor on. This pass removes the anchor by **ordering**, not by isolation:

1. **Derive first, read the supplied answers second — and write the derivation to an observable artifact.** Before reading the user's acceptance criteria (and any other pre-supplied decisions) *as answers*, independently re-derive the full Definition of Ready above — problem/beneficiary framing, every behavioral fork, edge/error cases, and the acceptance criteria — from the stated problem and the Step 1 codebase findings. **Write that derivation to `.devflow/tmp/issue-derivation-<slug>.md` before any clarification round begins** — run `mkdir -p .devflow/tmp` first, then **pick the kebab-case `<slug>` once (from the user story's topic) — reusing the slug already bound by this invocation's inline-checklist step if one exists, and binding it here only when this invocation has bound none — and reuse that exact path for the rest of this run**, so the gate and per-round re-checks below look for the same file and a later re-derivation can't make them false-fire. **As the very first action of this pass — before you derive anything — delete any pre-existing `issue-derivation-<slug>.md`, then write it fresh** (a prior run on the same topic yields the same slug, so a leftover would otherwise sit at this path; deleting it up front means the only way the file exists at gate time is that *this* run's pass produced it — a skipped pass leaves no file for the gate to pass on). This reuses Step 4's gitignored-`.devflow/tmp/`, pick-the-slug-once convenience-copy convention (Step 4 writes `issue-draft-<slug>.md` under the same rule); reuse the same `<slug>` for both so one issue keeps a single stem. **Note the deliberate root divergence:** this derivation artifact stays **cwd/worktree-anchored** (it is an internal gate re-checked across turns within one session, where cwd is stable), whereas Step 4's user-facing *draft* is written under the **main** repo root (`resolve-main-root.sh`) so its displayed path opens correctly from a worktree — same `.devflow/tmp/` subpath and slug, intentionally different root; do not unify them. On a filesystem that refuses the write, follow `references/fallback-read-only-sandbox.md`, loaded per the root's routing table. The artifact is the observable proof of the ordering: you commit your own list to a file before the user's list can frame your thinking. This is a forced self-check **with an observable artifact**, not an isolation guarantee — you still see the supplied criteria; you simply derive, and record the derivation, *before* weighing them. (A blind-subagent variant that never sees the supplied list was rejected here, to keep this **derivation pass ordering-based rather than isolation-based** and avoid the added latency; the derivation artifact adds no new script, dependency, or subagent — only the `mkdir -p` + file write Step 4 already performs. The pipeline stays inline overall; its sole subagent is the **Step 3.6 fresh-context auditor**, whose entire value *is* context isolation — a distinct mechanism applied at a distinct seam, after the draft is assembled, not during derivation.)
2. **Treat supplied acceptance criteria as suspect on two axes.** *Completeness* — which behavioral forks, edge cases, and factors does your independent list have that theirs omits? *Correctness* — is each supplied criterion atomic, testable, and a genuinely resolved decision, or a buried unresolved fork wearing the costume of a decided one? A polished, comprehensive-looking list earns the **same** scrutiny a terse story gets, since it has more to challenge.
3. **Drive the clarification rounds from the delta.** Diff your independently-derived list against the story and clarify from the **delta plus any conflicts** — the forks, edge cases, and factors you derived that the story left unresolved; the supplied criteria that fail the correctness test; and anywhere the two lists disagree. The supplied criteria are one of the two inputs to that diff, never a shortcut past it.

This pass **feeds** the clarification rounds; the no-options gate (Step 3) still governs the final body. Its output is a working derivation to clarify against — never a place to park options in the issue itself.

**Evidence-bundle sub-pass (mandatory, part of the independent-derivation pass — runs on every run, after the Definition-of-Ready derivation and before the first clarification round).**

Recent runs asked the user to approve an implementation mechanism before the producers, values, consumers, execution tiers, lifecycle states, failure paths, migrations, and coupled surfaces that mechanism touches had been verified — so the missed surfaces surfaced serially across the Step 3.5 and Step 3.6 rounds instead of before the approach question was ever asked. This sub-pass records that evidence up front. It asks the user nothing (it draws nothing from the ~18-question clarification budget) and, mirroring the Step 3 drafting rule, performs **no whole-codebase re-exploration** — its reads and searches are targeted, seeded from the Step 1 findings (plus the currency re-evidencing below, seeded from clarification answers and from mechanism classes added during solution-space expansion).

- **Axis floor.** After writing the derivation, append a `## Evidence bundle` section to the same `.devflow/tmp/issue-derivation-<slug>.md` artifact (in a read-only sandbox, to the same visible inline-in-chat block that stands in for it). The bundle records, per axis on the effective list, what the mechanism space under discussion touches, covering **at minimum** these generic axes: authoritative producers and the values they emit; consumers of each touched value or surface; execution environments (tiers, host OSes, degraded arms); persistence paths; lifecycle states and termination paths including retries and backstops; migration and coexistence surfaces (what is already deployed that the change must coexist with); and coupled tests and docs. The `at minimum` marker is deliberate — a consumer `## Evidence axes` section extends the list.
- **Entry form.** Every axis entry takes exactly one of three legal forms — one to three lines in `Verified:` form (the executed command or file reference plus what it established), the single line `N/A — checked <how>` (the axis genuinely does not apply, with the check named), or the single line `unestablished — <why>` (the probe was unavailable or denied, or the entry was superseded by a later clarification answer) — and **an axis with no entry is not a legal bundle state.** **Proportionality clause:** on a run whose mechanism space plainly does not touch a set of axes (a one-line docs fix touches no migration surface), those axes may be batch-marked `N/A — checked: scope inference, <one line>` in a single pass, so a trivial issue pays seconds rather than a seven-probe tax; a scope-inference entry is **not citation-grade** (a `(Recommended)` citation must name an entry whose check was executed).
- **Self-describing header.** The `## Evidence bundle` section opens with a compact fixed header restating the three entry forms, the three currency triggers, and **the effective axis list as computed at pass time**, so every per-round re-check — which already re-reads the artifact — recovers both the operating rules and the completeness comparand from disk rather than from evictable turn-one context, and each gate site cross-checks its freshly recomputed effective list against the header's pass-time list (a difference is a currency trigger).
- **Information diet unchanged.** Bundle claims the **draft relies on** land in the issue's `Technical Context` as `Verified:` bullets, where the Step 3.6 auditor re-verifies them through the draft; **the `## Evidence bundle` section itself stays inside the derivation artifact and is never carried into the Step 3.6 dispatch prompt.** The audit's file-arm and embed-arm out-of-bounds lists already name `issue-derivation-<slug>.md`, so the bundle rides inside an already-excluded artifact — the count-locked out-of-bounds list literals (and the suite pins that assert them) are byte-unchanged by this pass.

**`## Evidence axes` forwarding.** At pass time the skill re-runs `load-prompt-extension.sh create-issue --section '## Evidence axes'` fresh (via the portable anchor recipe above, and for the same compaction rationale the Step 3.5/Step 3.6 re-loads state) and appends any section headed exactly `## Evidence axes` to the generic axis floor; an extension without that heading still contributes no consumer section, and that absent heading is now breadcrumbed and reported rather than invisible. **On any loader stderr from this re-load, read the exit status first.** A loader **exit-2 refusal** (a malformed flag or a guard failure) at this site is an **unestablished** load, surfaced per the loader-failure arm's discipline below — never laundered into the designed absent-heading no-op. **At exit 0, an absent-heading breadcrumb from a non-empty extension** is instead the designed no-op: report the headings the breadcrumb lists — surfacing a near-miss heading (case drift, say) and equally the routine single-hook extension that carries only the other heading — and then proceed treating the selection as no consumer section for that hook: a **report-then-proceed** step, never a stall, a user question, or a degraded-arm claim. **The heading-extraction rule is owned by `scripts/render-audit-prompt.py` (issue #600)** — the loader invocation above stays this hook's operative fresh re-load (its stderr absent-heading breadcrumb is what the report-then-proceed step reads), and the renderer exposes the **same rule** as `render-audit-prompt.py extract --hook evidence-axes`, applying the positional two-marker delivery check — `render-status:` first line, `render-end:` last line. The renderer implements the four extraction clauses (a section spans its heading line to the next line beginning `## ` — two hashes plus a space, so a `###` sub-heading terminates nothing — else to end of file; duplicate same-heading sections concatenate in file order; an empty section equals an absent heading; and a heading line inside an HTML comment block or a fenced code block is not a heading, an unclosed fence running to end of file), documented in `skills/create-issue/references/audit-prompt-template.md`; the **same extraction rule also serves the Step 3.6 `## Audit dimensions` hook**, which consumes it spliced into a dispatch arm as `{CONSUMER_DIMENSIONS}` rather than through a standalone `extract` call — so both hooks resolve against this single owner, and `scripts/load-prompt-extension.sh` remains the sibling implementation for its own `--section` callers. **Loader-failure arm:** a denied or failed pass-time re-load is surfaced, never treated as "no extension" — the bundle records the dedicated line `consumer axes: unestablished — loader denied or failed`, the gate evaluates the generic floor with that line present, and the Step 3.5/Step 4 coverage reporting names the effective list as **floor-only-degraded** (the same unknown-is-not-zero discipline the rest of the pipeline uses).

**Bundle-coverage gate.** The bundle-coverage gate fires at the **same two sites** as the derivation gate it extends — before the first clarification question, and unconditionally at the Step 3 drafting precondition on every run — so the zero-question path, the disengagement path, and the ordinary path are all covered. At each site it requires the `## Evidence bundle` section to exist with an entry for every axis on the effective list (the generic floor plus any consumer `## Evidence axes` section), and **each gate site recomputes the effective list** — re-running `load-prompt-extension.sh create-issue --section '## Evidence axes'` per the fresh re-load recipe — so the comparand is never recalled from evictable context. **On any loader stderr from this re-load, read the exit status first.** A loader **exit-2 refusal** (a malformed flag or a guard failure) at this site is an **unestablished** load, surfaced per the Step 2 loader-failure arm's discipline — never laundered into the designed absent-heading no-op. **At exit 0, an absent-heading breadcrumb from a non-empty extension** is instead the designed no-op: report the headings the breadcrumb lists — surfacing a near-miss heading (case drift, say) and equally the routine single-hook extension that carries only the other heading — and then proceed treating the selection as no consumer section for that hook: a **report-then-proceed** step, never a stall, a user question, or a degraded-arm claim. The gate reuses the derivation gate's shape: this-run freshness, the read-only-sandbox visible inline stand-in, and on-disk distrust when the filesystem is read-only.

**Bundle currency (per-round re-check).** The existing per-round Definition-of-Ready re-check additionally re-checks bundle currency against **three triggers** — a clarification answer that changes the mechanism space; a clarification answer that contradicts an axis entry; and a mechanism class added to the approach menu after the bundle was written (drafter-generated during the mandated solution-space expansion, or user-proposed) — updating the affected entries or re-marking them `unestablished — superseded by <the trigger>` before the approach question is asked, so the `(Recommended)` citation is current as of the question. **Asking-order rule:** the implementation-approach fork question is asked in its own question-tool call, never batched with any question whose answer is a currency trigger, and only after the mechanism-space-shaping answers are in; if an answer arriving in the approach question's own call nevertheless invalidates the cited entry, the approach answer is treated as unconfirmed and the question is re-asked with a refreshed citation.

**Visual-specification guidance (user-visible UI changes only — skipped entirely otherwise).**

Some issues change what the user sees, and for those a text-only description leaves the visual
intent ambiguous. Handle the visual specification as part of this Step 2 clarification — as
**prose guidance you follow, not a new hard gate** (it adds no enforcement mechanism; an
unresolved item lands in the existing Blocked section like any other):

1. **Infer whether the issue involves user-visible UI changes** as part of your normal scope assessment above — an inference, not a dedicated "is this UI?" question. When the issue is obviously non-UI (a script, a config key, an internal doc, a CLI-only change), **the whole path below is skipped and adds no new questions**; a false positive here is the failure to avoid, so do not ask visual questions on a non-UI issue.
2. **On a UI change, check the user-provided resources/context** — pasted images, attached files, URLs, and design-tool links such as Figma — for an existing screenshot or mockup before asking for anything.
3. **If a screenshot/mockup is present, record it** in the issue's Visual Specification section (see `references/issue-template.md`): embed it when a hosted URL is available, otherwise reference it with a one-line note on how the implementer can obtain it. Do not then ask the user for one they already supplied.
4. **If none is present, ask the user to provide a screenshot or mockup** via the runner's user-question tool (the same tool Step 2 uses elsewhere).
5. **If the user has none, verify the visual details with the user before finalizing the draft** — pinning down, as applicable to the specific task: placement & layout, visual states (hover/focus/error/empty/loading/disabled), responsive behavior across breakpoints, and design-system/style match, plus any further visual dimension the task makes relevant. This checklist is a **non-exhaustive prompt**, not a fixed form: add task-specific dimensions and skip inapplicable ones.
6. **Write the screenshot reference and/or the verified placement description into the Visual Specification section** of the drafted issue.

A screenshot/mockup is **preferred, not mandatory** — verbal verification is an accepted substitute, so a UI issue is never blocked solely for lacking an image. If a UI-placement detail is still unresolved when the user disengages, it **flows to the existing `## 🚫 Blocked` section like any other unresolved decision** (the existing mechanism from the disengagement rule below — not a new gate).

**How to ask:**

- **Gate — confirm *this run's* derivation artifact exists before the first clarification round.** Before the first clarification question — **or, if the story is complete enough that you will ask none, before drafting in Step 3** (the gate must fire on every run, not only ones that clarify; see the Step 3 drafting precondition, which is the unconditional backstop for the zero-question path) — verify `.devflow/tmp/issue-derivation-<slug>.md` is present **and holds the derivation you produced for the current story** — presence alone is not enough, because the slug is deterministic from the topic, so a prior run on the same topic can leave a stale file at this exact path. (The pass above deletes any leftover before writing, so on a correct run the file is unambiguously this run's; the freshness check is the backstop for the case the pass was skipped, where deletion never happened either.) If the file is missing, or you cannot confirm its content is this run's derivation, the independent-derivation pass was skipped — **stop, run the pass and write the artifact now, and only then ask anything.** Do not begin clarification without it; surface its absence rather than silently continuing. On a filesystem that refuses the write, follow `references/fallback-read-only-sandbox.md`, loaded per the root's routing table. **The same gate also confirms this run's `## Evidence bundle` is present and axis-complete** (an entry for every axis on the effective list, recomputed here per the *Bundle-coverage gate* above) — a missing section or a missing axis entry means the evidence-bundle sub-pass was skipped, and the same stop-and-run remedy applies.
- **Ask via the runner's user-question tool** — `AskUserQuestion` (Claude Code, the canonical example), or the equivalent your runner exposes (e.g. `ask_user` on GitHub Copilot CLI). Source the questions from the independent-derivation **delta plus conflicts** (the pass above). Batching is **conditional on the tool**: where it supports multi-question batches (as `AskUserQuestion` does), batch 2–4 related questions per call rather than one long interrogation; where it is one-question-per-call, ask them sequentially (as on `ask_user`) — the total budget below is the same either way.
- For each question, offer concrete multiple-choice options. When the codebase or findings make one choice clearly best, list it **first** and mark it `(Recommended)` with a one-line why. **Where the tool cannot present structured choices** (a free-text-only question tool), enumerate the concrete options in the question text itself — best first, that one marked `(Recommended)` — so the multiple-choice discipline holds regardless of the tool's affordances.
- **On the implementation-approach fork question specifically, the `(Recommended)` one-line why cites at least one bundle entry by axis name, and the cited entry must be in `Verified:` or `N/A — checked` form** (an `unestablished` entry cannot ground a recommendation). That question's rationale additionally **discloses by name every effective-list axis that is `unestablished` at ask time, plus the count of axes batch-marked N/A by scope inference**, so an approval solicited on thin evidence is visibly so to the user at the approval moment; the question is still asked (this skill is best-effort and never blocks issue creation), with the gap disclosed rather than hidden. `(Recommended)` markings on **other** clarification questions (behavior forks, formats, channels) keep the existing one-line-why rule and gain no citation or disclosure obligation — the bundle's axes are mechanism-space facts, and forcing a citation on a non-mechanism question would misground it. **No-citation-grade arm:** when no citation-grade entry exists at ask time (every entry inference-marked or unestablished), the drafter executes exactly one targeted probe to upgrade the most decision-relevant axis before asking; if that probe too is unavailable or denied, the question is asked with **no** `(Recommended)` marking and its rationale states that no citation-grade evidence exists — the marking is withheld visibly for evidence reasons, never silently.
- After each round of answers, **re-check the Definition of Ready** — including that the derivation artifact still exists and still holds this run's derivation (or, in a read-only sandbox where the write failed, re-post the visible inline-in-chat stand-in in this turn), **and re-check bundle currency per the *Bundle currency* rule above** (update or re-mark superseded any axis entry the round's answers changed or contradicted, and any axis a newly-added mechanism class touches). If the file has vanished — or, on a read-only filesystem, you have not re-posted the inline derivation this turn — treat it exactly as the gate above: stop and re-write the file (or re-post the inline derivation) before asking the next batch. If gaps remain, ask another batch. Keep going until the list is fully satisfied — do not draft with items still open.
- Cap clarification at a **runner-neutral total-clarifying-question budget** of ~18 questions total (equivalent to the ~6 batches of up to 3 this previously allowed) — a *count of **answered** questions*, not a count of rounds or of questions merely asked, so a one-question-per-call runner gets the same budget as a batching one instead of being starved to ~6 single questions. A silent non-response is **not** an answer: it never counts toward the budget and never trips "treat the remainder as disengagement" — re-ask it instead (see the silent-non-response rule below, which defines what counts as silence). If the budget is exhausted by **explicit** answers and facts are still missing, treat the remainder as disengagement (below).

**Push back once before accepting total disengagement.** If the user disengages while the issue is still *unbuildable* — no decided scope, or the single core behavior fork is still open (the ticket would be almost entirely Blocked) — do not silently produce a hollow issue. Say so plainly, once: e.g. *"As-is this issue won't be buildable — every decision is still open. Can you answer just two things: (1) is this one feature or several, and (2) <the single most defining behavior fork>? Otherwise I'll file it with everything flagged as blocked."* Ask those via the runner's user-question tool (one batch where it batches, sequentially where it is one-question-per-call). If the user answers, continue clarifying; if they give an explicit disengagement signal again (silence is not one — see the silent-non-response rule below), proceed below. This push-back happens **at most once** — do not nag. It does not apply when only peripheral forks remain open (e.g. link expiry) — Block those without comment.

**A silent non-response is not disengagement — never proceed on silence.** A question-tool timeout, a `No response after 60s` return, or the user simply stepping away is **not** a disengagement signal — the user's answer often arrives seconds later. When a question has been asked and no answer has come back, **pause and re-ask that question in your final chat message** (where the user can answer at their own pace); do **not** proceed, draft the issue, guess a default, or route the item to the Blocked section on that basis. Only an **explicit** reply from the user — "just create it", "you decide", or "I don't know" — or an exhausted clarification budget of **explicit** answers (the budget rule above), engages the disengagement / draft-from-decided / Blocked path below; a silent non-response never does. This mirrors the Step 4 confirmation gate: when the user has not responded, the pipeline stays paused rather than moving on without them.

**When the user disengages** — gives one of the explicit signals "just create it", "I don't know", or "you decide" to a Definition-of-Ready question (these three explicit replies are the *only* user-reply disengagement triggers — the sole non-reply entry is an exhausted budget of explicit answers, above; silence is never a trigger — see the rule directly above):

- Stop asking. Draft the issue from what *is* decided.
- Every still-unresolved Definition-of-Ready item goes into the issue's **`## 🚫 Blocked — resolve before implementation`** section, phrased as a direct question with one line on why it blocks work (see template).
- Do **not** invent a default and bury it in the body. Do **not** rephrase the open decision as an "option" or "recommended approach" elsewhere. The Blocked section is the *only* place an unresolved decision may appear.
- "You decide" is not permission to guess silently — it is an unresolved item that belongs in Blocked, unless the choice is genuinely inconsequential to scope (e.g. a variable name).
