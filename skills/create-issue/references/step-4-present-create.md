### Step 4: Review with the user, then create

**The issue is never created until the user has seen the full rendered draft and explicitly approved it.** The user story they gave you was a rough input; this is their one chance to read the assembled ticket as a whole and correct it before it becomes a real, notification-sending GitHub issue. This gate is **unconditional** — it applies no matter how thoroughly Step 2 resolved every decision, and no matter what the user said earlier.

**Presentation gate — confirm *this run's* audit artifact exists before the draft is shown.** Mirroring the Step 3 derivation-artifact gate: before showing anything, confirm `.devflow/tmp/issue-audit-<slug>.md` is present **and holds this run's audit** (a Step 3.6 report, not a same-slug leftover from a prior run — the Step 3.6 write deletes any leftover first, so on a correct run the file is unambiguously this run's). On a filesystem that refuses the write, follow `references/fallback-read-only-sandbox.md`, loaded per the root's routing table. If neither this run's artifact nor its inline stand-in is present, Step 3.6 was skipped — **stop and run it now before presenting the draft.**

1. **Show the complete rendered issue in chat.** Post the exact title and the full body — every section, verbatim, as Markdown — directly in your message. Do not summarize it, abridge it, or describe it; the user reviews the literal text that would be filed. Do not use the runner's user-question tool (`AskUserQuestion` or its runner equivalent) to stand in for showing the body (such tools truncate); render the body first, then you may use it for the confirm/edit prompt if you wish.
2. **Also write the draft to a file for easy review — at the MAIN repo root, so its displayed path is correct inside a worktree.** Reuse the kebab-case `<slug>` you picked for the Step 2 derivation artifact (so one issue keeps a single stem; Step 2's pass runs on every run, so there normally is one). Only if Step 2 never bound a slug should you derive a short kebab-case slug from the issue title here (e.g. "Add CSV export for survey results" → `add-csv-export-for-survey-results`) — its different source (title vs. the topic Step 2 uses) doesn't matter, since the reuse path is the norm. **Resolve the main working-tree root** with the bundled `resolve-main-root.sh` (via the same portable anchor recipe used above), then create its `.devflow/tmp` and write there:
   ```bash
   MAIN_ROOT="$("${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../scripts/resolve-main-root.sh)"
   mkdir -p "$MAIN_ROOT/.devflow/tmp"
   ```
   Capturing the resolver's **output** (the root path) into `MAIN_ROOT` is fine — the *anchor* itself is still expanded inline in that single statement, per the never-capture-the-anchor rule above; only the resolved root is reused. Write the rendered title + body to the **absolute path** `<main-root>/.devflow/tmp/issue-draft-<slug>.md` (i.e. `$MAIN_ROOT/.devflow/tmp/issue-draft-<slug>.md`), with the title as a top `# ` heading above the body — that path is gitignored, so it never lands in a commit. Using the **main** root (not the cwd) is deliberate: inside a linked Claude worktree (`.claude/worktrees/<name>/`) the cwd-relative path would resolve to a file the user — whose editor is rooted at the main repo — cannot open; the resolver returns the main repo root in a worktree, the repo root in a normal checkout, and falls back to cwd (with a stderr breadcrumb) when git can't answer. This is a distinct resolution from the `.devflow/config.json` read, which stays cwd/worktree-anchored (per #295) — do **not** unify them (nor with the tiered *draft* root this write binds below). **Bind the draft root once, then reuse it (issue #569) — bind if not already bound.** The binding is recorded by the **first landed canonical-draft write** of the run; on the normal path that write is Step 3.6's pre-dispatch write, which carries the query-then-bind procedure. **Do not assume what that site did — detect the case here:** read `query-draft-binding "<slug>" --nonce "<nonce>"` and branch exactly as Step 3.6's binding procedure specifies — already-bound (skip, take the `bound=` root), `bound=none` with no `reason=` (run Step 3.6's `record-draft-binding … --tier main-root` fence here; the procedure is query-first and idempotent, so running it at either site is safe, and `binding-already-recorded` is the benign expected outcome), or `reason=foreign-nonce` (Step 3.6's **foreign-nonce arm** — surface the drift, never the quiet unbound arm, and never compose a path). **The case this binding branch serves:** Step 3.6 ran `init` — so a state file exists — but its pre-dispatch write did not land, so the run arrives here with `bound=none` and the conditional bind above is what records the binding. On the `state-owner unavailable` fallback the picture is different: there is no state file at all, so `record-draft-binding` cannot bind here either and the run stays legitimately unbound. **An unbound run is legal and must stay safe**: `none` is a decided token, not a path, so **never compose a path from it**. Fall back to the root resolved by the `MAIN_ROOT=` fence above for this write, and follow the `bound=none` display arm in sub-step 3.

Every **later** write site (a Step 3.6 re-dispatch write and the sub-step 4 overwrite) — the sites that run in a **later turn**, after a possible context compaction — then obtains the bound root from `query-draft-binding "<slug>" --nonce "<nonce>"` (its `bound=` field — never context recall and never a second `resolve-main-root.sh` resolution), so a compacted context cannot drift which root the run writes, displays, and audits against; `<bound-root>` in the sub-step 3 display note is that same `bound=` root. **This presentation write is deliberately not in that enumeration:** it is the same-turn write whose root is already in hand from the `MAIN_ROOT=` fence directly above it (equal by construction on the normal path — a `main-root`-tier binding resolves to this same root), so it needs no re-read. When the binding read reported `bound=none`, `<bound-root>` is unavailable for these sites too — write to the main root resolved for this turn and take the `bound=none` display arm rather than composing a relative `none/…` path. On the **file arm**, forward the exact absolute canonical path to `record-dispatch --write-path <that path>`, which the state owner cross-checks against the recorded binding and refuses with `write-path-mismatch` on divergence. (The full tier-2 active-worktree / tier-3 embed selection ladder and the strict *binding-required* file-arm enforcement are the coupled follow-up to this pass; a `main-root`-tier binding is what ships here, so the bound root equals the resolved main root.) **Keep this one `<slug>` for the issue throughout, reusing that exact absolute path from then on** — so revisions overwrite the one file (next sub-step) and a second or third issue you draft later gets its *own* distinct file instead of clobbering this one. This is a **convenience copy** so the user can open the draft in their editor; it never replaces showing the body in chat (todo above). On an **embed/inline-arm epoch** it is **not** the source the issue is created from; on a **file-arm epoch** this canonical file *is* the posting source — but only **through the gated `emit-body` emitter** (sub-step 5), never by passing the path to `--body-file <path>` directly. If the write fails (e.g. read-only main root / sandbox), note it briefly and continue **without showing a path** — the chat render is what matters, and a path to a file that was not written would only mislead.
3. **Ask for explicit confirmation or feedback.** Plainly invite the user to either approve creation or request changes. **The draft-save-location note (the absolute `<bound-root>/.devflow/tmp/issue-draft-<slug>.md` path from sub-step 2) always renders *below* the rendered issue preview — never above it**, so every confirmation message has the same layout: the full rendered body (sub-step 1) first, then the **one-line Step 3.6 audit summary**, then the draft-path note, then the confirm/edit question. The summary line is one of the root's non-degradable invariants, so it renders on every arm and on every round count. **Do not assemble its fields from your own recollection of the run — read them from `query-summary`** and render what it reports:

```bash
python3 "${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../scripts/issue-audit-state.py query-summary "<slug>" --nonce "<nonce>" --draft-file "<absolute issue-draft-<slug>.md path>"
```

It reports the findings count, revisions applied, verdict, rounds run, whether a consumer `## Audit dimensions` section was appended, the degraded flag, the user-declined and cap-reached flags, any embed-arm markers, the eligibility token, `attestation=<match|mismatch|attestation-unavailable|none>` (what the post-creation attestation established; `none` before any creation attempt), and — new for the evidence bundle (issue #548) — `adjudicated_verdict`, the per-actionability-class counts (`must_revise`, `advisory`, `invalid`), and `unresolved_must_revise` for the latest completed round — plus, rendered immediately before `bound_root=`, the run-wide `effective_unresolved` count and the `convergence_basis` token. **The Step 4 audit summary line reports the bundle's coverage (`axes evidenced, axes marked N/A, axes unestablished`) plus those post-adjudication counts by actionability class, the number of unresolved must-revise findings attributable to an axis on the effective list as frozen at that round's close, and — distinctly, never collapsed onto it — the run-wide effective unresolved count after every post-close resolution, reopen, invalidation and supersession** (a round whose actionability or axis attribution could not be established renders `unestablished` rather than a number — unknown is not zero, and that applies to the effective count exactly as it does to the at-close one). **When `convergence_basis` names a resolution basis, the line additionally carries the phrase `converged via self-verified resolution`** — rendered on the stale variant as **`converged via self-verified resolution (stale — later revisions unverified)`** — so a convergence the drafter verified for itself is never rendered indistinguishably from one an auditor's `VERDICT: FILE` vouched for. Render them as one line (e.g. *"Fresh-context audit: 2 findings, 1 revision applied, VERDICT: FILE, 2 rounds (no consumer audit dimensions); bundle: 5 axes evidenced, 2 N/A, 0 unestablished; adjudication: 1 must-revise, 1 advisory, 0 invalid, 1 unresolved at close, 0 effective; converged via self-verified resolution."*), carrying each flag the query reports: the word "degraded" when it reports degraded, the phrase `user declined further audit` when it reports a decline, the ceiling when it reports the cap, and the embed marker when it names one. On the `state-owner unavailable` fallback — where the query itself is what failed — the line carries that marker instead and is composed in-chat; it still renders. Display the **absolute path built from the bound root** (`<bound-root>/.devflow/tmp/issue-draft-<slug>.md`, with `<bound-root>` read back from `query-draft-binding`'s `bound=` field), never `$MAIN_ROOT` from context recall and never a bare relative `.devflow/tmp/…` string — since the write records the binding and the display reads that same bound root back, the shown path always resolves to the exact file written. Separate the note from the preview rather than embedding it inline ahead of the body — e.g. after the rendered body, add a line such as *"Draft also saved to `<bound-root>/.devflow/tmp/issue-draft-<slug>.md` for review."* (with `<bound-root>` replaced by the resolved absolute bound root — the `bound=` root `query-draft-binding` reports, which on a `main-root`-tier binding is the resolved main root) and then ask *"Want me to create it as-is, or change anything first?"* If the sub-step 2 write failed, omit the draft-path note entirely rather than showing a path to a nonexistent file. **`bound=none` arm (no `reason=`):** when `query-draft-binding` answers the literal `bound=none` token for a legitimately unbound run, there is no bound root to display; **never** substitute the token into the path (that would show the user a relative `none/.devflow/tmp/issue-draft-<slug>.md` under the cwd). **Omit the draft-path note entirely**, exactly as the write-failure arm above does, rather than showing a misleading path. **`reason=foreign-nonce` is a distinct arm, never this one:** the run *is* bound and only the nonce drifted, so a quiet omission would hide a state in which the tool emits from the recorded binding while you write elsewhere. Take Step 3.6's **foreign-nonce arm** — recover the nonce and re-query, else stop and tell the user the drift — instead of quietly omitting the note.
4. **Iterate on feedback.** If the user requests any change, revise the draft, re-run the no-options gate (Step 3), then run **Revision-delta verification**, then run **Ledger maintenance after a revision** — record the revision with `record-revision --after-round <round>` and, when the per-finding verification confirmed at least one finding fixed, `record-resolution` naming those confirmed ids against the printed revision ordinal (cross-round when the fix clears an earlier round's entries); when it confirmed none fixed, record no resolution; a later-discovered regression uses `record-reopen`, and a finding discovered misclassified uses `record-invalidate`, never `record-resolution`. Make each of those decisions against the `query-findings` read-back, never context recall — and **overwrite the same `<bound-root>/.devflow/tmp/issue-draft-<slug>.md` file (the identical absolute path from the recorded binding — read it back via `query-draft-binding`, never re-resolve it) you chose in sub-step 2** (keep the original filename even if the title changes, so you don't leave orphaned drafts behind). If that read-back carries `reason=foreign-nonce`, take Step 3.6's **foreign-nonce arm** (recover and re-query, else stop and tell the user) — do not treat it as unbound. If it answers a bare `bound=none` with no `reason=`, the run is unbound: **do not compose a path from the token** — overwrite the file at the main root resolved for this turn (`resolve-main-root.sh`, the sub-step 2 recipe), and omit the draft-path note when re-presenting, per sub-step 3's `bound=none` arm. Each revision is re-presented in full — never apply edits and create in the same turn without showing the updated draft.

   **Order matters here — resolve the re-audit offer BEFORE asking for approval.** A just-revised draft is shown in-loop under `query-eligibility --mode iterate` (`iterate=ok ordinal=N`), which authorizes the **re-show only** — it is never a ground for acting on approval and never a ground for creation. So: show the revised draft, then **offer a fresh re-audit** via the runner's user-question tool. On **"yes"**, dispatch one. On **"no"** — the user declining further auditing on these just-revised bytes — record that decline with `record-override --kind user-decline --surface step4-offer` (on a **file-arm epoch** additionally pass `--draft-file "<absolute issue-draft-<slug>.md path>"` so the override is digest-bound to the bytes; omit it on embed/inline epochs). Recording this override is what lets `query-eligibility --mode approve` answer `eligible=yes` for the still-unaudited bytes — **without it, a draft the auditor keeps returning `VERDICT: REVISE` on can never clear the sub-step 5 eligibility gate and filing deadlocks, which the audit must never do (it informs, it never blocks filing).** Only once that offer is resolved do you ask the confirm/edit approval question, on bytes that `query-eligibility --mode approve` answers `eligible=yes` for. **Approval is solicited only on bytes eligible in `approve` mode**; asking the user to approve a draft whose eligibility has not been established invites a yes you cannot act on. Repeat until the user approves.

   This sub-step-4 offer is the **only offer surface inside Step 4's iterate-on-feedback loop** — the Step 3.6 → Step 4 boundary's T1/T2 offer fires only *before* presentation, so it never double-asks about a revision this loop handles (one offer per pause). A dispatched re-audit runs the pre-dispatch canonical-draft write first, then `query-arm` → `record-dispatch` → dispatch, exactly as Step 3.6 specifies. Its return is handled by the **same loop as Step 3.6** — `record-return`, obey `query-next-action`, verify findings against the code, revise, re-run the no-options gate, then run **Revision-delta verification**, and re-present the full draft — and its report **overwrites the same `.devflow/tmp/issue-audit-<slug>.md` artifact** (updating the audit summary line accordingly).
5. **Create only on explicit approval.** **Creation-time confirmation re-check (path-agnostic — this is the fourth re-read anchor named above, and it runs on every path, tool or fallback).** Immediately before running `gh issue create`, re-confirm two things in the current conversation: the user has explicitly approved the rendered draft, and the active tracking mechanism shows item 6 in progress — read it from the task tool (its read-back call, or the conversation record of the last status update where the tool exposes no read-back) on the tool path, or from the checklist state file on the inline-checklist fallback path. On the fallback path an absent or unreadable state file is a pipeline-paused signal, not a clear-to-create one: rebuild and rewrite it, and create only on explicit in-conversation approval, never on the guard's silence. Once the user clearly says to create it (e.g. "yes", "create it", "looks good, file it"), **first confirm `query-eligibility --mode approve` answers `eligible=yes`** — the user's approval and the tool's eligibility are two different gates, and creation needs both. **If it instead answers `not-eligible` because the bytes still carry an unresolved `VERDICT: REVISE`/`no-verdict` and the user — after the re-audit offer surface has been exhausted (an accepted re-audit returned `VERDICT: REVISE` and they have chosen to file the revised bytes without another round) — is explicitly electing to file anyway, record that election with `record-override --kind user-decline --surface step4-approval-after-exhausted-offer` (`--draft-file`-bound on a file-arm epoch) and re-query. This is the sanctioned path by which the audit informs without blocking a determined filer; proceed only once `query-eligibility --mode approve` answers `eligible=yes`.** Then bind creation to the audited round with `record-creation-epoch --round <round>` — **on a file-arm epoch additionally pass `--draft-file "<absolute issue-draft-<slug>.md path>"`** (the same canonical file `emit-body` posts from). On the file arm the epoch's attestation comparand is the digest of the bytes actually posted, so a legitimate override filing — where the current file was revised past the audited round — attests `match` against the faithfully-created issue instead of a false `mismatch`; omit `--draft-file` on embed/inline epochs, where no trustworthy canonical file exists and the audited round body stays the comparand.

   **Source the body from the single presentation source.** On a **file-arm epoch**, the body comes from the gated canonical file via `emit-body`, which emits the audited bytes only when eligibility grounds them. **Never pipe `emit-body` straight into `gh`** — without `pipefail` a refused `emit-body` (non-zero exit, **empty stdout**) would hand `gh` an empty body and create an **empty-bodied issue**. Emit the output to a temp file, **guard it non-empty**, and only then hand the guarded temp file to `gh issue create --body-file <path>` directly — never re-piped through `cat` into `--body-file -` (`cat` is a non-preflight PATH tool whose absence would feed `gh` empty stdin) — all in one single statement, byte-exact through the file (never a `"$(…)"` capture read by a later statement; see `references/issue-template.md` for the exact recipe). On **embed/inline-arm epochs** there is no trustworthy canonical file, so the in-context heredoc re-emission remains — a disclosed residual, not the preferred path. **Do not pass `--label` on `gh issue create`.** Report the issue URL that `gh` prints on success, and capture the new issue's number from the trailing path segment of that URL (e.g. `…/issues/42` → `42`) — sub-steps 5a and 6 need it. See `references/issue-template.md` for the exact invocation.

   After creation, attest what actually landed: fetch the created issue's body and pipe those bytes into `record-creation-attestation`, **byte-exact through a file, in one single statement** — the same discipline as the posting recipe (never a `"$(…)"` capture re-emitted via `printf`, which mutates trailing bytes against the recorded digest):
   ```bash
   if gh api repos/{owner}/{repo}/issues/<issue_number> --jq .body > "<main-root>/.devflow/tmp/issue-fetched-<slug>.md"; then python3 "${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../scripts/issue-audit-state.py record-creation-attestation "<slug>" --nonce "<nonce>" < "<main-root>/.devflow/tmp/issue-fetched-<slug>.md"; else python3 "${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../scripts/issue-audit-state.py record-creation-attestation "<slug>" --nonce "<nonce>" --attestation-unavailable; fi
   ```
   The tool tolerates exactly **one** fetch-framing trailing newline (jq's output framing) — reporting `matched modulo the fetch's single trailing newline` on stderr when that tolerance fired — and anything else stays a mismatch. **The failed-fetch arm is part of the recipe, not a manual afterthought — that is what the `if`/`else` shape is for.** A fetch can fail on network, auth, repo scope, or a deleted issue; were the `else` arm left to your own recall (or written as a bare `&&` chain that simply short-circuits), `record-creation-attestation` would never run and the summary would render `attestation=none` — which is defined above as *before any creation attempt*, i.e. indistinguishable from "never attempted". That silently defeats the detection surface for the embed/inline weaker-identity residual on exactly the input the policy exists to catch, so a failed fetch is reported unavailable, **never** as a pass and never as `none`. Branch on the **fetch's own** exit status (not a blanket `||` over the whole chain): the `attestation-unavailable` arm is scoped to a failed fetch, matching the tool's `fetch-failed` transition, so a *tool* failure after a successful fetch still surfaces its own breadcrumb instead of being re-run into an unavailable record. A recorded `match`/`mismatch` is forward-only tamper evidence either way — the tool refuses to overwrite one. **When that then-branch `record-creation-attestation` call itself exits non-zero** (a `git`/read/persist failure *after* a successful fetch, so nothing was recorded and `query-summary` still reports `attestation=none`), **surface it in the summary line as an errored post-fetch attestation** — e.g. append `attestation errored post-fetch: <breadcrumb>` — **not** the bare `attestation=none`, which means *no creation was attempted*: an attestation that was attempted post-creation and errored must be distinguishable from one that never ran (the same never-attempted misattribution the re-init `--force` attestation guard exists to prevent).
5a. **Stamp the reserved `DevFlow` provenance label (best-effort).** After a successful `gh issue create`, ensure the reserved label exists and apply it to the new issue. The label `DevFlow` is a hardcoded provenance constant (no config key controls it) — it is the branch-naming-independent signal the weekly retrospective uses to detect DevFlow-authored work. Apply it *after* creation (mirroring the `/devflow:implement` Phase 4 docs-label idiom) so a label hiccup can never block the issue from being created:
   ```bash
   "${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../scripts/ensure-label.sh DevFlow
   "${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../scripts/apply-labels.sh <issue_number> DevFlow
   ```
   Both steps are best-effort and need only the `repo` scope: `ensure-label.sh` and `apply-labels.sh` each always exit 0 (logging whether the label was created / already present / hit a `gh` error, and whether the apply landed). Application goes through `apply-labels.sh`'s REST `POST .../issues/{n}/labels` rather than `gh issue edit --add-label` (which resolves the repo via org-scoped GraphQL and fails under a repo-scoped token). The always-exit-0 contract applies only once a helper actually runs: if either invocation fails because the helper path does not exist (`No such file`, exit 127, or the platform equivalent), that is the **anchor-resolution** failure noted at the top of this skill (an empty `$CLAUDE_SKILL_DIR` with an unsubstituted placeholder), not a benign label outcome — fix the anchor and re-run both commands, since a silently skipped `DevFlow` label drops the issue out of the retrospective provenance detection. If the anchor cannot be resolved on this runner at all (neither `$CLAUDE_SKILL_DIR` nor a runner-reported base directory yields a usable path), report the degradation explicitly in the final outcome — issue created, `DevFlow` provenance label NOT applied — rather than proceeding silently. Continue to sub-step 6 regardless of the label outcome.
6. **Gated offer to start implementation.** This sub-step runs **only after `gh issue create` succeeds** (it exits zero and prints an issue URL) and the URL has been reported. If creation failed (non-zero exit, or no URL printed), stop here and surface `gh`'s error output verbatim — do **not** show this prompt. After a successful creation, evaluate a **two-condition gate** and present the yes/no implement prompt **only when both conditions hold** (an unconditional offer would post a comment that fires nothing on a tier-disabled repo, or green-light an implement run whose prerequisites are still open):

   - **(a) The created issue's body has no open prerequisites and no unresolved decisions** — the body you just drafted and posted contains **no `## Dependencies` entries** (the section is absent, or present but with no `Blocked by #N` line) **and no `## 🚫 Blocked` section**. Read this from that body.
   - **(b) The cloud implement tier is enabled** — `workflows.devflow` resolves to **exactly `true`** from `.devflow/config.json` (the same fail-closed semantics `.github/workflows/devflow-implement.yml`'s gate job reads as `.workflows.devflow // false`). Read it first through the bundled resolver as the command's **leading token** via the portable anchor:
     ```bash
     "${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../scripts/config-get.sh .workflows.devflow false
     ```
     When that invocation is **denied or fails** (the local-tier classifier can deny a script-by-path call; the anchor may be unresolvable), fall back to a direct read of the same key, **anchoring the config path to the git repo root the same way `config-get.sh` does** (the #295 SHARED REPO-ROOT CONFIG CONTRACT: `ROOT=$(git rev-parse --show-toplevel 2>/dev/null || pwd)`), so a run from a subdirectory reads the same root `.devflow/config.json` all three rungs read rather than a nonexistent cwd-relative one. **Check file existence first with a bash builtin `test`, so an absent config is detected as the *unconfigured* state — not by letting an interpreter crash on the missing path** (`json.load(open(...))` / `jq … file` both *error* on a missing file, so a raw one-liner would mislabel an absent config *unreadable*; the `[ -f ]` guard is what makes file-absent authoritatively unconfigured): `[ -f "$ROOT/.devflow/config.json" ] || { echo tier-unconfigured; }`. Only when the file **exists** do you read it with a preflight-guaranteed, classifier-permitted interpreter, using an extraction that **mirrors `config-get.sh`'s `.workflows.devflow // false` on both the value axis and the type axis** — the resolver (and the authoritative `.github/workflows/devflow-implement.yml` gate, which reads `.workflows.devflow // false` and string-compares to `"true"`) stringify the value, so a boolean `true` **and** a string `"true"` both read as `true`, while a wrong-type `workflows` container — **or a top-level non-object config** (a hand-corrupted `[]` or bare scalar) — reads as `false` (not a crash: an unguarded `d.get(...)`/`.workflows` on a non-object *errors*, and a crashed rung would misroute a merely-corrupted config to the *config unreadable* reason when `config-get.sh` itself reads the same file as `false` at exit 0). Use `python3 -c "import json,sys; d=json.load(open(sys.argv[1])); w=d.get('workflows') if isinstance(d,dict) else None; v=w['devflow'] if (isinstance(w,dict) and 'devflow' in w) else False; print(str(v).lower() if isinstance(v,bool) else str(v))" "$ROOT/.devflow/config.json"` or `jq -r 'if (type=="object") and ((.workflows|type)=="object") then (.workflows.devflow // false) else false end' "$ROOT/.devflow/config.json"`. (The mirror is exact for every decision-relevant shape — boolean, string, absent key, wrong-type container, top-level non-object; the corner values JSON `null` and empty string can print different bytes per rung — python prints `None` for `null` where `config-get.sh` applies the caller default — but no rung prints `true` on them, so the offer decision still agrees fail-closed.) **Lowercase only a JSON boolean** (so `true`→`true`) and emit any other value **verbatim** — exactly as `config-get.sh` does, and the compare is **case-sensitive** against the literal `true`: a string `"true"` reads enabled, but a capitalized `"True"`/`"TRUE"` reads **not** enabled (matching `config-get.sh` and the case-sensitive cloud gate), so a stray `.lower()` on the whole value must **not** be applied — it would fire an offer on `"True"` that the resolver and gate treat as disabled. Compare the read value to the literal `true` with **bash builtins only** (`case "$V" in true) … ;; esac`, or `[ "$V" = true ]`) — never `tr`/`sed` lowering, which fails **open** when the tool is absent (guard-class 2). Do **not** narrow the fallback to a boolean-only `== true` test either: `config-get.sh` and the cloud gate accept a string `"true"` as enabled, so a boolean-only fallback would silently *withhold* (and mislabel *tier disabled*) an offer the platform gate treats as enabled — a cross-rung divergence on exactly the local tier where the fallback is the only read path. The read is fail-closed: an absent key, an explicit `false`, a wrong-type shape, an absent file, and an unreadable/malformed config all resolve to something other than `true`, so the offer is withheld on each.

   **When both conditions hold**, present the prompt via the runner's user-question tool (`AskUserQuestion` on Claude Code, as used in Step 2) asking whether to start implementation now by commenting `/devflow:implement <issue_number>` on the newly created issue.
   - **On "yes":** post the comment yourself with `gh issue comment <issue_number> --body "/devflow:implement <issue_number>"`, using the issue number captured in sub-step 5. Use the bare `/devflow:implement <n>` form with **no** `@claude` mention, so the comment routes to the repo's `devflow-implement.yml` listener rather than the stock Anthropic listener. If `gh issue comment` succeeds, report that the comment was posted, and that it fires the implement run **only when** the repo's devflow workflows are enabled and the commenter is authorized — both enforced by the `gate` job in `.github/workflows/devflow-implement.yml` (authorization via `scripts/resolve-implement-trigger.sh`). Surface that caveat in the reported outcome. If `gh issue comment` fails, surface its error output and report that the issue was created but the implement comment could not be posted — do not claim the comment was posted.
   - **On "no":** finish without posting any comment.

   **When the gate withholds the prompt, print a one-line reason naming the exact failed condition** — one of: *declared dependencies present*, *blocked decisions present*, *cloud implement tier disabled or unconfigured*, or *config unreadable (all read paths failed)*. A withheld offer is **never silent**, and the issue-created outcome (the URL and the sub-step 5a `DevFlow`-label result) is still reported **in full**.

   **Selecting between *tier disabled/unconfigured* and *config unreadable* — observe the read, don't guess it.** A *value returned* and a *read that failed* are different signals, and only the second can mean "unreadable": inspect each rung's **exit status** (whether it returned a value at exit 0 vs. errored / was denied), not its stdout alone — an empty stdout from a crashed rung must never be read as the value `false`. Classify by what was actually observed:
   - **The config file is absent** (the `[ -f ]` guard above is false), **or** a rung read an existing file and returned a value at exit 0 that is not `true` (an absent key, an explicit `false`, a wrong-type `workflows` shape, a top-level non-object config — all read as `false` by both `config-get.sh` and the type-tolerant fallbacks) → *cloud implement tier disabled or unconfigured*. A merely-absent or unconfigured `.devflow/config.json` is **unconfigured on every rung**, never "unreadable" — the `[ -f ]` file-existence check, not an interpreter crash, is what establishes file-absence.
   - **The file exists but every attempted read path failed to parse it** — the resolver was denied/errored **and** the fallback interpreter errored on a genuine *parse or permission* failure (malformed JSON, or a file the process cannot read), i.e. no rung ever established a value from a file that is present → use the distinct *config unreadable* reason. Never conflate it with *cloud implement tier disabled or unconfigured*, which would assert a config state the read never established; and never conflate the reverse (an absent file, or one that read cleanly as `false`) into *config unreadable*, which would assert an unreadability the read never observed.

**This gate has no exceptions:**
- "The user already answered every clarifying question in Step 2" — answering decision forks is **not** the same as reviewing the assembled ticket. Show it and wait.
- "The user said 'just create it' / 'you decide' earlier" — that was said about a story they had **not yet seen written up**. It is not approval of *this* drafted issue. Render the draft and get explicit go-ahead on it.
- "The user disengaged, so I'll file it with a Blocked section and move on" — drafting from what's decided is correct (Step 2), but you still present the rendered draft and wait for confirmation before creating. If the user never returns to confirm, leave the pipeline paused at todo 6 — do **not** create the issue to "finish".
- "Showing the full body is verbose; a summary is enough" — no. Render the literal title and body that will be filed.

**Red flags — STOP, you are about to skip the gate:**
- You are constructing the `gh issue create` command in the same turn you drafted the body, with no user message approving it in between.
- You are about to create the issue because Step 2 was "complete" or the user "already decided everything".
- You are treating an earlier "just create it" as approval of a draft the user has not seen.

All of these mean: show the full rendered issue, ask, and wait for an explicit yes.
