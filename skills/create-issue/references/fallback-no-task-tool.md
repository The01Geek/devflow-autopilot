**Inline checklist fallback (when no task-tracking tool is usable).** When the runner exposes no task-tracking tool, or the exposed one is disabled or unusable, track the same seven items as a compact markdown checklist block rendered directly in your reply: render the seven items once, then re-render the whole block with updated status markers on each status change. The status markers are exactly three, complete by construction: `[ ]` pending, `[~]` in progress, `[x]` completed. The paused-at-todo-6 waiting state renders as item 6 marked in progress. Every rendered block ends with a one-line pointer to re-read the state file before creating the issue, so that reminder recurs with each re-render instead of living only in this turn-one text.

Because transcript prose is not harness-persisted state, mirror the checklist state to `.devflow/tmp/issue-checklist-<slug>.md`, rewritten on each status change, under this skill's existing artifact conventions (delete any leftover first, then write fresh; cwd/worktree-anchored like the Step 2 derivation artifact — reuse those conventions rather than restating them). **This fallback binds no slug of its own — Step 1 bound the run's kebab-case `<slug>` before it dispatched anything, and this write reuses it.** Read it from turn-one context when you still hold it, and otherwise from the fixed pointer `.devflow/tmp/issue-run-slug` Step 1 wrote; an absent or unreadable pointer is recorded **unestablished** and routes to the title-derived fallback `references/step-4-present-create.md` retains, never to a guessed stem. The first state-file write can only occur once the Prerequisites gate (Step 2) has produced a story; until then the block renders inline unbacked, harmless because no creation path is reachable before a story exists. Step 1's binding is scoped to the current invocation, so a second issue drafted later in the same conversation runs its own Step 1 and binds its own fresh slug. This checklist state file is process-progress metadata (the seven fixed items and their statuses, with no derivation or audit reasoning an auditor could re-anchor on) and is deliberately NOT added to the Step 3.6 out-of-bounds reasoning-artifact enumerations.

Re-read the state and re-render the block at four anchors: the skill's three existing per-turn gate re-checks on the fallback path (the Step 2 derivation re-check, the Step 3 drafting precondition, and the Step 4 presentation gate), plus a fourth path-agnostic anchor at Step 4 sub-step 5 — immediately before `gh issue create` runs, on every path, confirm the user's explicit approval of the rendered draft is present in the current conversation and that the active tracking mechanism shows item 6 in progress (read the task-tool state via its read-back call — e.g. `TaskList` — or, where the tool exposes no read-back, the conversation record of the last status update, on the tool path; read the state file on the fallback path). At each anchor, re-read the file and re-render the block whenever the current reply does not already show the current state. **Fail-closed arm (every anchor):** an absent, foreign-content, or unparseable state file is treated as pipeline-paused, never as the guard's silence — rebuild the block from the seven-item definition (marking only statuses you can confirm from the current conversation; item 6 in progress when the flow is at or past presentation), rewrite the state file, and proceed to creation only on explicit in-conversation approval. In a read-only sandbox where the file cannot be written, the fallback degrades to the re-post-the-block-in-the-current-turn rule the derivation artifact uses, and compaction survival is explicitly best-effort on that arm.
