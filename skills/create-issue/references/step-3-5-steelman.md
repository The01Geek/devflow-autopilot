### Step 3.5: Steelman the draft against the code (mandatory, before the user sees it)

*"Steelman this. Did we miss anything?"* — run that challenge against your own draft **every run**, immediately after the no-options gate passes and before Step 4 presents anything. The moment a draft passes the gate is when it looks most finished and is examined least; this step exists because a plausible-reading draft routinely carries claims the code contradicts, and the user should never be the one to catch them.

This is a **code-grounded verification loop, not a re-read**. Ambient context — project memory, the Step 1 findings, your recollection of the codebase — is where the wrong claims *came from*; checking the draft against the same context it was drafted from verifies nothing. Every check below runs against the **actual repository content, via tool calls made now** (targeted `Read`s and `Grep`s — never a full re-exploration; the same premise-verification discipline `references/issue-template.md` defines under *"Verify every load-bearing premise before drafting"*, re-applied to the assembled draft):

1. **Re-read the rendered draft end-to-end as an adversarial reviewer** whose job is to find what is wrong or missing, not to admire what is right.
2. **Verify each load-bearing claim, file reference, and acceptance criterion against the code.** Every named file/path exists as named (a `Glob`/`ls` per reference); every "the code currently does X" claim is confirmed by reading the code that does X; every AC is checked to be assertable against the code surface it names. Match the verification method to the claim class per the template's discipline — including the **relied-on-third-party-behavior** class: every behavior of an external platform, API, or service the draft **relies on** — load-bearing for the Desired Behavior, an acceptance criterion, **or** the Implementation Notes Approach, not only an AC's mechanism (webhook / event delivery, trigger syntax, token scopes, endpoint behavior, response shapes, rate limits) — is verified here too against the vendor's **official documentation via `WebFetch`**; when official docs are unreachable, via **`WebSearch`**; when search is unavailable or fails, by **asking the user to provide the documentation** (not memory), and the verified fact + source URL is recorded in the draft's `Technical Context`. An empty or inconclusive result is **unverified, never confirmed** — rewrite such a claim as an explicitly flagged assumption (the template's `— assumption, confirm before implementing` form). Terminal arm, two ways (mirroring the template): a relied-on behavior with **no** documentation and **no** in-repo example proving it becomes a `## 🚫 Blocked` vendor-behavior question; one a working in-repo example proves is written as a flagged `— assumption, confirm before implementing` line citing that example. An **incidental** third-party mention (not load-bearing) stays light and triggers no verification. **A mechanical claim is verified-or-obligation, never a bare prediction** (mirroring the template's Testing Strategy Move 3 rule): a Testing Strategy assertion of the form "running X reports Y" / "this must fail RED reporting Y" is written **either** as verified output you actually ran while drafting and are citing, **or** as an obligation on the implementer ("the pin must cover X") — never a prediction of the specific result you did not execute, which reads as a decided requirement and sends the implementer to encode a falsehood. `Relevant Classes/Files` references cite the symbol or section, not a `file:line` anchor, which rots. **Value-comparison ACs are checked for observed-output grounding (mirroring the template's Acceptance Criteria rule):** an AC (or Testing-Strategy assertion) comparing a produced value against a literal must state the comparison in the producing surface's observed-output terms, grounded by a probe that **exercises the distinguishing boundary fixture** (for a type-sensitive comparison, the type-boundary fixture — a JSON string `"true"` vs. a boolean `true`) or by a named implementer obligation stating the decided semantics or exact fixture-and-observed-output command. **Adjective-only ("explicit/exactly X") or probe-silent-on-the-axis comparison language is flagged non-conforming** and rewritten to observed-output terms — or, when the axis is a specification *choice* rather than a probeable fact, routed to Step 2's user-question machinery, not an obligation. **Universal-quantifier sweep (mandatory, same carve-out as the template's Acceptance Criteria universal-claim rule):** scan every universal quantifier the assembled draft asserts about the system under change — "never", "always", "each", "every", "all", "cannot" — outside `## 🚫 Blocked`, and confirm each is grounded one of the three decided ways (pinned per-arm/per-element, with an accepted-loss/suppression claim pinned by a fixture in which the suppressed input is present; scoped to the mechanism's supported form; or removed); the only exemptions are mandated-verbatim template boilerplate and rule text the change ships as artifact content, and an AC or Desired-Behavior sentence is never exempt however imperative its phrasing. A **detector/guard coverage claim** additionally requires a planted-defect positive-control obligation on the implementer. Rewrite or ground any ungrounded quantifier now, before the user sees the draft. **Occurrence counts and coupled-site lists are checked the same way (mirroring the template's occurrence-count premise class):** an in-repo occurrence count or mirror-site list is grounded by an executed whitespace-normalized search cited as a "Verified:" bullet with its command and hit list where the tier grants one, else by the specific evidence records consulted record-by-record — never a recall count. **A load-bearing claim additionally records the repository baseline it was grounded against** — per the template's claim-class enumeration, a measured-path digest for a location anchor and a full-domain search-result digest for a count or inventory, recorded with `record-claim-baseline` and cited as the bullet's `revision=`/`identity=`; where a claim already carries one, `check-claim-staleness` decides whether this pass re-derives it at all, so a `state=fresh` claim is verified by its recorded baseline rather than re-measured and every other verdict is re-derived. A count or inventory claim needs `--domain-stdin`; a non-zero exit or a missing line is an **unavailable read, not a verdict** — re-derive, never assume `fresh`. **A "code does X" premise is verified with its enclosing gates on the path to X (mirroring the template's conditional-path clause):** read the gates, conditionals, and defaults reaching X, and a claim that holds only under a non-default configuration states that precondition inside the claim.
3. **Hunt for what the draft omits**, specifically: **missed acceptance criteria** (behavior the Desired Behavior implies but no AC pins), **missed edge cases** (error paths, empty/absent inputs, concurrent or repeated events), **wrong assumptions** (claims step 2 could not confirm), **unstated scope** (surfaces the change touches that the draft never mentions — grep the identifiers the draft names for consumers it missed), **a missing input-is-data guard** (a new LLM/semantic-judgment surface over third-party text without the guard-AC-plus-hostile-case pair), **an undeclared enumerated AC list** (an enumerated test/case/example list inside an AC that declares neither a floor marker nor a closed-set exhaustiveness statement), and **an unreconciled multi-state contract** (when the draft states a multi-state contract — a set of outcome tokens, error/exit codes, a status enum, or a state machine, where each state has one or more trigger causes — in **more than one form**, e.g. a summary or table form *plus* the detailed per-state acceptance criteria, verify the forms agree: **no summary or table form lists fewer causes for a state than the detailed per-state ACs** collectively specify. This is a **within-text** comparison of two forms the draft already contains, so it is scoped only to drafts that actually state such a contract — a draft that states none draws no new hunt, question, or revision here, mirroring the "skipped entirely for non-UI issues" discipline. It makes **no** claim to catch a state that only a not-yet-written implementation will emit — that stays a review-time catch, out of reach at drafting time. A disagreement found here is fixed by revising the draft (step 5's revise-and-re-gate loop), not surfaced to the user). Also run the **AC mutual-consistency check (mandatory):** no acceptance criterion forbids a surface — a path, a file class, a tier — that another AC's discharge must touch; a pair that conflicts is an unresolved scope fork, not two independent criteria. And the **trust-boundary closure check (mirroring the template's Acceptance Criteria rule):** when an AC establishes a trust or integrity boundary over executable artifacts (scripts, hooks, anything sourced, exec'd, or imported), confirm the protected set is defined over the transitive source/exec/import closure of its entry points — a protected entry point that sources an unprotected sibling leaves the boundary open one hop deeper — or that the residual unprotected surface is stated explicitly.
4. **Sweep the draft's own unstated mechanism dependencies (mandatory).** Beyond the *stated* premises checked in step 2, enumerate the in-repo behaviors the draft's **designed mechanism relies on but never asserts as a claim** — a helper's exit code, a resolver's output shape, a gate's semantics — the exact class the claim-driven loop above cannot see, because nothing states it (issue #446's config-unreadable arm rested on `config-get.sh`'s malformed-input non-zero exit, unasserted, so nothing verified it). Resolve each with a **cited probe** (a "Verified:" bullet citing observed output) or an **implementer-obligation AC** — the obligation subject to the execution-tier constraint (a command already granted in `devflow_implement.allowed_tools`, or a code-reading obligation citing the producer code, never an ungranted-helper run). This extends the existing claim-driven discipline to unstated in-repo reliances and adds **no duplicate premise class**; it is the in-repo sibling of the relied-on-third-party-behavior class in step 2. **Extend the sweep to stated-but-unbound inputs (mandatory):** a mechanism input the draft describes by *role* — "the value the caller already tracks", "the branch diff it already has" — with no named referent is bound to a named variable, artifact, or file with a **cited code reference proving it exists at the consuming site**; when drafting cannot establish that binding, the draft carries an **implementer-obligation AC naming the binding as work**, never an unbound role description that expands to nothing at implement time. This is the stated-but-unbound sibling of the unstated-dependency sweep — the dependency sweep hunts reliances the draft never states, this clause binds inputs the draft names only by role.
5. **Revise the draft yourself and re-run the Step 3 no-options gate on the revision.** Then run **Revision-delta verification** (the shared procedure below). Fixing a false claim or adding a missed AC is your work, done now — do not present the flaws to the user as questions attached to an unrevised draft. If — and only if — the steelman surfaces a genuinely **new unresolved decision fork** (competing behaviors only the user can choose between), route it through the existing Step 2 machinery: ask via the runner's user-question tool, and on disengagement it lands in `## 🚫 Blocked` — no new decision-handling path.
6. **Self-check the draft against the Step 3.6 audit dimensions.** Before summarizing, run the same **dimension checklist the auditor will receive** — the Step 3.6 **generic dimension checklist** plus any consumer `## Audit dimensions` section, assembled by running `render-audit-prompt.py checklist` **orchestrator-side and consuming its stdout as a tool result** (applying the positional two-marker delivery check — `render-status:` first line, `render-end:` last line — and the fallback-to-Read-the-template-file rung when the markers are missing or out of position), for the same compaction rationale Step 3.6 states: turn-one extension text can be evicted, and a heading-less extension makes the loss invisible. **Read the delivered `render-status:` line, not a loader exit status — this site re-runs no loader.** A rendered **`unestablished`** status is surfaced per the Step 2 loader-failure arm's discipline — never laundered into the designed absent-heading no-op. A rendered **`absent`** status is instead the designed no-op: report that no consumer `## Audit dimensions` section was selected — the routine single-hook extension that carries only the other heading reads exactly this way — and then proceed treating the selection as no consumer section for that hook: a **report-then-proceed** step, never a stall, a user question, or a degraded-arm claim. Judge the assembled draft against each dimension, **fix what the check finds, and re-run the Step 3 no-options gate on any revision.** Then run **Revision-delta verification**. This is a drafter *self*-catch of checklist-class defects (a silent-output leak, a fails-open selection, a coupled doc site, an environment-variance flaw) before the subagent is dispatched — it reduces dependence on the auditor rather than adding audit rounds.
7. **Log the outcome either way.** Report a one-line summary in your next message — claims/file-refs/ACs checked (count), what was revised, or "steelman: N claims verified against the code, no revisions needed". **The summary additionally reports both new sweeps — the value-comparison type-semantics check (step 2) and the unstated-mechanism-dependency hunt (step 4).** When the hunt finds **no** dependencies, the summary states the falsifiable claim itself — "the mechanism invokes no in-repo helpers, resolvers, or gates" (checkable against the draft's Implementation Notes) — never a bare zero count. The summary **also reports the universal-quantifier sweep** — on a zero-findings run stating the falsifiable claim itself, "the draft carries no ungrounded universal quantifier", mirroring the unstated-dependency zero arm rather than a bare zero count. **The summary also reports the Step 3.6 dimension self-check (step 6) — its finding and revision counts, and on a zero-findings run the falsifiable claim "no dimension-checklist finding" rather than omitting the mention.** **The summary additionally reports the evidence bundle's coverage — `axes evidenced, axes marked N/A, axes unestablished` — as counts the reader can check against the artifact** (and names the effective list floor-only-degraded when the loader-failure arm fired). A silent pass is indistinguishable from a skipped step; the summary line is the evidence it ran.

Only a draft that has survived this step — revised and re-gated if anything was found — proceeds to Step 3.6.
