# DevFlow — project memory

DevFlow is a single [Claude Code](https://code.claude.com) plugin published at the repo root; the repo is also its own marketplace. It turns a GitHub issue into a reviewed, documented, merged PR, and a weekly retrospective loop improves the automation. Full system reference: [`docs/DEVFLOW_SYSTEM_OVERVIEW.md`](docs/DEVFLOW_SYSTEM_OVERVIEW.md). Contributor guide: [`CONTRIBUTING.md`](CONTRIBUTING.md).

## Commands

```bash
bash lib/preflight.sh      # verify git/gh/jq/python3.11+/PyYAML on PATH
bash lib/test/run.sh       # full suite: jq filters + shell helpers + python; gh-stubbed, no network/auth
git ls-files '*.sh' | grep -v '^lib/test/' | xargs -r shellcheck --severity=warning -e SC1091
git ls-files '*.py' | xargs -r ruff check
```

CI (`.github/workflows/ci.yml`) runs the same suite + lint on every PR. The **required** status check is the job name **`lib + python tests`** (not "CI", which is the workflow name and never resolves). CI checks out full history (`fetch-depth: 0`) so `origin/main` resolves and the `#434` stale-prose self-scan runs live there rather than self-skipping (issue #456).

**The suite reports three tallies — passed, failed, and *skipped* — and `0 failed` does NOT mean "everything ran" (issue #456).** A check can **self-skip** (it neither passed nor failed because the host could not run it or express its condition). With nothing skipped the summary is byte-identical to before — `N passed, M failed`; with `K` skips it reads `N passed, M failed, K skipped` followed by one line per skipped check (name, kind, reason). The exit code is unchanged (`[ "$FAIL" -eq 0 ]` — a skip never fails the suite). Each skip carries a **kind** — `blocking-gate` (a real gate that should have run here but could not — the `#434` self-scan arms) and `host-capability` (the host cannot express the condition — `#423 T6b`). Every self-skip routes through the single `skip <name> <kind> <reason>` helper in `lib/test/run.sh`; a meta-assertion turns the suite RED if a `printf '  NOTE ` emit appears outside that helper (it is the only sanctioned NOTE-emit; the scan is comment-aware and matches only the contiguous `printf` emit shape — an echo-based or variable-assembled emit is a disclosed non-goal). The renderer is `lib/test/summary.sh` (added to the CI shellcheck scope explicitly, since `lib/test/` is otherwise excluded). **A skipped check is never a clean pass:** when reading a suite run as verification (issue #405), a non-empty skip population must be surfaced, not laundered into `result: pass` — `/devflow:review-and-fix` records skips in `verification_evidence.skipped_checks` (distinct from `result: skipped`, which means the whole suite run was skipped) and a run whose `skipped_checks` is non-empty is not reported as a clean pass.

**Running the suite when the `bash <path>` wrapper above is denied — the tier matters.** The commands above are the *local* form. If the classifier denies the `bash <path>` wrapper (see the *LEADING token* gotcha and the classifier-fallback Convention below), do **not** silently fall back to "CI is the gate" — work the tiers in order:

1. **Local/interactive tier:** the suite scripts carry the executable bit + a `#!/usr/bin/env bash` shebang, so retry by invoking the path **directly as the leading token** — `lib/test/run.sh`, `lib/preflight.sh` (not `bash lib/test/run.sh`). The exec bit is necessary but not sufficient — the leading-token form must also be permitted; on this tier it typically is.
2. **Cloud `/devflow:implement` tier — in-env verification is the gate during the run; CI is the post-PR merge gate, not an in-run verification channel (issue #405).** This repo grants the suite/lint direct forms — `Bash(lib/test/run.sh:*)`, `Bash(lib/test/run-module.sh:*)`, `Bash(lib/preflight.sh:*)`, `Bash(shellcheck:*)` — in **both** `devflow_implement.allowed_tools` and `devflow.allowed_tools`, so a cloud implement run **runs the suite (or a focused module) in its own environment** (the direct leading-token form; the `bash <path>` wrapper stays deny-floored) and verifies every verification-command AC and the inline review pass's test evidence there. The run **never waits on, polls, re-checks, or cites CI for its own progress** — a verification command that is *not* granted goes **Blocked** naming `devflow_implement.allowed_tools` as the remedy (never a silent stall, never a verdict resting on a CI result the run never saw). CI (`lib + python tests`) remains the **required post-PR check that gates the human merge**; the run does not tick anything on it. Consumer repos follow the same rule: list your test/lint commands in `devflow_implement.allowed_tools` (and `devflow.allowed_tools` for the command path) and the run verifies in-env; leave them ungranted and the run goes Blocked naming the key. The read-only `review` tier (`devflow-runner.yml`) is a **separate** case — its wait-for-CI-then-review posture is the correct post-PR sequence and is untouched by #405.
3. **Auditable skip only for a genuine block:** only a *permission/sandbox denial* of the direct form (e.g. the runner's `mktemp` writes are blocked) — **never** a suite that *runs and fails* — routes a local run to the CI gate, and the skip must be recorded in the workpad so it is an auditable decision. A reflection that says "could not run the suite" without having tried the direct-leading-token form first (on a tier where it is available) is the bypass this rule exists to stop.

## Architecture

- `skills/` — one `SKILL.md` per command (`/devflow:implement`, `/devflow:review`, `/devflow:review-and-fix`, the `/devflow:docs` family, `/devflow:create-issue`, `/devflow:retrospective-weekly`, …), plus the two skills vendored from the `superpowers` plugin (`requesting-code-review` — the review engine's final-pass reviewer, dispatched as a `general-purpose` Task; `receiving-code-review` — the fix-loop principles) under their upstream MIT license (Jesse Vincent — see `LICENSES/superpowers-LICENSE`). Both are *runtime-dispatched* by DevFlow's engine; the `superpowers:writing-skills` authoring discipline is **not** vendored — it is a development-time tool, referenced as the external skill in the Conventions below. **Both vendored skills (`receiving-code-review`, `requesting-code-review`) are installed into third-party consumer repos — their SKILL.md bodies must be repo-agnostic: no `lib/test/run.sh`, no `lib + python tests`, no `CLAUDE.md tiered-runner convention`, no DevFlow-internal step numbers. Use generic language ("the project's test suite", "the fix step") instead.**
- `agents/` — first-party subagents: the review-engine trio `checklist-generator`, `checklist-deduper`, `checklist-verifier`, plus `code-explorer` and `code-architect` (the `/devflow:implement` discovery/planning agents, vendored from Anthropic's feature-dev plugin under Apache-2.0 — see `LICENSES/feature-dev-LICENSE`).
- `scripts/` — Python + shell CLIs (`workpad.py`, `config-get.sh`, `match-deferrals.py`, `file-deferrals.py`, `parse-acs.py`, `resolve-*-trigger.sh`, …).
- `lib/` — retrospective-loop helpers (`*.sh`, `*.jq`), `preflight.sh`, `test/`.
- `.github/` — optional cloud tier: workflows + composite actions (incl. `vendor-plugin`).
- `.devflow/` — `config.example.json` + `config.schema.json` (+ tracked `learnings/`, `logs/`). The live `config.json` is **tracked** here too — force-added past the `/.devflow/*` ignore rule (`git add -f`) so the cloud tier reads it from the committed tree.

## Gotchas (load-bearing — easy to get wrong)

- **Skills locate bundled helpers via the portable single-statement anchor `"${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../<dir>/…`, NEVER `${CLAUDE_PLUGIN_ROOT}`** (not substituted inside `SKILL.md` bodies). The cloud-tier **workflows** are the one exception: they use the literal vendored path `.devflow/vendor/devflow/scripts/…`. **Why this exact shape (issues #241 + #275, generalized to all 22 skill/phase files):** `${CLAUDE_SKILL_DIR}` is *empty* (not merely unset) on non-Claude-Code runners (confirmed on Copilot CLI; expected on Cursor, Codex CLI, Gemini CLI) — the runner instead reports a `Base directory for this skill:` context line the agent substitutes for the placeholder — AND Copilot CLI's inline `bash -c` marshaling **strips a variable assigned in one statement and read in a later statement of the same inline command** (`bash -c 'v=hi && echo $v'` → empty), so the old assign-then-reuse recipe (`SKILL_DIR="…"; … "$SKILL_DIR"/../…`) is banned: resolve the anchor **inline in the statement that uses it**, every call site (a reintroduced bare or cross-statement form turns `lib/test/run.sh`'s #275 pin loop RED). Windows-form runner-reported paths are normalized at **prompt time** (one standalone `wslpath -u`/`cygpath -u` probe or `lib/normalize-path.sh`'s textual rules; the skills carry **no** shell normalization mirrors anymore — create-issue's preamble is the reference prose paraphrase, kept in lockstep with the lib helper). On Claude Code the resolved value equals the bare form, so behavior is unchanged there. Relatedly (#275): `workpad.py` reads `.devflow.workpad_marker` from `.devflow/config.json` directly in Python — never by exec-ing `config-get.sh`, which Windows can't exec ([WinError 193]). The cloud-tier vendored-path carve-out is untouched. **Repo-root anchoring (#295, supersedes the #275 cwd-relative contract):** the five `.devflow/` readers — `config-get.sh`, `workpad.py` (marker read), `load-prompt-extension.sh`, `match-deferrals.py` (`_default_config_path`), `match-lint-adjudications.py` (`_default_config_path`) — resolve the *default* `.devflow/` path anchored to the git repo **root** (`git rev-parse --show-toplevel`, falling back to `pwd`/`Path.cwd()`), mirroring `lib/config-source.sh`, so a skill run from a **subdirectory** still loads the consumer's root config/prompt-extension instead of silently missing it. A **non-empty** explicit `CONFIG_FILE`/`--config` arg is still honored verbatim (an explicit empty value still selects the root-anchored default; root anchoring applies only to the default); a no-git-root-**and**-no-`.devflow/` tree emits a stderr breadcrumb instead of a silent default. The Python readers resolve the root via a **native `git` subprocess** (Windows-safe, like the `gh` calls — not a `.sh` exec). Keep the three contract mirror sites (`config-get.sh` + `workpad.py` + `match-lint-adjudications.py` headers, the `SHARED REPO-ROOT CONFIG CONTRACT` phrase) and the `lib/test/run.sh` pins in lockstep. **Known limitation:** `--show-toplevel` returns the *nearest* git root, so a nested submodule/inner repo or a monorepo whose `.devflow/` is not at the git root is **not** covered.
- **Config is `.devflow/config.json` (JSON), read via python3 by a single resolver** (`config-get.sh` / `config-source.sh`) — `node` is NOT required anywhere in the config-read path (issue #220; python3 is a hard preflight prerequisite). PyYAML is required only because `match-deferrals.py` (deferred-findings blocks in PR bodies) and `consolidate-changesets.py` (changeset frontmatter) lazily import it (config itself is JSON — no PyYAML needed to read it; `workpad.py` is stdlib-only and needs no PyYAML). Renaming a config key? grep `.py`, `.sh`, `.yml`, and jq lines — silent-fail consumers exist (`match-deferrals.py`, `workpad.py`).
- **The review engine is shared, and it is a BUNDLE — a thin root plus gated phase references (#529).** `/devflow:review` Phases 0–4.3 are executed *verbatim* by `/devflow:review-and-fix` (which adds the fix loop + shadow pass and skips Phase 4.4). Edit `skills/review/SKILL.md`; never paraphrase the engine into the fix-loop skill. The root keeps only shared state (the live-progress-comment machinery, the overrides block), the cross-phase invariants, the extension load, routing, and fail-closed entry; **each phase's authoritative procedure lives in `skills/review/phases/*.md`** and is reached through the root's entry-gate (re-derive bundle identity → `Read` the reference → clear its boundary contract, every entry, including a `/devflow:review-and-fix` Step 2.6 **shadow** entry). Three references are gated *out* of the default path by their existing predicates — `phase-0-3-6-blocker-recheck.md` (standalone PR mode **and** only over a prior REJECT driven solely by carve-out blockers; on a hit it *replaces* Phases 1–3), `phase-0-6-stale-prose-lint.md` (`devflow_review.stale_prose.enabled`), `phase-4-1-7-stale-adjudication.md` (PR mode, STALE adjudication) — which is what buys the AC3 budget; `phase-4-4-github-post.md` is standalone-only. **AC3's gate measures the SHIPPED-DEFAULT path, re-anchored by #618:** `stale_prose.enabled` defaults **true**, so an ordinary pass *does* read the stale-lint reference — the gate's comparand is therefore `_rb_shipped_w` (root + shipped extension + the six default stems + the stale-lint reference), not the hypothetical no-stale-prose default set. **The reviewed surface is every source, not the root:** `extract-command-heads.py`/`extract-command-shapes.py` take the whole bundle in one call, and `lib/test/run.sh` asserts engine-content pins against a concatenated `$REVIEW_BUNDLE` (so a sentence may move between references without breaking its pin), reserving `$REVIEW_ROOT` for pins that are genuinely about the root. **Budgets are a contract with thin headroom:** root+extension ≤ 8,500 words, and the re-anchored AC3 gate holds the **shipped-default per-pass path ≤ 32,399 words** (ceiling phrase pinned by `lib/test/run.sh` — an in-situ marker for the `revise-claude-md`/`claude-md-improver` tooling, which must not reword this phrase). That ceiling is an **interim bridge** (#618 Arm B: the measured shipped-default figure plus a fixed 60-word margin; a follow-up prose-reduction issue lowers it to ≤ 30,100), and a `/devflow:review-and-fix` fix loop **may self-apply the escape-valve procedure** on a future breach — re-measure with `_rb_words`, set measured-plus-60, reconcile the four mirrors (`lib/test/run.sh`, `docs/review-bundle-budget.md`, `.devflow/prompt-extensions/review-and-fix.md`'s own ceiling phrase, and this bullet), record it in the doc's decision record — **including making this bullet's ceiling-phrase mirror edit directly** (a this-repo-scoped sibling of the #366 carve-out, so the ceiling pin never traps the fix loop it is remedying); ordinary re-measures never touch this bullet. **This bullet carries NO measured word figure** — every live measured figure and the full decision record live in [`docs/review-bundle-budget.md`](docs/review-bundle-budget.md), so an ordinary re-measure never edits `CLAUDE.md`; only a ceiling renegotiation does. **Words are counted by `python3` (`lib/test/run.sh`'s `_rb_words`), NEVER by `wc -w`** — `wc -w` measures the bundle *plus the host* and no locale fixes it, because GNU and BSD `wc` disagree in two directions: under `LC_ALL=C` GNU drops a standalone non-ASCII token (every spaced ` — `, ` → `) entirely, and under a UTF-8 locale BSD splits `rc≠0` on the `≠`. Every pre-2026-07-17 figure in the record was a macOS/BSD reading that reproduced at one desk and nowhere else, until the AC4 reconciliation went red on Linux CI against a record that was right about the bundle and wrong about the host (PR #540). This is the `wc`-is-not-preflight-guaranteed rule below biting a *measurement*, not a selection. Re-measure with `_rb_words` before adding prose to the root or any shipped-default member; the record — with every live measured figure — is [`docs/review-bundle-budget.md`](docs/review-bundle-budget.md).
- **`skills/review/SKILL.md` executes under TWO allowlists, and a head neither grants is SILENTLY denied (issue #363).** The auto-review path uses the `review` profile's `TOOLS='…'` line in `.github/workflows/devflow-runner.yml`; the manual `/devflow:review` comment path uses `devflow.yml`'s hoisted `TOOLS='…'` (`Resolve allowed-tools` step, consumed by `claude_args` and by the injected block alike, so they cannot drift). A command the skill invokes but a profile omits is refused before it runs — it does not fail loudly, it burns budget, and a run can end with **no verdict at all** (PR #340: 7 of 14 denials were the engine trying to run the test suite). `lib/test/extract-command-heads.py` (driven by `lib/test/run.sh`) extracts every head from the skill's ```bash fences — quote/comment/heredoc-aware, `$(…)`-descending, wrapper-stripping, case-arm-position-aware (issue #392: arm patterns are stripped only where an arm may legally begin — after `case … in` and after each `;;` — so a command in a case *body*, e.g. a bare subshell `(cmd)`, keeps its head instead of being swallowed as a bogus arm) — and asserts that **each** allowlist grants it. **Scope boundary:** inline-backtick prose is deliberately out of reach (matching it resurrects the `git a`/`git failure`/`git said` false positives), so a prose-only command like Phase 0.3.6's `git cat-file` is pinned by direct literal instead. A second, narrower boundary (#392): the case-arm tracking is a flag, not a depth counter, so a **nested** `case` block is an accepted limitation — no fence in `skills/review/SKILL.md` nests a `case`. Adding a command to a fence? Grant it by adding the token to `lib/capability-profiles.json` (the versioned capability manifest — the single source of truth for all five runner/probe allowlist literals) and regenerating with `python3 lib/generate-capability-profiles.py`; the `--check` mode wired into `lib/test/run.sh` turns any manifest↔literal drift RED before merge, so you never hand-edit the `TOOLS='…'` literals (see the capability-manifest gotcha below). **Heads are not enough — the matcher denies composite SHAPES whose every head is granted (issue #401, evidence run 29105381021: 22 denials, no verdict):** leading `VAR=value` assignments, leading `cd`, `>`/`2>` redirects targeting `/tmp`, `cat`-heredoc writes, interpreter heads (`python3`), and the unexpanded `"${CLAUDE_SKILL_DIR:-…}"` anchor as the leading token are all refused in the cloud review runner. Probe-proven permitted shapes: the Write tool into `.devflow/tmp/**` (granted in the review profile), `… | tee`, `tee <<'EOF'`, and repo-relative vendored-literal helper paths; the skill's "Cloud command-shape discipline" section and the grounding block's command-shapes section carry the recipe (two denials of a shape → switch to a permitted alternative, never iterate variants). `lib/test/extract-command-shapes.py` (driven by `lib/test/run.sh`) turns a denied-shape fence RED at the desk. `.github/workflows/matcher-probe.yml` (repo-internal; NOT shipped by `install.sh`) is the re-runnable evidence source — re-run it after a `claude-code-action` upgrade before changing shape rules or grants. `Bash(cd:*)` is deliberately ungranted (its probe row was redirect-confounded — unproven, pinned absent in `run.sh`); do not re-add it without a fresh redirect-free probe row.
- **The five runner/probe allowlist literals are GENERATED from one versioned manifest — never hand-edit them (issue #561).** `lib/capability-profiles.json` (integer `manifest_version`, named token `groups`, a `readme`, and exactly the `review`/`implement`/`command` profiles composing group refs + inline tokens into flat ordered lists) is the single source of truth. `python3 lib/generate-capability-profiles.py` compiles it into exactly five regions — `devflow-runner.yml`'s review `TOOLS='…'`, `devflow.yml`'s command `TOOLS='…'`, `devflow-implement.yml`'s `--allowed-tools` base list (up to the `${{ needs.config.outputs.allowed_tools_extra }}"` splice, preserved verbatim), and `matcher-probe.yml`'s `REVIEW='…'` and `IMPLEMENT='…'` baselines — each carrying a banner comment with `manifest_version` + the sha256 of that region's resolved token list (the banner is placed where it is syntactically inert and never contains the byte sequence `TOOLS='`). `--check` (wired into `lib/test/run.sh`, so the required `lib + python tests` CI job gates it) byte-compares every region and turns any drift RED with a token-level **directional** diff — a hand-added workflow token is named as workflow-side with "add it to the manifest and regenerate" (blind regeneration would silently revert the grant). **The review profile is a security boundary:** the generated review literal IS the read-only reviewer allowlist (the deny floor filters only appended consumer extras, never the base), so `lib/review-profile.tokens` locks its exact resolved token list — the generator NEVER writes it, and any manifest edit (including to a shared group) that changes the resolved review list fails closed until you update that lock in the same PR, so widening the reviewer always needs a visible diff. The generator is python3 stdlib-only (no `yaml`), reads no git history, and has no runtime footprint (a `run.sh` assertion greps the six workflows for zero non-comment references to it — desk/CI-time only, mirroring `extract-command-heads.py`). **Bump rule:** increment `manifest_version` exactly once in any PR that changes the manifest (review convention, not machine-enforced; the per-region checksums are the machine truth). Empirical territory stays hand-maintained: the probe's candidate rows, verdict tables, command-shape verdicts, and the `EXTRAS` config-mirror row are NOT generated — the manifest states policy, never measurement. Adding a grant = edit the manifest (+ the lock if it widens review) → regenerate → the same `--check` gate covers what the retired #450 token-sync pin used to, plus the review-tier equality that never had a pin. **Implement-tier bundled-helper grant flow (issue #555):** a bundled helper that a `/devflow:implement` fence invokes — the §4.0.5-class case, e.g. `scripts/discover-deferral-manifests.py` — is granted by adding its vendored-literal token `Bash(.devflow/vendor/devflow/scripts/<helper>:*)` (the row-I2-proven explicit leading-token form) to the `implement` profile in `lib/capability-profiles.json` and regenerating; that one edit rewrites `devflow-implement.yml`'s baked `--allowed-tools` baseline **and** `matcher-probe.yml`'s `IMPLEMENT` baseline in lockstep, so the probe's baseline can never drift from the tier it is probing, and the generator's `--check` (driven by `lib/test/modules/capability-profiles.sh`) enforces it. **Never hand-edit either workflow literal** to add such a grant. An implement-only grant leaves the review security boundary untouched, so `lib/review-profile.tokens` and the review-region checksums stay byte-identical and only `manifest_version` moves.
- **Implement-profile head guard + inline-engine surface (issue #484).** Phase 3 runs the review engine **inline** under `devflow-implement.yml`'s baked `--allowed-tools` (not the review profile), so every helper the normal inline flow can reach needs an implement-profile grant — the review engine is shared. `lib/test/run.sh`'s #484 head guard deliberately over-approximates that runtime surface: it drives `extract-command-heads.py` (an `implement-block` parse mode reading ONLY that baked block — never the whole file or `.devflow/config.json`, so a `Bash(...)` cited in a YAML comment is not a grant; fail-closed on an absent/malformed block) over all fenced source in `skills/implement/**`, `skills/review*/**`, and the dispatched `skills/requesting-code-review/**` final pass, including standalone-only review Phase 4.4. It fails when an audited fenced head is neither granted nor in the exact withheld list (`gh pr checkout`, `git rev-list`, `mktemp`); a separate suppression list covers shell builtins + parse artifacts, and a removal-proof contract requires inline `workpad.py` shorthand to expand to the portable granted helper path before emission.
- **Implement-tier coverage (issue #455 → matcher-probe.yml's `implement-probe` job).** The read-write `devflow-implement` profile is a **separate** allowlist with its **own** probed denied shapes — a shape proven on the review tier is unproven here — so the two are covered independently: the implement-probe table (evidence of record on issues #450/#455) proved the explicit vendored-literal grant **form** PERMITTED (row I2, measured on `apply-labels.sh`), the config `*/basename` glob DENIED against a vendored-literal leading token (row I3), and the unexpanded anchor leading token (I1) plus the `for` / piped-`while read` / `VAR="$(…)"` wrappers around a label helper (I4/I5/I6) DENIED. **Grants are per-HEAD across the whole pipeline, not just the leading token** (a repo rule from #363/#401, *not* an implement-probe row): one ungranted head in a tail refuses the entire statement and it produces **no output**. `paste` is granted nowhere — an in-PR draft of the reworked fences ended the label **normalizer** in `| paste -sd, -`, which would have refused that normalizer statement outright — leaving the resolved labels non-empty but the normalized list empty, so the applies silently did nothing (caught at the desk). Use the granted `tr`/`sed`/`grep`. So `devflow-implement.yml`'s baked literal grants `apply-labels.sh`/`ensure-label.sh` explicitly, all **four** label call sites — Phase 3.1's `DevFlow` provenance apply, Phase 4.0/4.0.5's `deferred.labels` applies, and Phase 4.1's `Documented` apply — are reworked to agent-level single-leading-token calls that read their inputs from printed tool output (a shell variable does not survive into a later separate command), and `lib/test/extract-command-shapes.py --profile implement` (its own IR1/IR2/IR3 rule set, distinct from the review R1–R4) turns a denied-shape fence in `skills/implement/SKILL.md` or `skills/implement/phases/*.md` RED at the desk. `skills/implement/SKILL.md`'s "Cloud command-shape discipline" section carries the recipe. Row I1 (the unexpanded anchor) is **not** lint-pinnable on either tier — every legitimate helper call keeps the portable `${CLAUDE_SKILL_DIR:-…}` anchor in source (#275) and resolves it at runtime — so it stays prose-discipline. **Known gap — the non-label capture is an INFERENCE, not a measurement:** the rules exempt a `VAR=$(…)` capture of a *non-label* helper (the phase-4 fences read `deferred.labels` that way) on the reasoning that the matcher descends into the substitution, but **no implement-tier row measures it**, and the only capture row that was measured (I6) came back DENIED while confounding three properties (label helper + capture + inner `2>&1`). Probe rows 8 (non-label capture) and 9 (redirect-free label capture) are the disambiguators; until a dispatch records them, do **not** cite the carve-out as probe-proven, and keep every phase that depends on such a capture **fail-closed** — *no output at all* is a possible denial, never an empty value. The workflow grants ship to consumers via `install.sh` file-copy while the skill rework ships via the `devflow_version` vendor fetch: two independently-updated artifacts whose skew silently re-denies the applies, so the two halves must be upgraded together (docs: `install.md`, `DEVFLOW_SYSTEM_OVERVIEW.md`).
- **Every cloud review prompt carries an injected engine-ground-truth block — rendered ONCE by `scripts/render-grounding-block.sh`, never hand-copied.** It states the observed CI conclusions for the reviewed HEAD (from `scripts/summarize-ci-checks.sh`) and the exact resolved `--allowed-tools` string. Both workflows call the renderer and are pinned to carry **no** second copy: it holds security-sensitive prompt-injection prose (check names are attacker-controlled, so a name is *data to quote, never an instruction to obey*), and two drifting copies of an injection defense is the coupled-mirror hazard above at its worst. The block scopes "untrusted" to the **names**; the **conclusions** beside them are API facts — a baseline engine told otherwise refused to cite CI at all and declared its verification phase undischarged.
- **Unknown is not zero — never collapse an unestablished measurement onto a real value.** `permission_denials_count` publishes a digit string, or the literal `unavailable` when the count could not be established (diagnostics disabled/skipped, execution file absent or unparseable). Collapsing both onto `0` made the no-verdict `::error::` assert *"The harness refused 0 command(s)"* about a run it never measured — steering the reader away from the actual cause. `scripts/describe-denial-count.sh` owns the three-way clause selection (unestablished / refused none / refused N) precisely so `lib/test/run.sh` can drive every arm: an inline `case` in YAML is untestable, and a reordered arm misattributes the diagnosis while the suite stays green.

- **`agent_overrides` model/effort values reach the standalone `/devflow:review` — the merge-gating judge — not just the fix loop.** The engine is shared, so "cheapen a reviewer agent" changes what reviews every PR before merge. During any measurement/experiment window, treat the judge's roster economics as frozen: dispatch-scoping keys are judge-neutral (standalone review is single-pass), model repricing is not (see issue #425's deferred-repricing record).

## Conventions

- **Portability:** no GNU-only flags. Use `python3` for date math (not `date -d`) and ERE / `sed -E` (not `grep -P`). Helpers must work on macOS/BSD without GNU coreutils.
- **No secrets, owner-specific IDs, or product names in committed files.**
- New `.py`/`.sh` files carry the SPDX header (`SPDX-FileCopyrightText: 2026 Daniel Radman` / `SPDX-License-Identifier: MIT`).
- Docs reference bare source paths (`scripts/workpad.py`), never `path:line` — line numbers rot.
- **Updating `CLAUDE.md`? Use `/claude-md-management:revise-claude-md` (targeted additions) or `/claude-md-management:claude-md-improver` (full audit) — never hand-edit without running at least one.**
- **Autonomous `/devflow:implement` runs make a *required* `CLAUDE.md` edit directly — carve-out to the rule above (issue #366).** Any `CLAUDE.md` edit an autonomous DevFlow run is *required* to make — whether by a Phase-3 review finding **or by the issue's own acceptance criteria** — is made **directly by the orchestrator**, citing this carve-out and recording it in the workpad, **never** by invoking `revise-claude-md`/`claude-md-improver` mid-run (a nested interactive skill runs as a tail call and stalls the run — the defect issue #366 fixes). Interactive/human sessions still use `revise-claude-md`/`claude-md-improver` per the bullet above. Other interactive skills a run is required to invoke are instead dispatched into an **Agent-tool subagent** (issue #362) — see `skills/implement/SKILL.md`. This bullet is the `CLAUDE.md` half of a coupled pair — its mirror is the `CLAUDE.md` edit carve-out in `skills/implement/SKILL.md`; edit both together.
- **Editing any skill file (`SKILL.md`)? ALWAYS invoke the `writing-skills` skill first** (`superpowers:writing-skills`) and follow its RED/GREEN discipline — never hand-edit a `SKILL.md` body directly. This applies to every `skills/*/SKILL.md` (the shared review engine, the docs family, retrospective, etc.); the skill enforces the verify-before-deploy loop that keeps a skill's triggering + behavior intact. **Autonomous `/devflow:implement` runs satisfy this mandate differently:** they never invoke `writing-skills` through the Skill tool mid-phase (a tail call that would stall the run), so the implement prompt extension (`.devflow/prompt-extensions/implement.md`'s "Prompt-surface edit routing" rule) routes each prompt-surface edit through a context-isolated Agent-tool subagent that invokes `superpowers:writing-skills`, recording a `Writing-skills evidence:` marker in the workpad (with the degraded inline mode noted when the skill is unavailable). Interactive/human sessions keep invoking the skill directly **and record a `Writing-skills evidence:` line in the PR description** (the review gate's discharge surface for human PRs). This bullet and that extension routing rule are a coupled pair — edit both together.
- **Helper cutover deletes the superseded prose — same change.** When an executable helper becomes the sole tested owner of a workflow decision on every path that previously consumed the prose, that same change removes the superseded mandatory prose, the duplicated branch/enum mirrors, and the obsolete prose pins — re-anchoring each removed pin's guarded regression to a helper-behavior test, with retirements recorded in the PR's cutover artifact. Policy, human decision points, invocation contracts (including fail-closed handling of the helper's absence), and essential stop conditions stay in the skill; rare-path explanation moves to progressively loaded references — operative decision logic never does. Removing decision-owning mandatory prose whose decision has no sole tested owner is unauthorized, and growing the mandatory surface is an audited decision recorded in a growth artifact. Full procedure: `.devflow/prompt-extensions/implement.md` "Prose cutover"; the mandatory-bytes census is `lib/test/prompt-mass-census.py`.
- **Marketing/positioning copy** (`README.md`, `docs/DEVFLOW_SYSTEM_OVERVIEW.md`) leads with the wedge — *out-of-the-box agentic coding works on pet projects but can't complete a real ticket in a large production codebase; DevFlow makes it ship*. Honor the honest-claims guardrails (review-ready, **not** auto-merged; shadow review **narrows, never closes** the gap). §21 is the messaging source of truth.
- Don't push to `main` from an agent; the classifier blocks `git push origin main`. Inside `/devflow:implement` & `/devflow:review-and-fix` there is standing permission to auto-commit+push to the *feature* branch.
- **Local-tier classifier blocks shell-script invocation — this is expected; use the documented fallbacks, don't fight it.** On the local/interactive tier the permission classifier routinely denies `bash <path>`, `shellcheck`, `ruff`, and helper scripts invoked by path (the recurring friction behind retrospective pattern `convention-violation`, PRs #72/#85/#87/#98). It is a **structural harness boundary**: an agent is barred from writing `.claude/` to widen its own allowlist (`docs/efficiency-trace.md` "Note on this repo's `.claude/settings.json`"), so the run cannot self-grant the missing permission — the fix is operator-side, not in-run. When blocked, fall back **without re-deriving it each run**: run helpers via `python3 <path>` (the classifier permits Python by path) and `jq`; rely on the required CI job `lib + python tests` to run `bash lib/test/run.sh` + lint on push (verify it is green before merge); for the workpad use `gh api` PATCH if `workpad.py` is denied. To actually widen the allowlist, a **human** adds the rules at user scope (`~/.claude/settings.json`) or runs the consent-gated `scripts/provision-auto-mode.sh` / `scripts/provision-local-settings.sh` — never commit a `permissions.allow` block to this repo's `.claude/settings.json` from an agent (the classifier denies it, and the cloud tier discards `.claude/` anyway).
