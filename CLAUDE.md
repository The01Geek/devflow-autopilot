# DevFlow — project memory

DevFlow is a single [Claude Code](https://code.claude.com) plugin published at the repo root; the repo is also its own marketplace. It turns a GitHub issue into a reviewed, documented, merged PR, and a weekly retrospective loop improves the automation. Full system reference: [`docs/DEVFLOW_SYSTEM_OVERVIEW.md`](docs/DEVFLOW_SYSTEM_OVERVIEW.md). Contributor guide: [`CONTRIBUTING.md`](CONTRIBUTING.md).

## Commands

```bash
bash lib/preflight.sh      # verify git/gh/jq/python3.11+/PyYAML on PATH
bash lib/test/run.sh       # full suite: jq filters + shell helpers + python; gh-stubbed, no network/auth
git ls-files '*.sh' | grep -v '^lib/test/' | xargs -r shellcheck --severity=warning -e SC1091
git ls-files '*.py' | xargs -r ruff check
```

CI (`.github/workflows/ci.yml`) runs the same suite + lint on every PR. The **required** status check is the job name **`lib + python tests`** (not "CI", which is the workflow name and never resolves).

**Running the suite when the `bash <path>` wrapper above is denied — the tier matters.** The commands above are the *local* form. If the classifier denies the `bash <path>` wrapper (see the *LEADING token* gotcha and the classifier-fallback Convention below), do **not** silently fall back to "CI is the gate" — work the tiers in order:

1. **Local/interactive tier:** the suite scripts carry the executable bit + a `#!/usr/bin/env bash` shebang, so retry by invoking the path **directly as the leading token** — `lib/test/run.sh`, `lib/preflight.sh` (not `bash lib/test/run.sh`). The exec bit is necessary but not sufficient — the leading-token form must also be permitted; on this tier it typically is.
2. **Cloud `/devflow:implement` and read-only `review` tiers:** the suite is **not allowlisted by default** (and the `bash <path>` wrapper is deny-floored, so it can *never* be granted), so there the **`lib + python tests`** CI job on push is the **expected** gate — not a fallback or a bypass. A maintainer can add `lib/test/run.sh` / `lib/preflight.sh` to `devflow_implement.allowed_tools` (and `devflow_runner.allowed_tools` if a reviewer must build) to make the direct form resolve in cloud, but absent that, CI is correct and sufficient.
3. **Auditable skip only for a genuine block:** only a *permission/sandbox denial* of the direct form (e.g. the runner's `mktemp` writes are blocked) — **never** a suite that *runs and fails* — routes a local run to the CI gate, and the skip must be recorded in the workpad so it is an auditable decision. A reflection that says "could not run the suite" without having tried the direct-leading-token form first (on a tier where it is available) is the bypass this rule exists to stop.

## Architecture

- `skills/` — one `SKILL.md` per command (`/devflow:implement`, `/devflow:review`, `/devflow:review-and-fix`, the `/devflow:docs` family, `/devflow:create-issue`, `/devflow:retrospective-weekly`, …), plus the two skills vendored from the `superpowers` plugin (`requesting-code-review` — the review engine's final-pass reviewer, dispatched as a `general-purpose` Task; `receiving-code-review` — the fix-loop principles) under their upstream MIT license (Jesse Vincent — see `LICENSES/superpowers-LICENSE`). Both are *runtime-dispatched* by DevFlow's engine; the `superpowers:writing-skills` authoring discipline is **not** vendored — it is a development-time tool, referenced as the external skill in the Conventions below. **Both vendored skills (`receiving-code-review`, `requesting-code-review`) are installed into third-party consumer repos — their SKILL.md bodies must be repo-agnostic: no `lib/test/run.sh`, no `lib + python tests`, no `CLAUDE.md tiered-runner convention`, no DevFlow-internal step numbers. Use generic language ("the project's test suite", "the fix step") instead.**
- `agents/` — first-party subagents: the review-engine trio `checklist-generator`, `checklist-deduper`, `checklist-verifier`, plus `code-explorer` and `code-architect` (the `/devflow:implement` discovery/planning agents, vendored from Anthropic's feature-dev plugin under Apache-2.0 — see `LICENSES/feature-dev-LICENSE`).
- `scripts/` — Python + shell CLIs (`workpad.py`, `config-get.sh`, `match-deferrals.py`, `file-deferrals.py`, `parse-acs.py`, `resolve-*-trigger.sh`, …).
- `lib/` — retrospective-loop helpers (`*.sh`, `*.jq`), `preflight.sh`, `test/`.
- `.github/` — optional cloud tier: workflows + composite actions (incl. `vendor-plugin`).
- `.devflow/` — `config.example.json` + `config.schema.json` (+ tracked `learnings/`, `logs/`). The live `config.json` is gitignored.

## Gotchas (load-bearing — easy to get wrong)

- **Skills locate bundled helpers via the portable single-statement anchor `"${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../<dir>/…`, NEVER `${CLAUDE_PLUGIN_ROOT}`** (not substituted inside `SKILL.md` bodies). The cloud-tier **workflows** are the one exception: they use the literal vendored path `.devflow/vendor/devflow/scripts/…`. **Why this exact shape (issues #241 + #275, generalized to all 22 skill/phase files):** `${CLAUDE_SKILL_DIR}` is *empty* (not merely unset) on non-Claude-Code runners (confirmed on Copilot CLI; expected on Cursor, Codex CLI, Gemini CLI) — the runner instead reports a `Base directory for this skill:` context line the agent substitutes for the placeholder — AND Copilot CLI's inline `bash -c` marshaling **strips a variable assigned in one statement and read in a later statement of the same inline command** (`bash -c 'v=hi && echo $v'` → empty), so the old assign-then-reuse recipe (`SKILL_DIR="…"; … "$SKILL_DIR"/../…`) is banned: resolve the anchor **inline in the statement that uses it**, every call site (a reintroduced bare or cross-statement form turns `lib/test/run.sh`'s #275 pin loop RED). Windows-form runner-reported paths are normalized at **prompt time** (one standalone `wslpath -u`/`cygpath -u` probe or `lib/normalize-path.sh`'s textual rules; the skills carry **no** shell normalization mirrors anymore — create-issue's preamble is the reference prose paraphrase, kept in lockstep with the lib helper). On Claude Code the resolved value equals the bare form, so behavior is unchanged there. Relatedly (#275): `workpad.py` reads `.devflow.workpad_marker` from `.devflow/config.json` directly in Python — never by exec-ing `config-get.sh`, which Windows can't exec ([WinError 193]). The cloud-tier vendored-path carve-out is untouched. **Repo-root anchoring (#295, supersedes the #275 cwd-relative contract):** the four `.devflow/` readers — `config-get.sh`, `workpad.py` (marker read), `load-prompt-extension.sh`, `match-deferrals.py` (`_default_config_path`) — resolve the *default* `.devflow/` path anchored to the git repo **root** (`git rev-parse --show-toplevel`, falling back to `pwd`/`Path.cwd()`), mirroring `lib/config-source.sh`, so a skill run from a **subdirectory** still loads the consumer's root config/prompt-extension instead of silently missing it. A **non-empty** explicit `CONFIG_FILE`/`--config` arg is still honored verbatim (an explicit empty value still selects the root-anchored default; root anchoring applies only to the default); a no-git-root-**and**-no-`.devflow/` tree emits a stderr breadcrumb instead of a silent default. The Python readers resolve the root via a **native `git` subprocess** (Windows-safe, like the `gh` calls — not a `.sh` exec). Keep the two contract mirror sites (`config-get.sh` + `workpad.py` headers, the `SHARED REPO-ROOT CONFIG CONTRACT` phrase) and the `lib/test/run.sh` pins in lockstep. **Known limitation:** `--show-toplevel` returns the *nearest* git root, so a nested submodule/inner repo or a monorepo whose `.devflow/` is not at the git root is **not** covered.
- **Config is `.devflow/config.json` (JSON), read via python3 by a single resolver** (`config-get.sh` / `config-source.sh`) — `node` is NOT required anywhere in the config-read path (issue #220; python3 is a hard preflight prerequisite). PyYAML is required only because `match-deferrals.py` (deferred-findings blocks in PR bodies) and `consolidate-changesets.py` (changeset frontmatter) lazily import it (config itself is JSON — no PyYAML needed to read it; `workpad.py` is stdlib-only and needs no PyYAML). Renaming a config key? grep `.py`, `.sh`, `.yml`, and jq lines — silent-fail consumers exist (`match-deferrals.py`, `workpad.py`).
- **The review engine is shared.** `/devflow:review` Phases 0–4.3 are executed *verbatim* by `/devflow:review-and-fix` (which adds the fix loop + shadow pass and skips Phase 4.4). Edit `skills/review/SKILL.md`; never paraphrase the engine into the fix-loop skill.
- **`skills/review/SKILL.md` executes under TWO allowlists, and a head neither grants is SILENTLY denied (issue #363).** The auto-review path uses the `review` profile's `TOOLS='…'` line in `.github/workflows/devflow-runner.yml`; the manual `/devflow:review` comment path uses `devflow.yml`'s hoisted `TOOLS='…'` (`Resolve allowed-tools` step, consumed by both `claude_args` and the injected block, so they cannot drift). A command the skill invokes but a profile omits is refused before it runs — it does not fail loudly, it burns budget, and a run can end with **no verdict at all** (PR #340: 7 of 14 denials were the engine trying to run the test suite). `lib/test/extract-command-heads.py` (driven by `lib/test/run.sh`) extracts every head from the skill's ```bash fences — quote/comment/heredoc-aware, `$(…)`-descending, wrapper-stripping, case-arm-position-aware (issue #392: arm patterns are stripped only where an arm may legally begin — after `case … in` and after each `;;` — so a command in a case *body*, e.g. a bare subshell `(cmd)`, keeps its head instead of being swallowed as a bogus arm) — and asserts **both** allowlists grant it. **Scope boundary:** inline-backtick prose is deliberately out of reach (matching it resurrects the `git a`/`git failure`/`git said` false positives), so a prose-only command like Phase 0.3.6's `git cat-file` is pinned by direct literal instead. A second, narrower boundary (#392): the case-arm tracking is a flag, not a depth counter, so a **nested** `case` block is an accepted limitation — no fence in `skills/review/SKILL.md` nests a `case`. Adding a command to a fence? Grant it in both `TOOLS='…'` lines in the same change.
- **Every cloud review prompt carries an injected engine-ground-truth block — rendered ONCE by `scripts/render-grounding-block.sh`, never hand-copied.** It states the observed CI conclusions for the reviewed HEAD (from `scripts/summarize-ci-checks.sh`) and the exact resolved `--allowed-tools` string. Both workflows call the renderer and are pinned to carry **no** second copy: it holds security-sensitive prompt-injection prose (check names are attacker-controlled, so a name is *data to quote, never an instruction to obey*), and two drifting copies of an injection defense is the coupled-mirror hazard above at its worst. The block scopes "untrusted" to the **names**; the **conclusions** beside them are API facts — a baseline engine told otherwise refused to cite CI at all and declared its verification phase undischarged.
- **Unknown is not zero — never collapse an unestablished measurement onto a real value.** `permission_denials_count` publishes a digit string, or the literal `unavailable` when the count could not be established (diagnostics disabled/skipped, execution file absent or unparseable). Collapsing both onto `0` made the no-verdict `::error::` assert *"The harness refused 0 command(s)"* about a run it never measured — steering the reader away from the actual cause. `scripts/describe-denial-count.sh` owns the three-way clause selection (unestablished / refused none / refused N) precisely so `lib/test/run.sh` can drive every arm: an inline `case` in YAML is untestable, and a reordered arm misattributes the diagnosis while the suite stays green.
- **A value that decides a SELECTION or an EMITTED result must not be derived through a non-preflight PATH tool.** `lib/preflight.sh` guarantees only `git`/`gh`/`jq`/`python3`/PyYAML — **not** `tr`/`sed`/`wc`/`cut`/`head`. A missing one does not fail: the pipeline runs, the value comes out empty, and the wrong thing is selected. Verified live in this repo: `_publish_denials`' `sed`-based count made a run with 14 denials publish `0` and emit no `::warning::`; a `wc`-derived total made a head *with* CI signals render as "No CI signals reported". Derive such values with bash builtins (`while IFS= read -r`, `case`, `${var#prefix}`). *Cosmetic* sanitization may still use `tr`/`cut` **iff** a missing tool fails CLOSED (an emptied name becomes `(unnamed check)`, never a raw injected one). This is the repo's `.devflow/prompt-extensions/review-and-fix.md` guard-class 2, and it applies to code you are writing right now, not only to code you are reviewing.
- **Partition invariant with Anthropic's Claude app.** Every DevFlow trigger `if:` negates `@claude`, so DevFlow and the stock `claude.yml` never double-fire. Tests in `lib/test/run.sh` enforce this. Never create/overwrite `claude.yml`. Triggers fire on real *comments* only — never issue/PR bodies or titles.
- **Cloud allowlist needs the helper as the command's LEADING token.** No `VAR=value` prefix, no `bash <path>` wrapper — otherwise the read-only `review` profile silently denies it (overrides/telemetry resolve to empty). Invoke `config-get.sh`/`efficiency-trace.sh`/`resolve-review-overrides.py` directly by path.
- **Vendor to `.devflow/vendor/devflow/`, never `.claude/`.** `claude-code-action` `rm -rf`s sensitive paths (incl. `.claude/`) and restores them from the *base* branch before installing plugins, which would wipe a plugin vendored under `.claude/`.
- **In a `claude-code-action` job, `git push` authenticates with the credential `actions/checkout` persists — NOT the `github_token` passed to the action.** `checkout@v6` writes `http.<server>/.extraheader` into an external config wired in via `includeIf.gitdir:` (covering `.git/worktrees/*`), not into `.git/config`, so the action's `git config --unset-all http.<server>/.extraheader` clears nothing and its `git remote set-url origin https://x-access-token:<app-token>@…` loses to that surviving preemptive `Authorization:` header. Unseeded, every push runs as `github-actions[bot]` (no `workflows` permission): ordinary pushes succeed and ONLY `.github/workflows/` pushes fail — `refusing to allow a GitHub App to create or update workflow … without workflows permission` — which reads as a missing App permission but is not (`devflow-autopilot`, app_id `3102164`, declares `workflows: write`). Every pushing job mints its App token **above** the checkout and seeds `token: ${{ steps.app-token.outputs.token || secrets.GITHUB_TOKEN }}` (`devflow-implement.yml` `claude`, `devflow.yml` `command`, `version-consolidate.yml`); the read-only `DevFlow-Reviewer` token is NEVER a checkout credential (#300). Non-pushing mints (`gate`, `review_dedupe`) hand their token straight to `gh` and stay put. Coupled sites: the two workflow mints ↔ the `checkout-token:` pins in `lib/test/run.sh`. (#357)
- **Changing one site of a coupled invariant? Update every mirror site AND reconcile the asserting tests in the SAME change — never a follow-up commit.** This is the dominant `convention-violation` sub-pattern: the bot edits one half of a two-sided contract and leaves the other half stale, so the artifacts silently disagree (and the change ships CI-red or with a desync a later shadow pass has to catch). A "coupled site" is any value/text that more than one file is required to carry identically: a label literal, a constant, a `SKILL.md` contract pin a `run.sh` grep asserts, a config key name (grep `.py`/`.sh`/`.yml`/jq lines — silent-fail consumers exist), an attribution marker `run.sh` counts, or a self-record (the issue workpad's Plan/AC) that must track what the PR actually shipped after a revert. Known live pairs to treat as one edit: the Phase 4.3 `--tick-progress` Progress-label literal in `skills/implement/phases/phase-4-documentation.md` (the implement skill is split into a thin `skills/implement/SKILL.md` orchestrator + `phases/phase-N-*.md` reference files) ↔ the `_PROGRESS_PHASES` / `_STATUS_TO_PROGRESS_PHASE` source-of-truth in `scripts/workpad.py`; and any test (`lib/test/run.sh`) that asserts a marker/pin you are removing — delete the assertion in the same commit, not after the gate goes red. Discipline: before editing such a value, `git grep` the literal across `.sh`/`.py`/`.jq`/`.yml`/`.md`/`.gitignore` to enumerate every site, change them together, and if a revert changes what shipped, update the workpad self-record to match.
- **A contract sentence assembled from wrapped adjacent string literals defeats line-based `git grep` — audit and pin the *rendered* surface, and search whitespace-normalized when enumerating coupled sites (issue #375, evidence from #371).** A phrase split across adjacent source literals — an argparse `help='… OLD does '` `'not) …'`, an f-string `raise`, any string concatenated across a line break — lives on **no single line**, so `git grep 'OLD does not'` finds nothing and a coupled-site sweep falsely reports "no stale mirrors" (this is how the #371 stale `help=` sentence survived three times). Two consequences bind here: (1) when enumerating coupled sites, also search **whitespace-normalized** — `tr -s '[:space:]' ' '` over the file (note this rejoins newline-wrapped phrases but does **not** remove the quotes *between* adjacent literals, so an argparse `help=` split at a literal boundary needs the rendered check next); (2) **pin the rendered surface, not the source** — for multi-literal `help=` text, pin the captured `prog --help` output or the real emitted stderr, never a source `grep` that can't see the concatenation. `lib/test/pin-corpus-lint.py` (the `#375` wrapped-literal meta-guard, self-scanned by `lib/test/run.sh`) turns a source-`grep` pin whose phrase is off-line — or lands in a multi-literal `help=` — RED at the desk; its sibling pin-in-comment lint turns a pin literal that *also* appears in a comment of its target RED (the #370 count-inflation defect). The mutation-taking `assert_pin_red_under <name> <literal> <mutation> [file]` is how a behavioral-fix pin proves it catches the *guarded regression* (a `sed -E` mutation re-introducing the bug), not merely its own line vanishing (what `assert_pin_red_on_removal`'s whole-line strip cannot distinguish) — the behavioral-fix-pin rule in `.devflow/prompt-extensions/implement.md`, `skills/implement/phases/phase-2-implement.md` §2.3, and `skills/review-and-fix/SKILL.md` Step 3 routes new pins through it and records the mutation run + the pin observed RED (evidence, not attestation).
- **External binaries resolve through the execution-verified `resolve-*.sh` family — never reintroduce a bare `gh`/`jq` invocation in a `.sh` helper.** `lib/resolve-bin.sh` is the single shared resolver (issue #247): `DEVFLOW_<TOOL>` override wins verbatim and is **never probed** (the test-stub contract); else the first of `<tool>`/`<tool>.exe` whose `--version` runs (network/auth-free); else the bare tool + stderr breadcrumb, always rc 0. `jq` routes through it via sourced `lib/resolve-jq.sh` (`DEVFLOW_JQ`) in every jq-calling helper — `lib/test/run.sh` pins this both ways (a de-converted helper or a reintroduced invocation-position bare `jq` goes RED); `lib/resolve-gh.sh` delegates to it; override derivation for jq/gh is pure bash (no `tr` dependency). `lib/normalize-path.sh` converts Windows-form paths (wslpath → cygpath → env-detect → unchanged+breadcrumb, all failure modes fail closed); the skills carry **no shell mirrors** of it (issue #275) — Windows-form anchors are normalized at prompt time, with create-issue's preamble as the reference prose paraphrase and the shared 17-way preamble (canonical copy `skills/docs/SKILL.md`, identity-pinned by run.sh P4) carrying the same rules — edit lib + create-issue + the canonical copy (re-propagated 17-way) + the T5/T5b/T5c pins together. Partial-copy/unreadable-sibling deployments degrade with breadcrumbs (never an empty `DEVFLOW_JQ`, never a raw set -e abort). `install.sh` alone carries an inline *adaptation* (standalone/curl-piped; a broken `DEVFLOW_JQ` warns and routes to python3 — a deliberate divergence, don't "unify" it).
- **Embedded `jq '…'` programs are inside bash single quotes — keep their `#` comments apostrophe-free ASCII.** `scripts/scaffold-config.sh` carries multi-line jq programs; an ASCII `'` (e.g. `user's`) terminates the bash string (shellcheck **SC1073/SC1011** + test failures), and a curly `'` trips **SC1112** (CI's lint job fails on it). Reword to avoid contractions. Same trap: `startswith`/string ops on a possibly-non-string field abort the *whole* filter — guard with `(.x | strings)` and type-check before indexing (`(.a | type) == "object"`).
- **Editing a best-effort parser (`scaffold-config.sh`, `config-get.sh`, the jq/`config.json` consumers)? Lead with an adversarial input-shape matrix, not the happy path.** These run on configs a human can hand-corrupt, so the bug class is "a shape detonates the filter or yields a misdirected/silent breadcrumb." Enumerate the finite matrix up front — `{top-level, devflow_review, agent_overrides, a value, model}` × `{object, array, scalar, valid-falsy (explicit false / 0 / empty string), missing, wrong-type}` — and add a `run.sh` block asserting *exit-0 + a specific (not generic) breadcrumb* for each. The **valid-falsy** row is load-bearing: a real `false` / `0` / `""` that an `// true` / `// default`-style extraction silently coerces to its truthy default is the documented-off-switch-that-never-worked bug (issue #312, the #304 workflow `// true` coercion). **Inline jq inside a workflow file is a parser this matrix applies to** — not only standalone `.sh` / `.py` / jq / `config.json` consumers; a workflow expression extracting a config value gets the same shape sweep. One deterministic sweep beats finding these one-at-a-time across review iterations.
- **Adding a guard, predicate, or coverage-invariant? Trace every operand it reads back to its producer and prove the producer emits it on every path the new code now selects — a guard whose comparand can be absent fails open exactly where it claims to fail closed.** This is the recurring `unverified-assumption` bug class: the bot writes a check that *asserts* an invariant but reads a field the upstream writer never guarantees, so the check silently passes on the very inputs it was added to catch. Discipline when hardening: (1) for **every** comparand/field your guard reads, name the exact producer line and the shape it writes; (2) when a new selection/predicate *widens* the set of inputs (a new union arm, a new label/closes-issue path), re-walk every *downstream* derivation those inputs flow into and confirm each still resolves — the gap is usually one path's worth of inputs the older derivation never saw; (3) add a `run.sh` assertion for the absent-operand / newly-selected-but-underived shape, asserting the guard fails *closed* (or the derivation still resolves), not just the happy path. Treat "I asserted X" and "X holds on the inputs that now reach this code" as two different claims — the second needs evidence.
- **`DevFlow` is the reserved provenance label — retrospective detection is label-first, not branch-prefix-first.** DevFlow stamps the literal `DevFlow` label (a hardcoded constant — *the provenance-detection path in `lib/scan.sh` / `lib/classify-pr-kind.jq` matches this exact string and no config key controls it*; `docs.labels` is the *separate* docs-pass label, default `Documented`) on every issue/PR it creates, via the idempotent best-effort `scripts/ensure-label.sh <name>` (always exits 0). `lib/scan.sh` selects a PR via a **union predicate** — (a) the `DevFlow` label (author/branch-agnostic), (b) a watched author that **closes an issue** (`closingIssuesReferences` non-empty — the branch-naming-independent fallback), (c) `implementation_branch_prefix` *only when set non-empty* (empty ≠ match-all). Don't reintroduce a branch-prefix-only predicate; it silently no-ops on any bot whose branches don't match the prefix. **The follow-up issues `/devflow:implement` files for deferred work (Phase 4.0 deferred ACs, Phase 4.0.5 deferred review findings) ARE labeled by a config key — `deferred.labels` (default `DevFlow,Deferred`)** — resolved/normalized/applied by the *skill* (Phase 4.0/4.0.5) exactly like `docs.labels`, never by `file-deferrals.py` (config stays resolver territory for shell/skill callers — read through `config-get.sh`, not re-parsed ad hoc in Python; the one sanctioned exception is a Python helper reading its own key in-process where a `.sh` exec cannot work on Windows, e.g. `workpad.py`'s marker read, issue #275). That key controls only *deferred-issue labeling*; it does **not** touch the hardcoded `DevFlow` provenance string the scan/classify path matches.
- **Label-create, label-apply, and PR-body edits go through REST `gh api` (repo-scope only), never GraphQL-resolving porcelain.** `gh label create` / `gh issue edit --add-label` / `gh pr edit --add-label`/`--body` resolve the repo via org-scoped GraphQL and fail silently under a repo-scoped token (GitHub App installation token, fine-grained `repo`-only PAT). Use the REST equivalents (`POST repos/{owner}/{repo}/labels`, `POST repos/{owner}/{repo}/issues/{n}/labels`, `PATCH repos/{owner}/{repo}/pulls/{n}` with `{owner}/{repo}` placeholders `gh` fills from the remote / `$GITHUB_REPOSITORY`). `scripts/apply-labels.sh <number> <label…|comma-list>` is the **single hardened label-apply path** every call site routes through (mirrors `scripts/ensure-label.sh`'s best-effort, always-exit-0, stderr-breadcrumb contract; `ensure-label.sh` does REST label *creation*). Don't reintroduce porcelain in a new call site (issue #228).
- **The bash that RUNS the `.sh` helpers is chosen at the INVOCATION boundary, honored via `DEVFLOW_BASH` — deliberately NOT a `resolve-*.sh` helper (issue #248).** Unlike `gh`/`jq` (tools a *running* bash calls, chosen by a sourced resolver), the shell that *runs* the scripts is picked before any `.sh` executes, so a sourced resolver can't select it (bootstrap: it would itself need a chosen bash to run). DevFlow supports **any** POSIX bash — WSL bash / Git Bash / MSYS2 bash, none mandated — and `DEVFLOW_BASH` is the invocation-layer override the agent/runner honors to point at one. `lib/preflight.sh` is a **diagnostic, not a selector**: it emits a `devflow-bash:` breadcrumb (interpreter path + `$BASH_VERSION`), surfaces `DEVFLOW_BASH` when set, and — when `$BASH_VERSION` is empty (a non-bash POSIX shell) — prints a remedy naming the three supported bashes + the override and **exits 1 BEFORE the first bash-only `${BASH_SOURCE[0]}`** (fail-closed; it must stay above that line). Non-goal: a host with **no** POSIX bash at all (PowerShell-only) is out of scope. `lib/test/run.sh` pins the breadcrumb + remedy (the `#248` block: the non-bash remedy runs under a real non-bash shell when present — `dash`/`busybox` — with removal-proof static pins backstopping bash-only hosts; `lib/test/` is excluded from CI shellcheck). Edit the preflight block, the `#248` test block, and `docs/install.md`/`README.md` together.
- **DevFlow's `gh` binary is chosen by ONE execution-verified resolver, `lib/resolve-gh.sh` (`devflow_resolve_gh`) — the sibling of `lib/resolve-python.sh`; never hardcode a bare `gh` default.** It honors a non-empty `DEVFLOW_GH` (top precedence, echoed with no probe — the probe runs only when `DEVFLOW_GH` is unset or empty (`:=` fires on both), so the test-suite's `DEVFLOW_GH` stubs are untouched), else returns the first of `gh`, `gh.exe` whose network/auth-free `gh --version` **runs** (rejecting a present-but-unrunnable Windows/WSL shim — the shadowing-`gh` bug), else bare `gh` (rc 0, so the caller's best-effort warning path still fires). Every gh-calling shell helper sources it and assigns via `: "${DEVFLOW_GH:=$(devflow_resolve_gh)}"`; `lib/preflight.sh` sources the same resolver but deliberately does **not** use that `:=` form — it DETECTS only (`_GH="$(devflow_resolve_gh)"`, re-probed, never assigned back to `DEVFLOW_GH`), consistent with `docs/DEVFLOW_SYSTEM_OVERVIEW.md`'s "preflight only detects" contract; the four Python gh-callers (`workpad.py`, `file-deferrals.py`, `match-deferrals.py`, `parse-acs.py`) read `os.environ.get("DEVFLOW_GH") or "gh"` (default `gh`) but deliberately do **not** run the probe. `DEVFLOW_GH` is the documented Windows/WSL escape hatch (`docs/install.md`). Adding a new gh-caller? Route it through the resolver — `lib/test/run.sh` pins "no bare `DEVFLOW_GH:=gh` default remains" **and** ">=13 helpers source `resolve-gh.sh`" (issue #245).
- **The workpad lives on the ISSUE, not the PR thread.** `lib/fetch-pr-context.sh` reads the `<!-- devflow:workpad -->` comment (and its `## Devflow Reflection` bullets → `reflections[]`, and final Status → `workpad_final_status`) from the **issue** comments — detect it by marker, never by author (it's authored by github-actions). - **Versioning is repo-policy, kept OUT of the engine — and DevFlow versions itself with changesets (issue #290).** `/devflow:implement` is versioning-agnostic; DevFlow manages its own version through its consumer prompt extension `.devflow/prompt-extensions/implement.md` — **edit that extension, not the skill**, to change the rule. A PR touching the engine surface (`skills/`, `agents/`, `lib/`, `scripts/`, workflows, config schema) adds a uniquely-named `.changeset/*.md` file (a `bump: patch|minor|major` frontmatter key + Keep-a-Changelog prose, PR-cited) and does **NOT** edit `.claude-plugin/plugin.json` or `CHANGELOG.md` directly — those two shared hot spots caused a version/CHANGELOG merge conflict on every concurrent PR. The Phase 3 review gate now FAILs an engine-surface change that carries **no** changeset (it replaces the old version↔`CHANGELOG` presence check); internal-only changes (tests, CI, dev-only docs) add no changeset; the default `bump` is **`patch`** (`minor`/`major` only when the issue explicitly authorizes it). On merge to `main`, the `version-consolidate` workflow runs `scripts/consolidate-changesets.py`: it bumps `plugin.json` by the **highest** pending bump, prepends one dated CHANGELOG entry from all pending prose, deletes the consumed changesets, and commits with the **load-bearing `chore: bump version`** subject (a coupled invariant with `skills/docs-release-notes/SKILL.md` Step 4b — the producer is now that workflow, not an in-PR commit; both pinned in `lib/test/run.sh`). **The workflow YAML lives at `.github/workflows/version-consolidate.yml`.** Editing any `.github/workflows/` file needs a `workflows`-scoped push — the App HAS that scope; what was broken is which credential reached `git push` (see the checkout-token gotcha above, #357). A malformed changeset fails the Action loudly (naming the file); zero pending is a clean no-op. The consolidator makes no `git` calls (the workflow stages/commits) so it is unit-tested directly in `lib/test/run.sh`.

## Conventions

- **Portability:** no GNU-only flags. Use `python3` for date math (not `date -d`) and ERE / `sed -E` (not `grep -P`). Helpers must work on macOS/BSD without GNU coreutils.
- **No secrets, owner-specific IDs, or product names in committed files.**
- New `.py`/`.sh` files carry the SPDX header (`SPDX-FileCopyrightText: 2026 Daniel Radman` / `SPDX-License-Identifier: MIT`).
- Docs reference bare source paths (`scripts/workpad.py`), never `path:line` — line numbers rot.
- **Updating `CLAUDE.md`? Use `/claude-md-management:revise-claude-md` (targeted additions) or `/claude-md-management:claude-md-improver` (full audit) — never hand-edit without running at least one.**
- **Autonomous `/devflow:implement` runs make a *required* `CLAUDE.md` edit directly — carve-out to the rule above (issue #366).** Any `CLAUDE.md` edit an autonomous DevFlow run is *required* to make — whether by a Phase-3 review finding **or by the issue's own acceptance criteria** — is made **directly by the orchestrator**, citing this carve-out and recording it in the workpad, **never** by invoking `revise-claude-md`/`claude-md-improver` mid-run (a nested interactive skill runs as a tail call and stalls the run — the defect issue #366 fixes). Interactive/human sessions still use `revise-claude-md`/`claude-md-improver` per the bullet above. Other interactive skills a run is required to invoke are instead dispatched into an **Agent-tool subagent** (issue #362) — see `skills/implement/SKILL.md`. This bullet is the `CLAUDE.md` half of a coupled pair — its mirror is the `CLAUDE.md` edit carve-out in `skills/implement/SKILL.md`; edit both together.
- **Editing any skill file (`SKILL.md`)? ALWAYS invoke the `writing-skills` skill first** (`superpowers:writing-skills`) and follow its RED/GREEN discipline — never hand-edit a `SKILL.md` body directly. This applies to every `skills/*/SKILL.md` (the shared review engine, the docs family, retrospective, etc.); the skill enforces the verify-before-deploy loop that keeps a skill's triggering + behavior intact.
- **Marketing/positioning copy** (`README.md`, `docs/DEVFLOW_SYSTEM_OVERVIEW.md`) leads with the wedge — *out-of-the-box agentic coding works on pet projects but can't complete a real ticket in a large production codebase; DevFlow makes it ship*. Honor the honest-claims guardrails (review-ready, **not** auto-merged; shadow review **narrows, never closes** the gap). §21 is the messaging source of truth.
- Don't push to `main` from an agent; the classifier blocks `git push origin main`. Inside `/devflow:implement` & `/devflow:review-and-fix` there is standing permission to auto-commit+push to the *feature* branch.
- **Local-tier classifier blocks shell-script invocation — this is expected; use the documented fallbacks, don't fight it.** On the local/interactive tier the permission classifier routinely denies `bash <path>`, `shellcheck`, `ruff`, and helper scripts invoked by path (the recurring friction behind retrospective pattern `convention-violation`, PRs #72/#85/#87/#98). It is a **structural harness boundary**: an agent is barred from writing `.claude/` to widen its own allowlist (`docs/efficiency-trace.md` "Note on this repo's `.claude/settings.json`"), so the run cannot self-grant the missing permission — the fix is operator-side, not in-run. When blocked, fall back **without re-deriving it each run**: run helpers via `python3 <path>` (the classifier permits Python by path) and `jq`; rely on the required CI job `lib + python tests` to run `bash lib/test/run.sh` + lint on push (verify it is green before merge); for the workpad use `gh api` PATCH if `workpad.py` is denied. To actually widen the allowlist, a **human** adds the rules at user scope (`~/.claude/settings.json`) or runs the consent-gated `scripts/provision-auto-mode.sh` / `scripts/provision-local-settings.sh` — never commit a `permissions.allow` block to this repo's `.claude/settings.json` from an agent (the classifier denies it, and the cloud tier discards `.claude/` anyway).
