# Changelog

All notable changes to DevFlow are documented here. The format follows
[Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and the project aims
to follow [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [2.8.72] — 2026-07-04

### Changed
- **DevFlow reviews now post under a dedicated `DevFlow-Reviewer` GitHub App identity, so the review path never collides with the PR author.** GitHub forbids requesting-changes-on / approving your own PR, so when the review agent ran under the same primary `devflow-autopilot` App that authors PRs, Phase 4.4's `gh pr review --request-changes`/`--approve` failed and no visible formal review was recorded (the merge still blocked via the required `Devflow Review` check). Every review path — the automated runner (`.github/workflows/devflow-runner.yml`) and the manual `/devflow:review` command (`.github/workflows/devflow.yml`) — now mints a downscoped installation token from a separate `DevFlow-Reviewer` App (repository variable `DEVFLOW_REVIEWER_APP_ID` + secret `DEVFLOW_REVIEWER_PRIVATE_KEY`, permissions `contents:read`/`issues:read`/`pull-requests:write`/`actions:read`) when the variable is set, otherwise `github-actions[bot]` (`GITHUB_TOKEN`) — and never the primary app token. `/devflow:pr-description` and `/devflow:review-and-fix` keep the primary App token. **Upgrade note:** existing single-app consumers' review attribution moves from their DevFlow App to `github-actions[bot]` until they configure `DevFlow-Reviewer`. See `docs/cloud-setup.md`. (#303)

## [2.8.71] — 2026-07-04

### Fixed
- **The `/devflow:implement` workpad Run link now stays current across stall/retry resumes, and the draft PR links back to the run that created it.** When the cloud `gate` job's early-workpad step finds an existing workpad (a re-trigger or a stall-backstop auto-resume), it now deterministically rewrites the workpad's `**Run:**` link to the current run via `workpad.py update --run-link`, at the workflow level — so an operator watching a stalled/retried run can click straight from the issue workpad to the currently-active job's logs instead of the original run's. This write is best-effort (a `::warning::` breadcrumb, then exit 0) and lands regardless of whether the subsequent `claude` job goes on to execute Phase 1.3. Separately, the Phase 3.1 draft-PR body now carries a `[View run](...)` line under `Resolves #{issue_number}`, omitted entirely on a local-tier run (no GitHub Actions run) rather than rendering a broken link, so a reviewer can trace a mid-run draft PR back to its originating run. (#302)

## [2.8.70] — 2026-07-04

### Fixed
- **DevFlow now resolves `.devflow/` from the git repo root, so a consumer's committed config and prompt extensions load even when a `/devflow:*` skill runs from a subdirectory below the repo root.** Previously four readers resolved `.devflow/` relative to the current working directory, so a skill invoked from a nested project directory (the norm in large enterprise/monorepo layouts) silently lost every committed customization: `scripts/load-prompt-extension.sh` printed nothing (the "no extension" no-op), and `scripts/config-get.sh`, `scripts/workpad.py` (the in-process `workpad_marker` read), and `scripts/match-deferrals.py` (`.devflow.allowed_bots`) all reverted to defaults — with no error and no warning, and invisible to CI (whose checkout cwd equals the repo root). All four now anchor the **default** `.devflow/` path to the git repo root (`git rev-parse --show-toplevel`, falling back to `pwd`/`Path.cwd()` when not in a git tree), mirroring `lib/config-source.sh` and removing the internal split-brain between the two config resolvers. A **non-empty** explicit `CONFIG_FILE` argument to `config-get.sh` (and `--config` to `match-deferrals.py`) is still honored verbatim (an explicit empty value still selects the root-anchored default); when cwd already **is** the repo root, resolution is byte-for-byte unchanged; and when neither a git root nor a `.devflow/` directory can be located, each reader emits a one-line stderr breadcrumb before falling through to defaults, so the silent-drop class cannot recur unnoticed. The Python readers resolve the root via a native `git` subprocess (Windows-safe, like the existing `gh` calls — not a `.sh` exec, per #275). This supersedes the #275 cwd-relative contract. **Known limitation:** `--show-toplevel` returns the *nearest* git root, so a nested submodule/inner repo, or a monorepo whose `.devflow/` is deliberately not at the git root, is not covered (consistent with `config-source.sh`). (#295, PR #299)

## [2.8.69] — 2026-07-04

### Changed
- **DevFlow now versions itself with changesets instead of an in-PR `plugin.json` + `CHANGELOG.md` bump, ending the version/CHANGELOG merge conflict that taxed every concurrent PR.** Each engine-surface PR adds a uniquely-named `.changeset/*.md` file (a `bump: patch|minor|major` frontmatter key plus Keep-a-Changelog prose) and no longer edits `.claude-plugin/plugin.json` or `CHANGELOG.md`. On merge to `main`, the new `version-consolidate` GitHub Action (`.github/workflows/version-consolidate.yml`) runs `scripts/consolidate-changesets.py`, which consumes every pending changeset: it bumps `plugin.json`'s `version` by the highest pending bump type (patch < minor < major — one increment even when several are pending), prepends a single dated, PR-cited entry assembled from all the pending prose, deletes the consumed changeset files, and commits with the load-bearing `chore: bump version` subject that `docs-release-notes` Step 4b still keys on. A malformed changeset (missing/invalid `bump`, unparseable frontmatter, empty prose) fails the Action loudly and names the offending file — never a silent skip or a partial write; with no pending changesets it is a clean no-op. The `/devflow:implement` versioning policy, the Phase 3 review gate (now "engine-surface change ⇒ changeset present"), the docs, and the `lib/test/run.sh` coupling pins all move together. (#293)

### Fixed

The `/devflow:implement` Phase 3.3 seam (`skills/implement/phases/phase-3-review.md`) restates that obligation for the inline driver, adds the Write-tool-for-`.devflow/tmp`-scratch discipline, and corrects the misattribution that a cloud `claude-code-action` denial is "the local-tier classifier" or license to leave the instrumented loop (it is neither — `Skill`/`Agent`/`Write`/`efficiency-trace.sh`/`workpad.py`/`config-get.sh` are all allowlisted on the implement job). The existing 3-layer observability backstop is preserved intact; the docs state honestly that only the **effectiveness** half of the telemetry is made recoverable by the emit-obligation — the **token/wall-clock-cost** half is live-only and cannot be reconstructed once the loop is abandoned. `lib/test/run.sh` pins the three new prose invariants (non-optional-emit obligation, the Write-tool-not-shell-redirect discipline at the seam, the misattribution correction) removal-proof, additive to the #235/#236 backstop pins. (#296, closes #296)

## [2.8.66] — 2026-07-03

### Changed
- **A review finding whose sole observable impact is an inaccurate or stale in-code comment on a comment the diff did not add or modify is now classified below Important — deterministically — instead of shipping at 🟠 Important.** The review engine already recognized this class (Phase 4.1.5 over-grade "shape 2", diagnostic-or-cosmetic-only), but by the #195 design it only *annotated* and never demoted, so a comment typo dressed as a Major finding still diluted the Important bucket and could drive a REJECT or a promoted fix-loop iteration. Phase 4.1.5 in `skills/review/SKILL.md` now defines a **deterministic in-code-comment cap**: a finding whose sole impact is an inaccurate/stale in-code comment on a comment the diff did **not** add or modify is capped at 🟡 Suggestion / Minor and Phase 4.2 does **not** REJECT on it, regardless of the severity a review agent assigned. The cap is framed as a severity-*classification* rule keyed only on two observable properties (impact is solely an in-code comment; the comment was not diff-touched), never on a re-judgment of the finding's merits, so it does not reopen the #195 lenient-verdict hole. It is narrow by construction: it **excludes** any comment the diff added or modified (still a non-demotable self-contradicting-diff REJECT at every threshold — the #263 carve-out is preserved verbatim) and applies to **in-code comments only** (the log/breadcrumb/error-message diagnostic sub-cases of shape 2 keep their advisory-annotate-only treatment). `skills/review-and-fix/SKILL.md` Step 2.6's over-grade calibration gate honors the same cap by consuming the single-sourced 4.1.5 definition (no forked shape copy) — a comment-only-on-unmodified-comment finding produces the required `severity-calibrated` record deterministically and therefore cannot drive a Decide-outcome-2 promotion. `docs/DEVFLOW_SYSTEM_OVERVIEW.md` (§4.1.5) and `docs/shadow-review.md` (the #195 "annotates, but never demotes" section) are updated to state the cap and its relationship to the never-auto-demote invariant, and `lib/test/run.sh` reconciles the existing advisory "verdict computation unchanged" pin (and the 4.1.5 heading pin) to the narrow carve-out — keeping the never-auto-demote pins intact — and pins the new cap, its diff-added/modified exclusion, its in-code-only scope (including the machine-significant-comment exclusion), and the review-and-fix consumption. (#291)

## [2.8.64] — 2026-07-03

### Fixed
- **The cloud `/devflow:implement` stall backstop now survives a run longer than a GitHub App installation token's ~60-minute lifetime, and a gh-api/transport/auth failure reading the workpad is no longer misclassified as an unreadable workpad.** The stall backstop previously inherited the App token minted at the *start* of the `claude` job, which can run well over an hour; on a long run that token was already expired by the time the backstop fired, so its `workpad.py status` read got `HTTP 401`, a perfectly healthy workpad was classified `unreadable → fail-unreadable`, and auto-resume could never fire on exactly the long runs that need it. `devflow-implement.yml` now mints a **fresh** `actions/create-github-app-token@v2` token (`id: backstop-token`) just-in-time immediately before the `Stall backstop` step — mirroring the writer mint (same `app-id`/`private-key`, full installation scope; the gate adds `always()` so it still mints after a failed `claude` job) — and the backstop's `GH_TOKEN` + `APP_TOKEN_PRESENT` reference that fresh token; when `DEVFLOW_APP_ID` is unset the step is skipped and the path falls back to `GITHUB_TOKEN` byte-for-byte as before. `scripts/workpad.py status` now exits **3** on a gh-api/transport/auth failure (the `gh repo view` repo-lookup and the `gh api` comment-fetch paths), kept distinct from exit 1 (content-shape: unparseable/absent/unrecognized Status) and exit 2 (no workpad); `_fail`/`_repo_full`/`_find_workpad_comment` gained an optional exit-code parameter (default 1 — every other caller unchanged). `scripts/stall-backstop-decide.sh` maps a new `auth-failure` class to the `fail-auth` decision, and the workflow classifies both a status exit 3 and a comment-fetch `gh` failure as `auth-failure`: it fails loud **without** consuming a resume attempt and never mislabels it `unreadable`. This repo's own `devflow_implement.effort` is lowered to `low` (`.devflow/config.example.json` is unchanged, still `high`), making runs cheaper and less likely to exceed the 60-minute window in the first place. `lib/test/run.sh` pins the exit-3 contract (both transport paths), the backstop's exit-3 → `auth-failure` branch, the comment-fetch `auth-failure` branch, the `fail-auth` decision, the load-bearing `always()` guard on the fresh mint step (so a "harmonization" to the writer mint's bare `if:` that would skip the mint on a failed `claude` job turns the suite RED), and the fresh-mint step coupling — all mutation-checked RED-on-break. `_find_workpad_comment` also guards a rc-0 `gh` response that parses but is not a JSON array (an error envelope like `{"message":"Bad credentials"}`), routing it through the same exit code as a parse failure (exit 3 for `cmd_status`) instead of an uncaught `AttributeError`, so the exit-3 promise covers a wrong-shape body, not only an unparseable one. (#287)

## [2.8.63] — 2026-07-03

### Fixed
- **The `review_dedupe` guard's HEAD/BRANCH resolution now surfaces `gh` failure causes instead of swallowing them.** The two `gh pr view` calls that resolve a PR's `headRefOid`/`headRefName` previously discarded `gh`'s stderr via `2>/dev/null || echo ""`, so an authentication failure, a rate-limit, a transient API error, and a genuinely-missing PR all collapsed into the same generic `::notice::Could not resolve PR #$PR HEAD` — a maintainer reading the Actions log had no way to distinguish a persistently-failing guard from one correctly finding no in-flight run. Both resolutions now mirror the guard's own Signal 1/Signal 2 pattern: a `gh` failure captures stderr to a separate temp file (never merged into the resolved value) and emits a `::warning::` naming the cause, while the success-but-empty/null case keeps its existing `::notice::` fail-open path unchanged. Control flow is unaffected in every branch — the guard still fails open (HEAD failure `exit 0`; BRANCH failure treats Signal 2 as `0`). `mktemp` itself is guarded against failure (falls back to `/dev/null`, matching the existing `devflow-implement.yml` stall-backstop precedent) so a `mktemp` failure can never abort the step under `set -e` and defeat the fail-open contract; captured stderr is newline-collapsed before embedding so a multi-line `gh` error can't break the `::warning::` annotation. `lib/test/run.sh` gains four static pins (no blind swallow remains; both failure warnings carry the captured-stderr prefix; the success-but-empty notice is retained), mutation-checked RED-on-break. (#282, closes #269 follow-up)

## [2.8.62] — 2026-07-03

### Fixed
- **`workpad.py status` now fails closed on a present-but-unrecognized `Status` word instead of silently misreading it as a healthy in-progress stall.** `cmd_status` derived its classification from `_status_glyph`, which defaults to the 🚀 (interim) glyph for any word that doesn't start with `complete`/`blocked` — so a corrupted or hand-mangled `Status` line (e.g. `**Status:** Frobnicating`) printed `interim 🚀 Frobnicating` at exit 0, and the cloud `/devflow:implement` stall backstop (#266) burned a real auto-resume attempt on a workpad it could not actually read, instead of failing loud via its existing `non-zero → unreadable → fail-unreadable` path. A new `_is_recognized_status_word` helper, single-sourced from `_STATUS_TO_PROGRESS_PHASE`'s keys plus the literal terminal word `blocked` (exact-match, deliberately **not** delegating to `_status_glyph`'s looser `startswith('complete'/'blocked')` prefix check — which would itself let a corrupted word like `Completely wrong` or `Blockeddependency` slip through), gates `cmd_status` before it classifies a word: an unrecognized word now exits 1 with a diagnostic naming the bad word and the full recognized vocabulary. Every canonical status word, the pre-existing missing/empty-`Status`-line paths, and `_status_glyph`/`_strip_status_glyph`'s other callers (the `--status` write path, the terminal self-record gate, progress-phase nesting) are unchanged — the write path is deliberately out of scope, since it takes an orchestrator-supplied (trusted) status rather than untrusted read input. (#281, carried forward from the #268/PR #277 run)
- **Every local-tier skill's bundled-helper anchor now resolves first-class on GitHub Copilot CLI / Windows (portability wave 2).** All 22 skill/phase files (create-issue, 16 further `skills/*/SKILL.md`, the implement orchestrator, and its four `phases/phase-*.md`) drop the bare `${CLAUDE_SKILL_DIR}/../../…` helper anchor — which expands empty on non-Claude-Code runners and collapsed every helper path to a broken `/../../…` — for a **single-statement portable form**, `"${CLAUDE_SKILL_DIR:-<absolute skill base directory this runner reports in context>}"/../../…`, resolved inline at each call site. The single-statement shape is load-bearing, not stylistic: Copilot CLI's inline `bash -c` marshaling strips a variable assigned in one statement and read in a later statement of the same inline command, which also defeated the previous create-issue recipe's assign-then-reuse form (`SKILL_DIR="…"; … "$SKILL_DIR"/../../…`) — that form is now migrated away everywhere and banned by test. Windows-form runner-reported base directories are normalized at **prompt time** (one standalone `wslpath -u`/`cygpath -u` probe or `lib/normalize-path.sh`'s textual drive-letter rules); create-issue's two inline shell normalization mirror blocks are removed, with its preamble prose paraphrase becoming the lib helper's lockstep reference. The `/devflow:implement` entry-gates keep failing closed — they halt with an attributable breadcrumb when neither `$CLAUDE_SKILL_DIR` nor a runner-reported base directory resolves — and on Claude Code every changed skill resolves to the identical path as before (zero behavior change on the reference runner). `scripts/workpad.py` additionally reads `.devflow.workpad_marker` from `.devflow/config.json` **directly in Python** instead of exec-ing `config-get.sh`, so a repo's custom workpad marker is honored on Windows (no more `[WinError 193]` silent fallback to the default marker); precedence (`--marker` flag → `DEVFLOW_WORKPAD_MARKER` env → config → built-in default) and the no-config fallback are unchanged and regression-tested. `lib/test/run.sh` pins the generalized invariant with a 22-file loop (a reintroduced bare or cross-statement form goes RED; the portable form is positively pinned per file; the enumeration is count-reconciled so a new skill is auto-covered) plus ERE mutation proofs, and `docs/install.md` documents the Windows/Copilot-CLI operator gotchas (the anchor recipe and inline-bash variable stripping, the PowerShell UTF-16LE write pitfall with the UTF-8-no-BOM remedy, and `workpad.py` text-argument single-quoting from PowerShell), alongside the existing `DEVFLOW_GH` shim remedy. Known residual: multi-statement `VAR=$(…); VAR_RC=$?` guard recipes inside some skill bodies still read a later-statement variable that comes back empty under Copilot's inline-bash marshaling (their rc-discriminating branches silently never fire) — a follow-up wave, out of this issue's anchor scope (resolved in portability wave 3, #284). The shared portable-anchor preamble now states the Windows path-normalization fallback generically ("the same success-and-non-empty acceptance the platform's path-normalization rules apply") instead of naming the DevFlow-internal `lib/normalize-path.sh`, so the two consumer-installed vendored skills (`receiving-code-review`, `requesting-code-review`) stay repo-agnostic; the `lib/normalize-path.sh` lockstep reference remains named in create-issue's internal-only extended variant. (PR #279, closes #275)

## [2.8.61] — 2026-07-03

### Added
- **DevFlow's user-visible cloud posts — the review agent's comments/verdicts/approvals, the trigger reactions, and the notice comments — are now attributed to the optional GitHub App, each via a per-site downscoped token.** Previously the opt-in `DEVFLOW_APP_ID` App covered only the two cloud writers' agent runs; the writers' own pre-flight jobs (the 👀/🚀 trigger reactions via `react-to-trigger.sh` and the notice comments) and the entire auto-trigger review (`devflow-review.yml` → `devflow-runner.yml`) still posted as `github-actions[bot]`, so an App-configured repo spoke with two identities — and a `github-actions[bot]` approval cannot satisfy a "required approving reviews" branch-protection rule. Now, when (and only when) the App is configured, four new sites mint short-lived `actions/create-github-app-token@v2` tokens downscoped via `permission-*` inputs to exactly what each site does: the review agent (`devflow-runner.yml`) gets `contents: read` + `issues: read` + `pull-requests: write` + `actions: read` — enough to read the repo/issue/CI and post comments, reviews, approvals, and rejections, but **incapable of pushing to the branch**; `devflow.yml`'s `gate` job and `devflow-implement.yml`'s `gate` job each get `issues: write` + `pull-requests: write` for the reactions and notices; and `devflow.yml`'s `review_dedupe` job gets `pull-requests: write` for the suppression notice (split into its own step so the dedupe queries stay on `github.token`, where a scoped token could 403 the check-runs/run-list reads and flip the fail-open default). The `permission-*` inputs are the sole least-privilege enforcement (an App installation token ignores the job `permissions:` block); every mint keeps the writers' opt-in contract — gated on `vars.DEVFLOW_APP_ID != ''`, fail-loud (no `continue-on-error`), consumed with a `|| GITHUB_TOKEN` / `|| github.token` fallback so unset-App behavior is byte-for-byte unchanged. The App installation now additionally requires `Pull requests: write`, `Issues: write`, and `Actions: read` (alongside the existing `Contents: write` + `Workflows: write`); a `permission-*` request the App lacks fails the mint loudly. Named exceptions: the `Devflow Review` check-run (runner-emitted, never token-authored) and the workpad comment's gate-job creation stay on `GITHUB_TOKEN` (workpad detection is marker-based, so its claude-job fallback creation under the App token is harmless); the stale-rejection housekeeping runs under whichever token the review agent holds. The `lib/test/run.sh` issue-#201 block is reconciled into a table-driven six-site pin set (positive mint/gate/id/fail-loud/fallback pins per site, exact per-site `permission-*` sets, write-widening absence pins, per-file mint totals, REST-not-porcelain pins on both notice posts). `docs/cloud-setup.md` and `docs/DEVFLOW_SYSTEM_OVERVIEW.md` §15 document the per-site downscopes and the extended App-permission prerequisite. (#278, closes #269)

## [2.8.60] — 2026-07-02

### Fixed
- **The required `Devflow Review` check now concludes `success` only on a positively-observed APPROVE for the *current HEAD*, failing closed in every other direction.** `finalize_check` previously derived the conclusion from `jq -r 'last.state'` (the latest formal review regardless of which commit it targeted) and defaulted `VERDICT=approve` when it found no REJECT, so the check went wrong both ways: a `CHANGES_REQUESTED` left on an *older* commit resurfaced as a phantom REJECT that wedged a mergeable PR (PR #246), and a stalled review that posted **no** verdict for HEAD (empty reviews, or an engine that stopped mid-run) fabricated a green APPROVE on a PR that was never reviewed to a verdict (PR #250). - **Config-driven `@claude` tool allowlist.** New `claude.allowed_tools` and `claude_implement.allowed_tools` arrays append repo-specific `--allowed-tools` entries (claude-code-action syntax, e.g. `Bash(make:*)`) on top of DevFlow's built-in base list — no workflow YAML editing. The two keys are independent: the implement path does not inherit the light path's extras. Documented in `docs/cloud-setup.md`.
- **`/create-issue` confirmation gate.** New Step 4 renders the full assembled issue and waits for the user's explicit approval before `gh issue create` — the drafted ticket is never filed unseen. A gitignored `.devflow/tmp/issue-draft-<slug>.md` preview copy is written for editor review (never the posting source).

### Fixed
- **`/devflow:review` standalone runs no longer post a dangling pointer.** When run directly from an IDE/CLI (`$GITHUB_ACTIONS` unset), Phase 4.4 now puts the full report in the `gh pr review` body instead of a stub pointing at a progress comment that only the auto-trigger workflow creates.
- **`claude-runner.yml` grants `actions: read`** to match its `additional_permissions` input, so Claude's CI-result reads no longer 403 (the explicit `permissions:` block had defaulted the scope to `none`).

## [2.0.0] — 2026-05-22

### Changed
- **BREAKING — config is now `.devflow/config.json` (JSON), not `.github/project-config.yml` (YAML).** Adopters re-run `install.sh` to scaffold the new file; the old YAML path is no longer read (no fallback ships). The live config stays gitignored; a committed `config.example.json` + `config.schema.json` (with `$schema` for editor autocomplete/validation) replace the YAML template.
- **Both config-parser prerequisites dropped.** A single Node-based resolver (`scripts/config-get.sh`, which `lib/conf.sh` now delegates to) reads the config — config parsing no longer needs PyYAML (plugin tier) or yq (cloud tier). The `read-project-config` action validates with Node and passes the JSON through; the previously inline-yq board workflows now use that shared action. (Note: this scopes only the **config layer** — PyYAML is still a DevFlow prerequisite for other helpers, e.g. `match-deferrals.py`'s PR-body parsing.)
- `setup.install` is now a JSON **array of shell lines** (joined with newlines at runtime) instead of a YAML block scalar.

## [1.1.0] — 2026-05-21

### Added
- `/devflow:review-and-fix` — `--push-each-iteration` flag and a PR-mode head-override, so the fix loop reviews its own local commits and can propagate each iteration to the remote PR (and its CI). `/implement` Phase 3.3 sets the flag.
- `/devflow:docs-verify` — `--report-only` mode that returns a findings report without editing, committing, or pushing; `/create-issue` now uses it so issue creation never writes to a protected branch.
- `/implement` — 2.3.0 changed-contract and 2.3.4 boundary-assumption sweeps over the diff.
- GitHub Actions "cloud tier" `install.sh` — one-command install/update — plus configurable cloud-workflow runtime provisioning.
- GitHub autolink-hygiene guidance (no bare `#`+digit unless a real issue/PR reference) across the GitHub-writing skills.
- CI lint job (ruff blocking; shellcheck/actionlint provisional) and expanded Python/shell test coverage.

### Changed
- Command references namespaced to `/devflow:review` and `/devflow:review-and-fix` across docs and skills; the bare forms still resolve when there is no name collision.
- Install instructions add the `claude-plugins-official` marketplace first so cross-marketplace dependencies resolve; documented that PyYAML is not auto-installed by `/plugin`.
- Cloud workflow runs on `ready_for_review`; Node-20 action bumps and a `claude-runner` ref input.

### Fixed
- Root-level devflow plugin now uses the relative-path `./` source in `marketplace.json`.
- `meta-issue.sh` dead `--title` flag.

## [1.0.0] — 2026-05-19

First public release. Extracted into a standalone, generic plugin from its
original home in a private product repository.

### Added
- `/implement` — 4-phase issue-to-PR orchestrator (setup → implement → review → docs).
- `/review` and `/devflow:review-and-fix` — verification-checklist review engine (report-only and auto-fixing).
- `/docs` suite — `docs-sync-internal`, `docs-sync-external`, `docs-bootstrap-internal`, `docs-bootstrap-external`, `docs-verify`, `docs-release-notes`.
- `/create-issue` and `/pr-description`.
- Self-improving retrospective loop — `/devflow-weekly`, `/retrospective`, `/audit-implementations`.
- Optional GitHub Actions "cloud tier" (autonomous issue/PR automation) — see `docs/cloud-setup.md`.
- Auto-installed plugin dependencies: `feature-dev`, `pr-review-toolkit`, `superpowers` (from `claude-plugins-official`).

### Notes
- The local tier (the in-editor skills) works with **no configuration**; every config value has a built-in default.
- Helper scripts reference their bundled location via `${CLAUDE_SKILL_DIR}`, so the plugin works from any install location.
- Shell helpers avoid GNU-only flags (`date -d`, `grep -P`), so the retrospective loop runs on macOS/BSD as well as Linux.
