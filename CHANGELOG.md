## [Unreleased]

### Added
- **Per-subagent model and effort are now configurable for the review engine (`devflow_review.agent_overrides`).** A new config block in `.devflow/config.json` lets you set the `model` and reasoning `effort` for each of the nine review-engine subagents individually, plus an optional `default` fallback. Because the engine is shared, the overrides apply identically to standalone `/devflow:review`, to `/devflow:review-and-fix`, and to the Phase 3 review pass of `/devflow:implement` â€” so you can move a mechanical pass to a cheaper model or lower effort and pin a high-value reviewer to a stronger model or higher effort. Resolution is entry-level: a subagent with its own entry uses only that entry (the `default` does not backfill its missing fields), an explicit empty entry opts a subagent out of the `default`, and a subagent with no entry runs exactly as before. The change is fully backward compatible â€” an existing config with no `agent_overrides` block is unaffected â€” and malformed values are handled gracefully: an out-of-range effort or an unusable model is dropped with a warning and falls back rather than aborting the run. See `docs/review-agent-overrides.md`. (#64)

### Changed

## [2.3.4] â€” 2026-05-26

### Fixed
- **`/devflow:review-and-fix`'s shadow review pass now runs the full multi-agent reviewer fan-out instead of silently degrading to a single-agent self-check.** The Step 2.6 shadow pass â€” the loop's end-of-run convergence audit â€” was dispatched as one `general-purpose` subagent told to execute `/devflow:review`'s whole engine. But the engine's Phase 3 fans out to up to six reviewer subagents (four always-on, plus two structurally-gated analyzers), and a subagent cannot dispatch its own subagents (nested dispatch is unsupported by the harness), so the shadow collapsed to a degraded single-agent pass and returned a plausible clean `APPROVE` â€” the audit never actually ran. This was the root cause of the recurring "a manual `/devflow:review` after the loop finds Important things the loop missed" pattern. Step 2.6 now has the **parent orchestrator** run the shadow's Phase-3 fan-out itself (the parent can dispatch), with independence enforced **per reviewer prompt** â€” each shadow reviewer runs in a fresh context with the loop's prior findings, fix decisions, and pushback history withheld. `coverage: "full"` is now a positively-verified assertion: the parent computes the expected reviewer roster (the four always-on agents plus any gated analyzer whose Phase 3.1 gate is true) and confirms the dispatched roster covered it before declaring convergence. When the fan-out cannot complete, the run records `coverage: "not_verified"` and reports "shadow agreement not verified" â€” never a clean verdict from a degraded pass (fail-closed). The chat output and the report's Coverage section now state explicitly whether the shadow ran with full coverage or was not verified. (#57)

## [2.3.3] â€” 2026-05-25

### Added
- **Standalone `/devflow:review` now narrates its work in a single live progress comment on the pull request.** In pull-request mode, and on by default through the new `devflow_review.live_progress_comment_enabled` setting, `/devflow:review` authors one `devflow:review-progress` comment and updates it in place as it works: a blueprint of the phases up front, then per-phase updates, each Phase-3 review agent's findings as they land, the verdict, and finally a run-telemetry summary and a per-run effectiveness trace. Watching a pull request now means watching the review accrue findings in real time, and the finished comment is a complete record of the run. The effectiveness trace uses a review-specific derivation â€” because a review never applies a fix, a finding counts as effective when it contributed to the verdict â€” and its records carry a `source: "review"` field. The feature works under the read-only cloud reviewer (the comment is posted through `gh`; the on-disk trace record is written only on writable or local runs), and the new setting is independent of `efficiency_telemetry_enabled`, which separately controls whether the telemetry and trace block is embedded. Set `devflow_review.live_progress_comment_enabled` to `false` to keep the previous behavior of producing the report once at the end. See `docs/efficiency-trace.md`. (#56)

### Fixed
- **`/devflow:review` efficiency telemetry no longer overstates cost on engine pull requests, records real verifier and per-phase data, and persists on headless runs.** Three corrections ship together. First, the `engine_self_modifying` Phase 0.5 override stops force-dispatching `type-design-analyzer` and `pr-test-analyzer`: it still runs the full checklist and the four always-on reviewers unconditionally, but those two analyzers keep their structural-applicability gates (`has_new_types` and a unified test-relevance predicate) on every diff profile, so a docs-only or config-only engine pull request no longer pads its cost with analyzers that have nothing to examine. Second, `/devflow:review-and-fix` now reliably writes the `checklist[]` array and the `telemetry` block to each iteration workpad whenever the checklist ran, so efficiency-trace records report the real lite-versus-agent verifier split and per-phase cost instead of `none-recorded` or null. Third, the efficiency-trace helpers are now invoked directly (no `bash` prefix) and `lib/efficiency-trace.sh` is committed executable and allow-listed, so headless and cloud runs persist the per-run record instead of silently skipping it at the permission gate. See `docs/efficiency-trace.md`. (#53)

## [2.3.2] â€” 2026-05-25

### Fixed
- **`/devflow:init` and `install.sh` now detect project languages on Windows again.** The native Windows `jq` build (the winget `jqlang.jq` package, run under Git Bash / MSYS) terminates every stdout line with `\r\n`. The two marker/preset `read` loops in `scripts/detect-project-tools.sh` were capturing each key with a trailing carriage return, so the follow-up `.presets["dotnet\r"]` lookup found no such preset and a repo with valid markers (e.g. a `*.csproj`, `package.json`, or `go.mod`) was reported as "no known language markers detected" â€” leaving the three `allowed_tools` allowlists and `setup.install` empty. Both pipelines now pipe jq's output through `tr -d '\r'` before `read`, so detection behaves identically across Linux, macOS, and Windows. The other jq calls in the script write JSON to a tempfile or feed jq-as-arg, where a stray `\r` is harmless whitespace to the JSON parser, so they need no change. A regression test shadows `jq` on `PATH` with a CRLF-appending wrapper to lock the Windows path in.

### Improved
- **The `/devflow:implement` workpad now appears almost immediately, and reads more cleanly.** The workpad â€” a run's single "job started" comment â€” is now created by the workflow's `gate` job right after authorization and de-duplication, *before* the heavy `claude` job boots and provisions its runtime (`setup-project-env`: Python/Node/service containers/dependency installs), which the acknowledgment never needed. On repos with containers or heavy installs that removes minutes of silence between the trigger and the first comment. The `claude` job's Phase 1.3 detects the pre-created workpad (`workpad.py id`) and resumes it â€” filling in the Plan and Acceptance Criteria on the *same* comment â€” so a run still posts exactly one comment. Two readability changes ship alongside: the `Last updated:` line is now friendly UTC (`2026-05-05 17:42 UTC`) instead of raw ISO-8601, and the separate `Decisions / Notes` section is gone â€” its timestamped notes now nest under the matching phase inside `## Progress`, so "what happened, by phase" lives in one place (`## Devflow Reflection` remains its own collapsible section). Workpads created before this change still resume without error and keep their existing notes. A new `workpad.py new-body` subcommand emits the lean initial skeleton the `gate` job posts. See `docs/workflow-triggers.md`. (#49)

## [2.3.1] â€” 2026-05-24

### Fixed
- **The cloud tier now reviews and implements on thin-install repos again â€” the runtime-vendored plugin moved out of `.claude/`.** `claude-code-action` runs a security step on every pull request that deletes each of its `SENSITIVE_PATHS` (including the whole `.claude/` directory) and restores them from the base branch *before* installing plugins. A plugin vendored at `.claude/plugins/devflow/` was therefore wiped, so `plugin install` failed with `Source path does not exist` and the run aborted before producing a verdict â€” breaking `devflow-review` and `devflow-implement` on every thin-install consumer (the default mode). The `vendor-plugin` action, `install.sh`, the local `marketplace.json`, and all workflow helper-script references now use `.devflow/vendor/devflow/`, which sits outside every `SENSITIVE_PATH` and survives the restore. `install.sh` also removes a stale committed tree at the old `.claude/plugins/devflow/` path on upgrade (signature-guarded), and â€” because the relocated tree now survives the restore â€” a thin install adds `/vendor/` to `.devflow/.gitignore` so a cloud run's `git add -A` never commits it (a `DEVFLOW_VENDOR=1` install keeps it tracked). Apply the fix by re-running `install.sh` (or picking up the updated workflows); a config-only `devflow_version` bump does not relocate the path. (#47)

### Improved

### Fixed

- **The automated Devflow Review now reliably fires on a pull request's first ready-for-review, instead of silently never running.** The first-ready gate that enforces "auto-review exactly once per pull request" was counting *every* `Devflow Review` check-run when deciding whether a pull request had already been reviewed â€” including the `skipped` check-runs that the dual `pull_request`/`pull_request_target` wiring routinely emits (the deduplication loser, plus draft and synchronize deferrals). ### Changed

- **`devflow_runner.allowed_tools` is now consumed by the automated reviewer â€” no longer deprecated/inert.** When `devflow_runner.provision_env: true` in your trusted base-branch config, the reviewer now appends the freeform `devflow_runner.allowed_tools` list (read verbatim from the base ref) to its read-only `review` profile, replacing the previous hard-coded build append (`npm`, `npx`, `node`, `yarn`, `pnpm`, `composer`, `php`, `make`). ## [2.2.7] â€” 2026-05-23

### Added
- **Opt-in build environment for the automated reviewer (`devflow_runner.provision_env`).** A new boolean config key (default `false`) lets the automated reviewer provision a real build environment before it inspects a pull request. When set to `true` in your repository's base-branch config, the reviewer runs the same `setup-project-env` provisioning the other paths use (Python/Node/PHP, service containers, and your `setup.install` lines) and gains a fixed build/verify command allowlist (`npm`, `npx`, `node`, `yarn`, `pnpm`, `composer`, `php`, `make`), so it can actually compile, lint and test a change instead of only flagging build-dependent claims. For security, both the flag and the `setup` block are read only from the trusted base branch â€” never from the pull request's own checkout â€” so a pull request cannot enable provisioning or inject install commands into its own review; when the flag is absent or `false` the reviewer stays byte-for-byte read-only with no added latency. See "Letting the reviewer build/test a PR" in `docs/cloud-setup.md`. (#20)
- **Support for subdirectory/monorepo Node builds in cloud provisioning (`setup.node_working_directory`).** A new optional config key names the directory that holds your Node project's `package.json`/lockfile, for repos whose JavaScript build lives in a subdirectory (a PHP/Rails app with a `/jsx` or `/resources/js` bundle, or a monorepo with a `frontend/` package) rather than at the repo root. When set, `setup-node`'s download caching resolves the lockfile under that directory, and `/devflow:init` auto-detects the subdirectory lockfile and scopes the generated Node install line into it (a subshell `cd`). When the key is empty or absent (the default), provisioning is byte-for-byte identical to the previous root-based behavior; `setup.install` lines still run from the repo root. (#19)
- **Per-run subagent effectiveness telemetry for `/devflow:review-and-fix`.** At the end of every run, the fix loop now derives a per-run effectiveness trace â€” printed to chat and written as one durable JSON record to `.devflow/logs/efficiency/<slug>-<timestamp>.json` â€” that scores each dispatched review subagent with exactly one of four verdicts (unique-effective, corroborating, noise, or null), so you can see which agents earned their cost on a given PR. Each iteration also records its Phase 0.5 diff profile and a verification posture, so cross-run analysis can segment by diff shape and the trace shows when verifier subagents were intentionally not dispatched (a cost-saving choice, not a gap). The record also folds in the existing per-phase cost telemetry so that data survives the ephemeral teardown of `.devflow/tmp/` â€” a destroyed cloud runner or a local scratch cleanup alike. Two new config keys under `devflow_review_and_fix` control it: `efficiency_telemetry_enabled` (boolean, default `true`) gates the behavior, and `efficiency_cut_candidate_min_dispatch` (integer, default `3`) is reserved for a follow-up cross-run analyzer. See `docs/efficiency-trace.md`. (#16)

### Deprecated

To give the reviewer build access, set `devflow_runner.provision_env: true` (see above), which grants a fixed build/verify allowlist. `/devflow:init` still auto-populates the array per detected language, but those entries do not change the reviewer's behavior; the `devflow.allowed_tools` and `devflow_implement.allowed_tools` keys continue to take effect in their own workflows. ## [2.2.4] â€” 2026-05-23

### Fixed
- **`devflow-review.yml` no longer aborts as `startup_failure`** (the required `Devflow Review` check was permanently stuck, wedging every PR in adopting repos). Root cause: GitHub validates a called reusable workflow's permission ceiling against the caller's grant across the *whole* called graph at build time.
